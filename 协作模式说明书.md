# 南塘云村 · AI 协作模式说明书

> 2026-07-23 · 本文件是项目协作模式的唯一权威说明。任何新加入的 AI 先读这份文件再开工。
> 一句话：**砚仁（皇帝）提问题 → Kimi Work（丞相）设计与写卡 → Kimi Code（监察）排查与校验 → Claude Code（施工）改代码 → 验收上线。**

---

## 一、四方角色（祖制，不可越权）

| 角色 | 是谁 | 干什么 | 绝不干什么 |
|---|---|---|---|
| 皇帝 | 砚仁（人） | 提需求、报现象、批设计、看政绩 | 不操心执行细节 |
| 丞相 | Kimi Work | 写任务卡、做设计、验收 diff、回答取舍问题 | 不搬砖（省 token） |
| 监察 | Kimi Code | 排查断点（只读）、校验施工成果、打回不合格 | **永远不改业务代码** |
| 施工 | Claude Code | 所有代码修改都归它，照卡执行 | 不 push、不自由发挥 |

## 二、每张任务卡的四步流转

```
① 监察排查（只读）→ 断点结论写 BUG_TRACKER.md（文件:行号+证据）
② 施工修复 → 单独 git commit（用卡里的 message），不 push
③ 监察校验（审 diff + 跑验收命令 + 实测）→ 「通过/打回+原因」写 BUG_TRACKER.md
④ 砚仁或 Kimi Work push（push = 上线 Render）→ 砚仁冒烟看政绩
```

**政绩凭证只有两个：`BUG_TRACKER.md` + `git log`。口说无凭。**

## 三、铁律

1. 只移不删：任何文件/代码不直接删除，移入 `存档/` 并留说明
2. 一次一个目标：一张卡只干一件事，单独 commit
3. 心脏文件（`nantang-mobile/js/nt-core.js` / `app-data.js` / `api.js`）的卡只由 Kimi Work 写
4. 同一时刻只有一方改代码，commit 后另一方才动手
5. 浏览器实测类验收必须实测，不许只静态检查

---

## 四、给新加入 AI 的边界（重要）

**你可以做**：阅读一切；在自己的工作目录（建议新建 `外援/` 目录）里产出文档、分析、方案；回答砚仁直接问你的问题。

**你不许碰**：
- `方案/任务卡/` —— Kimi Work 的派工区，只有 Kimi Work 能写卡改卡
- `BUG_TRACKER.md` —— 只允许按格式追加，不许修改已有条目
- `设计/` —— 已定稿的设计，改动需砚仁批准
- `存档/` —— 只读区
- 业务代码（`nantang-mobile/`、`server/`）—— 修改只能由 Claude Code 持卡执行
- git 操作 —— 不要 commit/push，除非你被明确授予

**如果你的任务和现有卡片冲突**：不要开工，把冲突点写出来告诉砚仁，由 Kimi Work 协调。
