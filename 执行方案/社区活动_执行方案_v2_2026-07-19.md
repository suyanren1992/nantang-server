## 一、设计总纲

### 1.1 五条核心设计理念

| # | 理念 | 含义 |
|---|------|------|
| ① | **Hub-Spoke 导航** | 村口是唯一中枢，所有功能从村口辐射出去。不做深层嵌套，永远可以一键回村口 |
| ② | **渐进式披露** | 营地主页 6 个 Tab 不是平等的——概览 Tab 聚合关键信息，其余 Tab 按需深入。不把 PC 端的 32 个区块一次性砸给用户 |
| ③ | **角色即界面** | 同一个屏幕，不同角色看到不同内容。底部栏、概览区块、管理入口全部按 `currentUser.role` 动态渲染。不是「做 3 套页面」，是「1 套页面 + 角色分支」 |
| ④ | **情感节拍** | 创营、启营、结营三个仪式节点用 GSAP 动画制造情感峰值。不是花哨——是在关键决策点给用户一个「此事重要」的信号 |
| ⑤ | **移动端原生思维** | 不搬运 PC 端的表格和并排卡片。拇指热区在下半屏，关键操作放底部。横向滑动用于日期选择，纵向滚动用于内容浏览 |

### 阶段 0：地基 — 数据层 + 身份系统

**为什么是阶段 0：** 所有后续功能依赖 `AppData` 和身份系统。没有用户隔离，社区主页不知道「谁在看」；没有身份流转，邀请码升级无从谈起。

**设计原理：**
- `AppData` 单例（`js/app-data.js`）是唯一的数据入口。所有读写走 `AppData.my()` / `AppData.shared()`，不再有散落的 `saveAppData()` 调用
- 设计上模仿后端 API 签名——`AppData.my('tasks')` 现在是读 localStorage，未来切后端时调用方代码不改
- 身份流转 `visitor → adventurer → builder` 是单向门。升级不可逆。`admin` 仅砚仁一人，不可通过邀请码获得
- 角色决定界面，但角色数据存在 `AppData.users[name].role` 里，不在 UI 状态里

**现有基础：**
- `js/app-data.js` 已创建（`AppData` 单例 + `my()` / `shared()` / `switchUser()` API）
- `js/nt-core.js` 已迁移到 AppData
- `js/auth.js` 已支持注册/登录/角色存储

**本阶段做什么（无 UI，纯验证）：**
1. 确认 `AppData` 覆盖所有现有数据路径（TASKS、LEDGER、NT、items、profile）
2. 确认 `switchUser()` 后所有 UI 正确刷新
3. 确认身份流转函数 `upgradeRole()` 可用
4. 删除残留的全局变量直读（如果有）

**验收标准：**
```javascript
AppData.switchUser('小云');
AppData.my('ntBalance');      // → 小云的余额，不是砚仁的
AppData.switchUser('砚仁');
AppData.shared('taskMarket'); // → 所有用户都能看到
NT.verify();                  // → {pass: true}
```

**⏭️ 无需 UI 审批，直接进入阶段 1 前置。**（阶段 0 已在 `存档/快照_20260719_Step0完成/` 验证通过）

### 阶段 2：创营 — 7步向导 + 启营仪式

> **UI 设计详案：** [`阶段2_创营_UI设计.md`](阶段2_创营_UI设计.md) · 📝 设计完成，待审批

#### UI 设计

7 步向导，每步一个独立 overlay。底部固定翻页按钮，← 可回退，已填内容保留。

```
① 创营仪式 → ② 时间参数 → ③ 财务参数 → ④ 任务参数 → ⑤ 人员选择 → ⑥ 分包确认 → ⑦ 启营仪式
```

##### ① 创营仪式

```
┌──────────────────────────────────────┐
│ ✕ 关闭    ① 创营 · 创建营队          │
├──────────────────────────────────────┤
│   营队名称  [例：第四期南塘共创营]     │
│   期数      [第四期 ▼]               │
│   类型      ○ 常规共创营  ○ 特别活动营 │
│   一句话主题 [例：南塘有风，共创有光]   │
│   描述      [可选，多行]              │
│   ☐ 🧪 测试模式                      │
├──────────────────────────────────────┤
│       [✨ 点燃火种 →]                 │
└──────────────────────────────────────┘
```

确认 → 🕯️ 创营仪式动画（烛火→火种→星光→「营队正式创立」，约 3 秒）→ 自动进入②。

##### ② 时间参数

日程表：左侧时段列 sticky，右侧日期区域横向滑动。每格可点击编辑。底部操作栏：`[+天] [-天] [+时段] [📋复制前一天] [💾存为模板]`。

里程碑节点自动计算（宣传日 = 开营-14天，展览日 = 开营+5天，结营日 = 开营+7天）。

##### ③ 财务参数

收入区：冒险者人数 × 天数 × 早鸟/原价 → 收入合计。
支出区：住宿单价 × 人数 × 天数 + 吃饭单价 × 人数 × 天数 → 支出合计。
底部实时显示预测损益（RMB 盈亏 + NT 池总额）。

##### ④ 任务参数

任务列表 + 类别筛选 chip + 状态筛选。每行：名称/类别/营队类型（6种彩色标签）/激励点/删除。
关键设计：类别（宣传/课程/生活/财务/管理/结项/其他）与类型（主线M/支线S/共创C/隐藏H/身份I/彩蛋E）是正交的两个字段。

##### ⑤ 人员选择 — 「虚位以待」

顶部是空圆圈（灰色虚线），每确认一个人 → 圆圈变实心头像 + 名字 + 任务数/NT。
选框是紧凑小列表（头像 28px + 名字 + 在线绿点/离线红点 + 搜索框），不显示角色标签。
点击头像可继续给已有共建人加任务。一个人可以多次认领。

##### ⑥ 分包确认

管理员发送确认 → 共建人逐人逐页确认（防跳过）：
- 第 1 步：确认预算（收入/支出/P&L 表 + ✓确认按钮）
- 第 2 步：确认日程（日期滚轮 + 完整时间线 + ✓确认按钮）
- 第 3 步：确认任务包（任务卡片 + ✓确认接受按钮）

每步未确认时下一步灰掉。管理员侧追踪确认状态（✅已确认 / ⏳等待确认）。全部确认 → 启营按钮亮。

##### ⑦ 启营仪式

⚔️ Logo 从天而降 → 光芒扩散 → 营队名 + 日期浮现 → 共建者头像卡片依次滑入 →「正式开营」按钮脉冲出现。
时长约 4 秒。之后营队状态变为「进行中」。

#### 设计原理

**为什么是 7 步独立 overlay 而不是一页长表单？**
- 每一步是独立的认知单元。长表单让人焦虑（「还有多少？」），分步让人有掌控感（「第 3/7 步」）
- 每步底部只有「← 上一步」和「下一步 →」，减少决策疲劳
- 已填内容保留——回退不是「丢失」，是「回去修改」

**为什么第⑤步叫「虚位以待」而不是「分配任务」？**
- 心理暗示不同。「虚位以待」暗示这个位置等待一个值得的人来填补——有仪式感
- 「分配任务」是管理术语，机械冰冷
- 空圆圈 → 实心头像的视觉转换，本身就是一个小型的仪式动画

**为什么创营/启营需要 GSAP 动画？**
- 这两个节点是营队生命周期的「门」。跨过这道门的时刻应该有重量感
- 不是装饰——动画给用户一个明确的信号：「刚才发生了一件重要的事」
- 时长控制在 3-4 秒，不阻塞操作。仪式完成后自动进入下一步

**为什么确认分包要 3 步逐页确认而不能一键确认？**
- 设计文档明确要求「防止直接滑到底就点了」。如果一页全确认，人会跳过阅读
- 每步一个独立的确认按钮，强制注意力停留
- 未确认时下一步灰掉——不给「假装读了」的捷径

- 底部栏留给角色快捷操作（👤/📅/📬/＋），那是拇指热区最重要的四个按钮
- Tab 是「营地内的视角切换」，底部栏是「跨营地的常驻操作」——两个层次，分离是对的

**为什么概览 Tab 是聚合视图而不是独立页面？**
- 概览 = 「我现在需要知道什么」。不是信息堆砌，是信息优先级排序
- 营队概览卡片（全局进度）→ 我的任务包（个人责任）→ 今日安排（时间锚点）→ 里程碑（方向感）
- 这个顺序对应人的自然阅读顺序：整体→个人→当下→未来

**为什么资金账本用双卡片而不是表格？**
- 移动端表格 = 横向滑动 + 字体缩小 = 体验差
- 卡片天然适配手机宽度。RMB 卡和 NT 卡并排或上下堆叠，数据一目了然
- 进度条可视化——不需要读数字就能感知「花了多少」

**为什么底部栏按角色切换？**
- 共建者的核心操作是「管理 + 沟通」（📬消息，＋快捷菜单）
- 冒险者的核心操作是「生活 + 学习」（🥬订餐，🏨住宿）
- 如果做通用栏（5个按钮），两个角色各有两个按钮用不上——浪费拇指热区

- 区块之间有业务流（审批通过→进入待结算→一键结算→NT池更新），视觉上独立但逻辑上流水线

- 任务包类别 → 区块映射是纯规则，没有 AI——可预测、可调试

- 筛选 chip 放在顶部（不是下拉），降低操作成本——点一下就行

**为什么＋菜单也是个性化的？**
- 共建者的「＋」是他的工具集。不同工种需要不同工具
- 底部 Sheet 而不是跳转页面——保持上下文，操作完还在营地主页

#### 设计原理

**为什么结算详情要展示「NT 计算方式」？**

- 这是实景仪式的数字映射。现实中大家在敬字亭烧掉练习稿，App 里的按钮是仪式的触发器

## 四、审批流程

```
我输出 UI 设计说明
        │
        ▼
   你审查 UI
   ├── 「这个改一下...」→ 我调整 → 再看
   ├── 「这个为什么这样设计？」→ 我解释原理
   └── 「可以，开始写代码」→ 我实现
        │
        ▼
   我写代码 + 自测
        │
        ▼
   你验收 → 进入下一阶段
```

每个阶段独立审批。不需要一次性看完 6 个阶段再做——**先看阶段 1 的 UI 设计，觉得方向对了就往下走。**
