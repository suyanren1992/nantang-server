## 0. 一句话

**第一代用现成工具马上跑起来（多签钱包 + 链上查验 + 平台流水），第二代用托管合约把"人管钱"升级成"代码管钱"。两代的卡片室、任务、打赏等前端体验完全不变。**

## 1. 三层资产（继承 v11，不变）

| 资产 | 性质 | 规则 |
|---|---|---|
| **NT** | 结算工具 | OP Chain 上的 ERC20；平台日常记虚拟账本，定期净额清算 |
| **贡献值** | 劳动净值 | 决定权限；消费时 75% 给卖家 + 25% 进社区公共池 |
| **经验值** | 永久资历 | 只涨不跌；解锁称号与权限（陪审团 ≥500） |

**边界：不做真钱充值**（支付合规风险）。NT 只能通过劳动赚取或从自己的钱包充入。

## 2. 第一代：多签储备池 + 链上查验（现在就能做）

### 2.1 角色

- **社区多签钱包**（推荐 Safe，OP Chain）：社区储备池，所有充值的目的地
- **chain_scanner**（后端已有 `server/chain_scanner.py`）：扫描链上转账，验证"他真的转了吗"
- **平台账本**（`nt-core.js` → 服务器数据库）：记录平台余额、冻结、流水

### 2.2 五条资金流

```
① 充值
成员钱包 ──链上转账──▶ 社区多签钱包
chain_scanner 查验到账 ──▶ 平台余额增加
（云村民打赏、发任务的弹药都从这里来）

② 发任务（冻结）
发任务必须先冻结平台余额中的等额 NT ──▶ 任务发布
（冻结是防"发空头任务"的纪律）

③ 任务完成（释放）
完成 + 确认 ──▶ 冻结额释放给完成者
涉及链上 P2P 转账的：查验通过 → 释放；查验不过 → 不释放，转校核
**"释放与否"本身就是一个校核问题**

④ 打赏
云村民充 NT 到储备池 → 平台查验 → 平台余额 → 打赏给在地成员

⑤ 线下 P2P 互付
成员之间可直接用钱包互相支付 NT（不过平台）
平台上发生的操作只记平台流水；两条轨道并行
```

### 2.3 双轨并行与月度结算

- **平台上赚的 NT ≠ 自己钱包里的 NT**，两条轨道并行
- **月底净额清算**：汇总每人净应收/净应付 → 公共池债权优先 → 生成清算方案 → 成员决定要不要把平台赚的 NT 提现回钱包
- 净额算法参照 DeepSeek 方案第五章（45 笔流水压成最少笔数，省 80% Gas）

## 3. 第二代：托管合约（有链上开发能力时）

将 DeepSeek 方案（v1.0，已收录）落地：

- **NantangEscrow.sol**：`deposit()` 充值 / `createTask()` 冻结 / `completeTask()` 释放 / `batchSettle()` 批量净额清算
- 多签钱包降级为合约 owner（仅紧急暂停用）
- 安全要求（按 DeepSeek 清单）：OP Sepolia 测试网验证 → Slither 审计 → 测试覆盖率 >90% → 主网
- 争议处理：双方提证 → 随机抽 3 名高信誉陪审员 → 2/3 投票 → 合约按结果释放（衔接 v11 陪审团 ≥500 经验值）
- 信誉分系统：可信/待观察/受限/冻结四级，与仲裁结果联动

**升级为第二代的触发条件**：社区月流水超过多签人工处理能力，或有成员具备 Solidity 开发能力。没到这一天，第一代就一直跑。

### 第二代落地前要写清的两个细节

1. 从不充值、只接任务赚钱的成员（多数在地成员），净额清算时的收款路径
2. 周结 vs 月结（DeepSeek 建议周结，v11 建议月结——由公约投票定）
