## 〇、核心设计哲学

> **“把复杂的链上交互藏在后端，让用户只感受到劳动与收获。”**

```
┌─────────────────────────────────────────────────────────────────┐
│                    用户感知层                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  用户看到：发布任务 → 冻结保证金 → 完成任务 → NT到账    │   │
│  │  用户操作：充值一次 + 日常点击（无钱包弹窗）             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                  │
│                              ▼                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    系统处理层                             │   │
│  │  日常记账（链下）→ 净额清算 → 批量结算（链上）           │   │

│                              │                                  │

│  ┌─────────────────────────────────────────────────────────┐   │

### 2.1 整体闭环

```
┌─────────────────────────────────────────────────────────────────┐
│  ① 发布任务（链下）                                             │

## 五、净额清算机制

### 5.1 核心逻辑

平台采用**净额清算**（Net Settlement），将 N 笔交易压缩为最少笔数，大幅降低 Gas 成本。

```
┌─────────────────────────────────────────────────────────────────┐
│  原始流水（一周 45 笔任务）                                      │
│  ─────────────────────────────────────────────────────────────── │
│  小红 → 朝林 30NT                                               │
│  小红 → 砚仁 20NT                                               │
│  朝林 → 小红 15NT                                               │
│  朝林 → 砚仁 10NT                                               │
│  砚仁 → 小红 25NT                                               │
│  ...（共 45 笔）                                                │
├─────────────────────────────────────────────────────────────────┤
│  净额汇总                                                        │
│  ─────────────────────────────────────────────────────────────── │
│  小红：应收 40NT - 应付 50NT = -10NT（净应付）                  │
│  朝林：应收 30NT - 应付 25NT = +5NT（净应收）                   │
│  砚仁：应收 30NT - 应付 25NT = +5NT（净应收）                   │
│  ...                                                           │
├─────────────────────────────────────────────────────────────────┤
│  最短路径清算                                                    │
│  ─────────────────────────────────────────────────────────────── │
│  小红 → 朝林 5NT                                                │
│  小红 → 砚仁 5NT                                                │
│  ...（仅 8 笔链上交易，而不是 45 笔）                           │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 清算算法伪代码

```python
# 输入：一周所有待结算流水
def calculate_net_settlement(entries):
    # Step 1: 按人汇总
    balances = {}  # userId -> 净额
    for entry in entries:
        balances[entry.from] -= entry.amount
        balances[entry.to] += entry.amount
    
    # Step 2: 分离净应付和净应收
    payers = [(user, -amount) for user, amount in balances.items() if amount < 0]
    receivers = [(user, amount) for user, amount in balances.items() if amount > 0]
    
    # Step 3: 匹配生成最少转账路径
    settlements = []
    i, j = 0, 0
    while i < len(payers) and j < len(receivers):
        payer_user, payer_amount = payers[i]
        receiver_user, receiver_amount = receivers[j]
        
        transfer = min(payer_amount, receiver_amount)
        settlements.append((payer_user, receiver_user, transfer))
        
        payers[i] = (payer_user, payer_amount - transfer)
        receivers[j] = (receiver_user, receiver_amount - transfer)
        
        if payer_amount - transfer == 0: i += 1
        if receiver_amount - transfer == 0: j += 1
    
    return settlements
```

## 八、安全与风控

### 8.1 安全措施矩阵

| 层级 | 风险 | 应对措施 |
|------|------|----------|
| **合约层** | 代码漏洞 | OpenZeppelin 标准库 + Slither 审计 + 测试网验证 |
| **合约层** | 重入攻击 | ReentrancyGuard |
| **合约层** | 私钥泄露 | 管理员使用多签钱包（3/5） |
| **平台层** | 恶意发布 | 保证金冻结机制（发任务必须先充值） |
| **平台层** | 作弊完成 | AI验证 + 照片GPS校验 + 人工抽检 |
| **平台层** | 发布者不确认 | 超时自动确认 + 信誉扣分 |
| **平台层** | 资金被盗 | 合约暂停机制 + 用户可自主提现 |

### 8.2 信誉风控系统

```js
// ===== 信誉规则 =====

// 加分行为
+5：完成任务并获得确认
+3：发布任务并按时确认
+2：成功核验他人标记/任务

// 扣分行为
-10：发布任务后恶意不确认（经仲裁）
-15：提交虚假证明（经仲裁）
-5：无故取消已接任务
-20：严重违规（经仲裁）

// 信誉等级
100-80：🟢 可信 → 正常使用
79-60：🟡 待观察 → 任务需额外验证
59-40：🟠 受限 → 不可发任务，仅可接任务
< 40：🔴 冻结 → 暂停使用，需人工解封
```

## 十、方案总结

| 维度 | 说明 |
|------|------|
| **安全** | 合约托管，无人能挪用资金；OP Chain 去中心化保障 |
| **经济** | 月成本 < $1（30人社区），Gas费极低 |
| **体验** | 日常操作无钱包弹窗，每周自动结算 |
| **透明** | 所有交易链上可查，公开透明 |
| **可控** | 代码自写自部署，不依赖第三方 |
| **合规** | 平台不持币，无资金池风险 |

**一句话总结**：

> **日常记账在链下（快、免费），资金托管在合约（安全、透明），批量结算在链上（经济、可追溯）。三者结合，构成了南塘云村完整、安全、经济的结算闭环。**
