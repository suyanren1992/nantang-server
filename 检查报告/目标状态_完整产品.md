## 一、理想：这个产品用起来是什么样

**让社区里每一份劳动被看见。不是监控，不是管教——是把已经发生的美好放大出来。**

### 核心场景：一件事被看见的全过程

小红在画室打扫了 15 分钟。摄像头感知到了——系统生成一条匿名线索：「画室，下午 2:00-2:20，有人持续活动」。**但不公告。** 只是存着。

朝林路过画室，发现变干净了。他在系统里发起一个发现：「画室变干净了，谁做的？」系统把线索推给他：该时段有 1 人在画室活动。朝林猜是小红。

系统内部已经知道答案——就是小红。所以**朝林不需要等小红回应**。猜对了，朝林立刻 +5 NT。

小红随后收到通知：「朝林发现了你打扫画室。确认公开可获得 +10 NT，也可以保持匿名。」确认了——公开记录写入个人面板。不确认——朝林还是拿到他的 5 NT，但小红的身份不被公开。

**全过程没有管理员、没有填表、没有汇报。** 驱动这一切的不是系统——是朝林注意到了变化，小红确认了他的注意。

### 五层架构：怎么做到的

```
L5 🎮 元游戏层    卡片室 · 赏金池 · 称号 · 标签 · 新手引导 · AI 助手
                  「把看见变成一件有反馈的事」

L4 📊 价值结算层   NT / 贡献值 / 经验值 · 发现者与被发现者共同确认
                  「让每一次看见都可量化」

L3 🗺️ 数字孪生层   空间语义 · 标记 · 物品追踪 · 田间 · 住宿 · 卫生 · 地图
                  「维护一个与现实同步的虚拟副本」

L2 🧠 感知推理层   多源融合 · 行为推理 · 匿名线索 · 冲突仲裁
                  「记录线索，不直接公告」

L1 🔌 硬件接入层   摄像头 · ESP32-CAM · 手环 · 手机GPS
                  「从现实世界采集原始数据」
```

### 为什么不是「任务分配系统」

传统方式：管理员看谁干了什么 → 分配任务 → 催促执行 → 评判好坏。管理员很累，管理员不在系统停摆。

产品说明书的方式：**把管理员拆掉。** 发现劳动 → 任何成员发起。确认价值 → 系统提供数据，人做判断。分配奖励 → 公约定价 + 双方共同确认。记录存档 → 系统自动写入。**没有人是管理员，每个人都是管理员。**

### 系统最终退场

当每个人习惯了主动观察、主动劳动、主动认可——系统就不再是必需品。**留下来的是人：一群会看见、会认可、会创造的人。**

## 六、防漂移规则

1. **每次开始工作前，看 §四 确认在做哪一轮。** 不在 R/M/L/F 任何一项里 → 停下来。
2. **做完一项，立刻在 §三 打 ✅。** 不改文档 = 没做完。
3. **顺手优化只能写 `ponytail:` 注释，不能展开。**
4. **新需求先问：它在五层架构的哪一层？如果是 L5——等 L3+L4 地基跑通。**
5. **每周跑一次 `NT.verify()`。** 不过 = 停手头一切。

## 七、"做完"的定义

| 场景 | 验证方法 |
|------|---------|
| S1 | 地图打扫 → NT 到账 → `NT.verify()` 通过 |
| S2 | 发任务→换人接→提交→审核→两人 NT 对 → `NT.verify()` 通过 |
| S3 | 工作台看到 NT 余额 + 进行中任务 + 活动记录 |
| S4 | 创建营队→加人→营地主页→看到概览 |
| S5 | 卡片室：创建一条线索→有人猜→猜对→做事的人和猜的人都得 NT → `NT.verify()` 通过 |

**S1-S4 跑通 = MVP 完成。S5 跑通 = L5 卡片室验证通过。**
