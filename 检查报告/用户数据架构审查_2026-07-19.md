# 南塘云村 · 用户数据架构全面审查

> 审查日期：2026-07-19
> 审查范围：用户数据管理、隔离、完整性、安全性、迁移路径
> 参与 Agent：security-engineer / technical-director / lead-programmer
> 验证 Agent：security-engineer / technical-director / lead-programmer（交叉核查）

---

## TL;DR

**根因**：6 个 localStorage 命名空间 + 8 个内存全局变量各自持有重叠的用户数据，没有单一真相源。认证依赖一个全局字符串 `CURRENT_USER`，Console 一行代码即可冒充任何人。NT 金融核心无服务端权限检查。

**现在最该做的三件事**（按顺序）：
1. **存储合并**（8→3）：auth(身份) + NT(财务) + AppData(应用)。删除 `data` 全局、`MOCK_ITEMS`、`CAMP_MOCK`、`DRAFTS`。预计 ~400 行 / 2 天。
2. **userId 替代名字作主键**：名字会改、有 Unicode 碰撞。改为整数 PK，影响所有查询和 API 设计。
3. **任务状态机统一**：`draft → active → claimed → submitted → reviewed → settled`，含驳回和取消分支。当前散落在 10+ 个内联 onclick 里。

**迁移路径**：Phase 0（现在，存储合并 + userId）→ Phase 1（1-2周，Express + SQLite 后端，UI 不动）→ Phase 2（2-4周，zod 校验 + 审计日志 + 限流 + 备份）。

**最严重的安全发现**：`renderProfile()` 存储型 XSS（用户名/简介未转义直接 innerHTML）、NT 核心零权限检查（Console 可无限铸币/转账）。

### 1.3 同一份数据写在几个地方？

以 `avatar_seed` 为例：

```
registerUser()      → nt_users + camp_data        (不写 AppData)
changeUserRole()    → nt_users + camp_data        (不写 AppData)
switchUser()        → nt_users → AppData           (仅首次创建时复制)
saveProfileEdits()  → nt_users + AppData + camp_data (三向写入)
```

任意一个写入路径被跳过，三家存储就不同步。AppData 只在首次 `switchUser` 时从 `nt_users` 复制，之后外部对 `nt_users` 的任何修改（角色变更、种子数据导入）都不会自动同步到 AppData。

### 1.4 其他分叉数据

**邀请码三重系统：**
- `nt_world_codes`（auth.js：`getWorldCodes()`/`saveWorldCodes()`）— 世界通行码
- `nt_app_v2.inviteCodes{}`（AppData：`validateInviteCode()`/`generateInviteCode()`/`consumeInviteCode()`）— 营地邀请码
- `nt_invite_codes`（auth.js：`getInviteCodes()`/`saveInviteCodes()`）— 另一套邀请码

这三套邀请码系统互不知情，各自独立管理。Phase 0 存储合并计划已覆盖此分叉（合并到 `AppData._data.inviteCodes`）。

## 二、认证与授权

### 2.1 当前认证机制

```
登录流程：
  1. 用户输入名字 + 密码
  2. loginUser() 从 nt_users 读密码哈希
  3. encodePassword(input, name) 做 2000 次迭代哈希
  4. 比对成功 → CURRENT_USER = '名字'
  5. 写入 nt_session = { name, role }

注册流程：
  1. 用户输入名字 + 密码
  2. registerUser() 调 encodePassword() 存哈希
  3. 写入 nt_users + camp_data.members
  4. AppData.switchUser() 初始化 AppData
```

### 2.2 发现的安全问题

| 严重度 | 问题 | 详情 |
|--------|------|------|
| 🔴 致命 | **认证可被 Console 绕过** | `CURRENT_USER = '砚仁'` 一行代码即可冒充任何人。所有后续操作（claimTask, doSubmit, reviewTask, approveTx）都信任这个全局字符串。 |
| 🔴 致命 | **NT 核心无权限检查** | `NT.topUp()`, `NT.cashOut()`, `NT.transfer()` 不验证调用者身份。Console 可无限铸造/销毁/转移 NT。 |
| 🔴 致命 | **NT._getUser() 自动创建幽灵账号** | 任何 NT 操作传入不存在的 userId → 静默创建新用户，不报错。 |
| 🟡 严重 | **密码哈希算法非标准** | 自定义 DJB 风格 2000 次迭代哈希，非 bcrypt/scrypt/argon2。使用**硬编码全局 salt** (`'N4nT4nG_v2_$4Lt'`，utils.js:141)，非每用户独立 salt，且存储于源码中。 |
| 🟡 严重 | **file:// 协议无安全上下文** | `crypto.randomUUID()` 在非安全上下文失败，回退到 `Math.random()`。无法使用 HttpOnly cookie、Secure flag、CORS 限制。 |
| 🟡 严重 | **postMessage bridge 虽有 nonce 但 origin 检查弱** | `e.origin !== 'null'` 在 file:// 下总是 'null'，任意本地 HTML 可发送消息。nonce 缓解但不能根除。 |
| 🟡 严重 | **approveTx 的 tx.user 未验证存在性** | 即使 UI 层有 admin 检查，如果 tx.user 不存在于 NT 系统，`NT._getUser()` 会**自动创建幽灵用户**并充值/提现。 |
| 🟢 中等 | **升级角色跳过 builder 前置检查** | `upgradeRole()` 传 `skipAdventurerCheck: true` 硬编码，绕过冒险者身份门槛。 |

### 2.3 已实施的缓解措施

| 措施 | 位置 | 效果 |
|------|------|------|
| FIX-04: `approveTx`/`rejectTx` admin 守卫 | nantang-mobile.html | UI 层拦截非管理员调用 |
| FIX-06: `topUp`/`cashOut` txId 去重 | nt-core.js | 防止管理员双击重复到账 |
| FIX-08: bridge nonce 验证 | nantang-mobile.html | 缓解 file:// 下 postMessage 伪造 |

### 2.4 剩余的缺口

1. **NT.transfer()** — 无任何权限检查，任意用户可划转任意账户
2. **NT.earn() / NT.spend()** — 无权限检查，但金额上限 10000 在 bridge 层有拦截
3. **NT.createTask()** — 依赖调用方传入 posterId，不验证
4. **NT.verifyTask()** — 不验证 verifierId 是否为任务审核人
5. **Console 直接调用** — file:// 协议下 F12 是开放的，无法阻止
6. **approveTx 静默跳过** — `if(window.NT){...}` 无 else 分支。若 NT 加载失败，交易状态标记为 `approved` 但实际未执行 NT 余额变动，UI 状态与实际余额不一致。

##三、用户隔离

### 3.1 隔离机制

当前系统依赖一个全局字符串 `CURRENT_USER` 来判断"当前是谁"。所有操作通过对比 `CURRENT_USER` 与数据中的用户名字段来判断权限。

```javascript
// claimTask 中的典型模式
if(t.publisher===CURRENT_USER){showToast('不能领取自己发布的任务','error');return}
```

### 3.2 跨用户污染向量

| 函数 | 污染方式 | 严重度 |
|------|---------|--------|
| **doSubmit()** | 找 `x.name === CURRENT_USER` 然后写 submission。CURRENT_USER 错误 → 替别人提交 | 🔴 |
| **claimTask()** | 写 `{name: CURRENT_USER}` 到 claimants。CURRENT_USER 错误 → 替别人认领 | 🔴 |
| **reviewTask()** | 调 `NT.verifyTask(taskId, CURRENT_USER, true)` 释放托管 NT | 🔴 |
| **confirmSettle()** | 设 `settler = CURRENT_USER` + 自动升级访客为共建者 | 🔴 |
| **saveProfileEdits()** | 同时写三家存储，都基于 CURRENT_USER 索引 | 🟡 |
| **approveTx()** | 管理员审批充提，tx.user 不验证存在性 | 🟡 |
| **NT.transfer()** | `fromId` 参数不验证与当前用户的关系 | 🔴 |

### 3.3 注销泄露清单

| 泄露项 | 影响 |
|--------|------|
| `data` 全局对象 | 下一个登录用户调用 `getPayableBills()` 或 `calcNtByScope()` 时读取到前用户数据 |
| `_avatarSeedPool` | 前用户的随机生成头像种子出现在下一个用户的头像选择器中 |
| `MOCK_ITEMS` | 物品列表残留 |
| `NT._users/_tasks/_ledger` | NT 设计为共享账本，但内存副本不随注销刷新 |
| `nt_session` localStorage | 桌面版 `logoutUser()` (auth.js:398) 会 `removeItem`，但移动版 `logout()` (nantang-mobile.html) 只清 `nt_app_v2`，不碰 `nt_session`。若未来加自动恢复登录，将复活旧会话。 |
| `MOCK_ITEMS` IIFE | 脚本加载时执行 `(function(){ if(AppData&&...CURRENT_USER) return...; return []; })()`。此时 `CURRENT_USER=''`，永远返回空数组。用户添加的物品（`_saveItems` 写 `AppData._data.myItems`）在 `renderItemsInTab` 中被空的 `MOCK_ITEMS` 覆盖，导致物品无法正确显示。 |

## 四、数据完整性

### 4.1 任务状态机

当前任务状态跳转散落在 10+ 个内联 onclick 处理函数中，没有统一状态机：

```
                    ┌──────────┐
                    │  draft   │  (草稿, 仅发布者可见)
                    └────┬─────┘
                         │ doPublish()
                    ┌────▼─────┐
                    │  active  │  (进行中, 可被认领)
                    └────┬─────┘
                         │ claimTask()
                    ┌────▼─────┐
                    │ claimed  │  (已认领, 等待提交)
                    └────┬─────┘
                         │ doSubmit()
                    ┌────▼─────┐
                    │submitted │  (待审核)
                    └────┬─────┘
                    ┌────┴────┐
             通过    │         │   打回
                    ▼         ▼
            ┌──────────┐  ┌──────────┐
            │ reviewed │  │ rejected │──→ 重新提交 → submitted
            │ (待结算) │  │ (退回修改)│
            └────┬─────┘  └──────────┘
                 │ confirmSettle()
            ┌────▼─────┐
            │ settled  │  (已结算, 终态)
            └──────────┘

异常分支:
  active/caimed ──withdrawTask()──→ cancelled (已取消, 退还托管NT)
  submitted ──disputeTask()──→ disputed (争议中, 需管理员仲裁)
  cancelled/disputed ──不可逆──→ (终态)
```

**三层状态映射表**（NT 核心 → 提议统一 → UI 中文）：

| NT 层 (英文) | 提议统一状态 | UI 层 (中文) | 说明 |
|-------------|------------|-------------|------|
| — | `draft` | 草稿 | 仅 UI 层，未进入 NT |
| `pending` | `active` | 进行中 | 已发布，无人认领 |
| `active` | `claimed` | 待提交 | 已认领，等待提交成果 |
| `completed` | `submitted` | 待审核 | 已提交，等待审核 |
| — | `rejected` | 退回修改 | 审核打回，可重新提交 |
| `verified` | `reviewed` | 待结算 | 审核通过，等待释放 NT |
| `settled` | `settled` | 已结算 | NT 已释放，终态 |
| `cancelled` | `cancelled` | 已取消 | NT 已退还发布者，终态 |
| `disputed` | `disputed` | 争议中 | 需管理员仲裁，终态 |

> **注意**：当前代码中 UI 层存在 `已完成` 状态，与 `待结算` 等价混用（10+ 处 `\|\|` 判断）。提议统一后消除 `已完成`，只保留 `reviewed`（待结算）。

发现的状态流转漏洞：

- FIX-10 修复前，confirmSettle 可对任何状态的任务调用
- FIX-03 修复前，verifyTask 只接受 'completed' 状态，导致正常流程卡死
- editTask() 直接覆盖任务字段，不检查当前状态

### 4.2 任务存储分叉

- `AppData._data.tasks[name]` — 新任务走这条路径
- `data.tasks[]` — legacy mobile-bundle 任务
- `getPayableBills()` 只读 `data.tasks[]`，看不到新任务
- 两套任务系统互不同步

### 4.3 财务账本分叉

- `NT._ledger[]` → nt_core_v1（FIX-01/02 后才正确同步）
- `data.finance[]` → camp_data（通过 recordTransaction 写入）
- 两套账本独立运行，结算一个任务可能只在一套账本中记录

### 4.4 已修复的完整性问题

| 修复 | 问题 |
|------|------|
| FIX-01: doSubmit → NT.submitTask | 任务提交后 NT 状态停留在 active，奖励永久冻结 |
| FIX-02: claimTask → NT.acceptTask | 认领后 NT 不知道 assignee 是谁 |
| FIX-03: verifyTask 状态放宽 | 只接受 completed 导致正常流程断裂 |
| FIX-05: 冻结NT poster视角修正 | assignee 视角导致冻结栏永远显示 0 |
| FIX-06: txId 幂等去重 | 管理员双击审批 → 双倍到账 |
| FIX-07: 即时写盘 + beforeunload | 关键操作后关标签 → 状态丢失 |

## 五、输入校验缺口

| 函数 | 缺失校验 | 风险 |
|------|---------|------|
| `NT.transfer(fromId, toId, amount, reason)` | fromId 不验证就是调用者；reason 不限长 | 🔴 任意转账 |
| `NT.topUp(userId, amount, reason, txId)` | 仅 UI 层 admin 守卫 | 🔴 无限铸币 |
| `NT.cashOut(userId, amount, reason, txId)` | 同上 | 🔴 无限销毁 |
| `doSubmit(name)` | photo/file URL 不校验协议 | 🟡 XSS 风险 |
| `saveProfileEdits()` | bio/location/wallet 不限长 | 🟡 localStorage 配额耗尽 |
| `submitRecharge()/submitWithdraw()` | amount 无上限 | 🟡 请求天文数字 |
| `showInvitePrompt()` | 邀请码消费成功后才调 changeUserRole，失败则码已丢失 | 🟡 失败域顺序问题 |
| `claimTask(name)` | name 直接当对象 key，无 prototype 防护 | 🟢 原型污染（理论） |
| `confirmReject(name)` | `reviewReason` 不限长，传入 `NT.verifyTask()` 存为 `disputeReason` | 🟡 localStorage 配额耗尽 |
| `renderProfile()` | `CURRENT_USER`、`wallet`、`bio`、`location`、`uid` 未经过 `esc()` 直接写入 `innerHTML` | 🔴 存储型 XSS |

### 5.1 特别关注：renderProfile() 存储型 XSS

`renderProfile()` 函数（nantang-mobile.html）在渲染用户资料卡时，将以下字段直接拼入 `innerHTML`，未经过 `esc()` 转义：

- `CURRENT_USER`（用户名）
- `u.wallet`（钱包地址）
- `u.bio`（个人简介）
- `u.location`（坐标）
- `u.uid`（用户 ID）

编辑表单的 `value="..."` 也未做属性转义——引号可突破属性边界注入事件处理器。

`esc()`（utils.js:163）已在任务名称等处使用，但资料卡渲染路径遗漏。如果攻击者在自己的简介中存入 `<img src=x onerror=alert(1)>`，任何查看其资料卡的用户都会执行该脚本。在 file:// 协议下，这意味着可以读取该用户的所有 localStorage 数据。

此外，页面未设置 Content-Security-Policy（无 `<meta http-equiv="CSP">` 标签），缺少 XSS 防御纵深。

### 5.2 AppData._save() 防抖丢数据风险

AppData（app-data.js）的 `_save()` 使用 200ms 防抖写入 localStorage，但没有像 NT 核心那样的即时写盘支持和 `beforeunload` 兜底：

- `_save()` 只有 `setTimeout(..., 200)` 一条路径，无 `immediate` 参数
- 无 `beforeunload` 监听器
- 用户在 `addTask`/`updateTask`/`saveProfileEdits` 后 200ms 内关标签 → AppData 变更丢失
- NT 核心数据（余额、账本）因 FIX-07 已受保护，但 AppData 数据（UI 状态、营地、日记、邀请码）不受保护

**升级路径**：为 `AppData._save()` 加 `immediate` 参数，关键操作调用 `_save(true)`；加 `beforeunload` 守卫。

### 5.3 MOCK_ITEMS 初始化时机错误

`MOCK_ITEMS` 在脚本加载时通过 IIFE 初始化（nantang-mobile.html:710），此时 `CURRENT_USER` 为空字符串。之后 `renderItemsInTab()` 使用 `MOCK_ITEMS` 而非 `AppData._data.myItems[CURRENT_USER]`。用户添加的物品虽被 `_saveItems()` 写入 AppData，但渲染时读的是启动时初始化的空数组。此 bug 导致物品功能完全不可用。

### 6.2 选型理由

| 组件 | 选择 | 理由 |
|------|------|------|
| 运行时 | Node.js 20+ | 团队已有 JS 技术栈，前后端同语言降低维护成本 |
| 框架 | Express.js | 生态最大、中文资源丰富、与现有 middleware 模式兼容 |
| 数据库 | SQLite (better-sqlite3) | 单文件零运维，50 用户规模同步读写足够；迁移路径见 §八 |
| 认证 | bcrypt + JWT (jsonwebtoken) | 替代当前的自定义哈希 + CURRENT_USER 字符串；无需外部认证服务 |
| 会话 | 无状态 JWT + httpOnly refresh cookie | 不需要 Redis 存储会话；refresh 轮转解决 token 泄露问题 |
| 部署 | 单 VPS + Caddy | $5/月；Caddy 自动 Let's Encrypt 证书，零配置 HTTPS |
| 文件 | multer + 本地目录 | 50 人规模不会产生有意义的存储量；Phase 2 迁 OSS |
| 校验 | zod | 类型推导 + 运行时校验一套代码，替代手写 if 判断 |

### 6.3 现有 schema 引用

当前 `schema.js` 定义了 `camp_data` 的完整字段结构（`tasks[]`, `members{}`, `finance[]`, `inventory[]`, `activity_log[]` 等 30+ 字段）。迁移到 SQLite 时，每张表的字段定义应从此文件推导，而非重新设计。新表只需补充 schema.js 中不存在的实体：`audit_log`, `sessions`。

### 6.4 数据架构设计模式

当前代码隐含使用了三种标准模式，但未显式命名。显式采用这些模式的共同语言有助于团队对齐：

| 模式 | 定义 | 本项目中的应用 |
|------|------|---------------|
| **Pattern A: Single Source of Truth（单一真相源）** | 每种实体只有一份权威存储，其他"存储"都是衍生视图 | `getUsers()` (nt_users) 应作为用户身份唯一真相源；`NT` (nt_core_v1) 作为财务唯一真相源；`AppData` (nt_app_v2) 作为 UI 状态真相源。其他存储全部删除 |
| **Pattern B: Domain-Driven Data Ownership（领域数据所有权）** | 每个模块独占其数据，其他模块通过 API 访问而非直接读写 | auth.js 拥有身份数据，NT 拥有财务数据，AppData 拥有应用数据。跨模块只用 ID 引用，渲染时按需 join |
| **Pattern C: Event Sourcing / CQRS（事件溯源）** | 不存当前状态，存事件日志；当前状态是事件日志的投影 | `NT._ledger[]` 已经是事件日志。所有余额可通过重放账本推导。可扩展到任务状态：`task_created → task_claimed → task_submitted → task_approved → task_settled` |

**推荐**：当前项目最适合 Pattern A。Phase 0 的存储合并本质上就是建立单一真相源。

## 八、关键决策清单

| 决策 | 建议 | 为什么现在决定 | 可推迟？ |
|------|------|---------------|---------|
| **userId: 整数 PK vs 名称** | 整数 PK | 名字会改、有 Unicode 碰撞。影响所有查询和 API 设计 | ❌ 必须现在 |
| **任务状态机统一** | `draft → active → claimed → submitted → reviewed → settled`（含 `rejected` 退回修改、`cancelled` 取消、`disputed` 争议分支） | 当前散落在 onclick 里，加后端时无法 enforce | ❌ 必须现在 |
| **存储合并** | 8→3（auth + NT + AppData） | 每次加功能都加重碎片化 | ❌ 必须现在 |
| **离线能力** | 保留 PWA + Service Worker | 乡村社区网络不稳定 | ✅ 可推迟到 Phase 2 |
| **Redis 缓存** | 不需要 | 50 用户 SQLite 足够 | ✅ Phase 1 验证后决定 |
| **WebSocket 实时** | 不需要 | 页面刷新足够 | ✅ 需要协作功能时再加 |
| **PostgreSQL** | SQLite 足够 | 50 用户不需要 | ✅ >100 并发写时考虑 |

### 9.1 遗留数据迁移决策

`data.finance[]`（camp_data）与 `NT._ledger[]`（nt_core_v1）的双账本问题，Phase 1 迁移时需决定：

| 选项 | 做法 | 风险 |
|------|------|------|
| **导入 NT 账本** | 将 `data.finance[]` 历史记录批量插入 `NT._ledger[]` | 需重算会计等式；重复条目风险 |
| **只读归档** | `data.finance[]` 保留为只读历史，新交易只走 NT | 两套数据需要不同的查询路径 |
| **直接删除** | MVP 阶段数据量小，废弃遗留账本 | 丢失历史财务记录 |

推荐**只读归档**：`data.finance[]` 保留但不再写入，新交易全部走 NT 账本。迁移前做一次差额核对。
