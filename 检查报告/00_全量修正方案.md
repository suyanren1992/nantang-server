## 六、不改的项（有意的取舍）

| 问题 | 理由 |
|------|------|
| `innerHTML` 全量重绘 | 当前数据量(10-20条)下性能完全够用，引入虚拟滚动得不偿失 |
| 死代码 `toggleCard`/`openSub`/`getTask` | 删除有引号匹配风险，保留无害 |
| `confirm` 双向 postMessage | 当前原生 confirm 功能可用，暂不投入 |
| Firefox file:// 兼容 | 用户群用 Chrome/Edge |
| `will-change: transform` 多余 | 无性能影响，不改不坏 |
| NT `disputeTask` 陪审团 | 结算方案标注 Phase 3，MVP 只需状态标记 |
| NT `deposit()/withdraw()` 激活 | 等钱包/合约接入后再接，当前桩代码保留 |

## 七、NT 系统数据流（参考）

```
地图端操作                       游戏端
─────────                      ─────────
_post({type:'ntEarn',         message handler
  amount:20,           →      NT.earn(CURRENT_USER, 20, ...)
  reason:'打扫厕所'})            │
                                ├─ user.ntBalance += 20
                                ├─ LEDGER 写一条流水
                                ├─ SYSTEM_POOL -= 20
                                └─ NT.verify() 等式仍然成立

结算操作（confirmSettle）
t.nt = 15                     LEDGER.personal.balance += 15  （旧UI）
                              NT.earn(user, 15, ...)          （新数据层）
                              → 两套账本同步更新
                              → NT.verify() 验证通过
```
