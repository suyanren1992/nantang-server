### 1.2 具体哪里断了

| 用户操作 | 期望 | 实际 | 问题根源 |
|---------|------|------|---------|
| 在地图点「厨房打扫」 | 打开任务大厅，看到厨房打扫任务 | 没反应 | 地图调用了不存在的 `Game.openTask()` |
| 在地图点返回按钮 | 回到村口 | 回到地图首页 | `backToVillage()` 调了不存在的 `parent.showVillageSquare()` |
| 在地图打扫后 toast | 和主应用一样的绿色 toast | 地图自己弹了 alert | 地图调了不存在的 `Game.toast()` |
| 看地图上的用户名 | 显示登录用户 | 永远显示"砚仁" | `_me()` 的 fallback 是写死的 |
| 地图上的 NT 操作 | 更新主应用的 NT 余额 | 只在 toast 里出现，余额不变 | 没有调用主应用的账本 |

**一句话总结：地图以为外面有个 `Game` 对象提供一切，但外面什么都没给——所以地图的按钮和提示全成了摆设。**

## 二、改完什么样（逐页对比）

### 2.1 全局：数据通了

```
┌── nantang-mobile.html（主应用）───────────────┐
│                                                │
│  ┌──────────────────────────────────────┐      │
│  │  window.Game = {                     │      │
│  │    getUser()    → 当前登录用户         │      │
│  │    getData()    → 建筑/空间/市集数据    │      │
│  │    openTask()   → 关闭地图，开任务大厅   │      │
│  │    openMe()     → 关闭地图，开工作台     │      │
│  │    openMarket() → 关闭地图，开物品tab    │      │
│  │    toast()      → 统一提示             │      │
│  │    confirm()    → 统一确认弹窗          │      │
│  │    logActivity()→ 写活动记录           │      │
│  │    earnNT()     → 赚NT（更新账本+记录） │      │
│  │    spendNT()    → 花NT（更新账本+记录） │      │
│  │    getMyNT()    → 查余额               │      │
│  │    getMarketItems() → 按地点查市集物品   │      │
│  │  }                                   │      │
│  └──────────────┬───────────────────────┘      │
│                 │ parent.Game                   │
│  ┌──────────────┴───────────────────────┐      │
│  │  ACTIVITY_LOG []                     │      │
│  │  LEDGER { personal, camp, community }│      │
│  │  TASKS {}                            │      │
│  │  MOCK_ITEMS []                       │      │
│  └──────────────┬───────────────────────┘      │
│                 │ iframe 共享 parent.Game       │
├─────────────────┼──────────────────────────────┤
│  ┌──────────────┴───────────────────────┐      │
│  │ iframe: 地图 app.js                   │      │
│  │                                       │      │
│  │ 第一行: window.Game = parent.Game     │      │
│  │                                       │      │
│  │ 之后所有 Game.xxx() 调用 → 都通 ✅     │      │
│  │ 返回按钮 → parent.closeOverlay ✅     │      │
│  └───────────────────────────────────────┘      │
│                                                │
│  ≡≡≡ 一个数据源，两边读写 ≡≡≡                    │
└────────────────────────────────────────────────┘
```

**关键决策：主应用是唯一的数据主人。地图不存任何状态，只是主应用的「另一块屏幕」。**

这样做的**好处**：
- 永远不会出现「地图里的 NT 和账本里的 NT 对不上」
- 以后换成小程序/App/Web，只要实现同一套 `Game` API，地图就能适配任何外壳
- 地图代码可以独立部署、独立调试（`parent === window` 时走自己的 mock 数据）

### 2.5 NT 的流动：从「地图操作」到「账本更新」

```
                        改前
地图操作（打扫厕所 +20NT）
        │
        ▼
  Game.toast('+20 NT')  ← toast 弹了一下
        │
        ▼
  💨 消失了。账本余额不变。第二天用户打开工作台，NT 数量完全没变。


                        改后
地图操作（打扫厕所 +20NT）
        │
        ├──► Game.earnNT(20, '打扫厕所', 'community')
        │         │
        │         ├──► LEDGER.personal.balance += 20   ← 余额变了
        │         ├──► LEDGER.personal.records 写入     ← 账本有记录
        │         ├──► ACTIVITY_LOG 写入               ← 活动记录有痕迹
        │         └──► Game.toast('+20 NT · 打扫厕所') ← toast 仍然弹
```

**这意味着**：用户在地图上的每一次劳动（打扫、农活、管理厨房）都会在主应用留下记录。下次打开工作台，不仅余额对得上，还能看到「我今天做了什么、在哪里做的、赚了多少 NT」。地图不再是「用了就忘」的孤岛。

## 三、为什么这样设计——每个决策的理由

### 决策 1：iframe + Game 桥接，而不是全内联

**好处**：
- CSS/HTML 彻底隔离——地图的样式永远不会影响主应用，反亦然
- 地图可以独立打开、独立调试（mock 数据仍然生效）
- 以后地图升级换代，只要 Game API 不变，主应用不需要改

**坏处**：
- iframe 内无法使用主应用的 CSS 变量（颜色、字体），需要地图自己维护一套视觉风格
- postMessage 通信比直接函数调用多一层序列化开销（目前影响可忽略——地图的 Game API 调用频率很低）

**为什么不选 Web Component / 微前端 / 同源内联**：
- Web Component 需要改造地图的全部 DOM 生成逻辑——改动量太大（几千行 app.js）
- 微前端框架（qiankun 等）引入了一个新依赖 + 构建工具链——当前项目是纯 HTML/JS，没有构建步骤
- 上次同源内联的教训：两个庞大的 CSS 体系撞在一起，一个引号处理不对全崩

### 决策 2：主应用是唯一数据源（地图不持有状态）

**好处**：
- 永远不会出现数据不一致——只有一个 LEDGER，只有一个 TASKS，只有一个 ACTIVITY_LOG
- 以后换数据库（从内存 mock 换成 localStorage / IndexedDB / 后端 API），只改 Game API 的实现，地图零改动
- 离线场景下，所有数据在主应用 localStorage，地图 iframe 不需要单独存

**坏处**：
- 地图每次渲染都要通过 Game API 读数据（`getMarketItems`、`getMyNT` 等）——但这些调用频率很低（用户切换房间时才触发），性能影响可忽略
- 如果 Game API 的某个函数抛异常，地图的对应功能就挂了——需要做好 try/catch + fallback

### 决策 3：两期分开

**第一期（~15 行）**：基础桥接。让 `Game.openTask`、`Game.toast`、`Game.confirm`、`Game.getUser` 工作 + `backToVillage` 修复。用户能用的最基础功能：返回村口、点任务链接有效、名字显示正确。

**第二期（~150 行）**：双向互通。市集展示、活动记录、NT 互通、工作台入口。这些依赖于第一期的基础桥接已经稳定。

**为什么分开**：
- 第一期改动极小（15 行），出问题的概率极低，可以立刻上线验证
- 如果第一期有问题（比如某个 iframe 安全策略导致 parent.Game 读不到），排查范围只有 15 行
- 第二期依赖第一期的 `Game` 对象已经是稳定的——不会出现「花了 150 行发现基础桥接就有问题」
- **ponytail：最小改动先上线，验证通过后再叠第二期。不要把鸡蛋放在一个 commit 里。**

### 决策 4：活动记录用内存数组而不是 localStorage

**好处**：
- 零依赖，和现有 `LEDGER`、`TASKS`、`MOCK_ITEMS` 的数据模式一致
- 不需要序列化/反序列化逻辑
- 页面刷新即清空——符合当前「演示/原型」阶段的需求

**坏处**：
- 刷新页面活动记录就没了——不是持久化存储
- 长期来看需要换成持久化方案

**什么时候升级**：当用户开始说「我昨天的活动记录怎么没了」的时候，把 `ACTIVITY_LOG` 迁移到 `localStorage` 或后端。迁移成本：给 `logActivity` 函数加一行 `localStorage.setItem('activity_log', JSON.stringify(...))`。

## 四、风险和应对

| 风险 | 严重程度 | 发生条件 | 怎么兜底 |
|------|---------|---------|---------|
| iframe 跨域导致 `parent.Game` 不可读 | 🔴 高 | 地图部署到不同域名 | 地图第一行的 `parent.Game` 读取失败时，`window.Game` 为 undefined，地图的所有 `Game?.xxx?.()` 调用静默失败，走 fallback（硬编码数据 + alert）。地图仍然可用，只是数据不通。 |
| `Game` 对象中的函数引用了未定义的变量 | 🟡 中 | `nantang-mobile.html` 的 `<script>` 加载顺序变化，`CURRENT_USER` 等变量后移 | 在 `Game` 对象前加存在性检查：`if (typeof CURRENT_USER === 'undefined') return null` |
| 活动记录数组无限增长 | 🟡 中 | 用户频繁操作，几天不刷新页面 | `logActivity` 加长度上限：超过 500 条时删除最旧的 |

## 附录 A：与上一版审查的关系

上一版审查（`经济设计师审查_2026-07-14_v2.md`）关注的焦点是 NT 经济系统的平衡性。本方案的 **场景 D（NT 互通）** 直接关联：地图打扫/住宿/农活的 NT 定价应该在 `MGMT_DATA` 中配置（已有 ponytail 标注），并通过 `Game.earnNT` / `Game.spendNT` 流入 LEDGER。未来经济系统调整时，只需改主应用的定价配置 + 地图的 `MGMT_DATA`，数据流路径不变。

## 附录 B：与 `nantang-mobile-ui.js` 的关系

`nantang-mobile-ui.js:36-43` 已有一个 `window.Game` 对象（包含 `openTask`、`toast`、`confirm`、`getUser`、`getData`、`refresh`），功能与本方案的 Game API 重叠但不冲突。两个文件目前不共存（`nantang-mobile.html` 不加载 `nantang-mobile-ui.js`）。未来如果统一为单一 UI 实现，应优先采用 `nantang-mobile-ui.js` 的 Game 版本（它的 `openTask` 使用了 taskId 参数，可以导航到具体任务），并把本方案的 `openMe`、`getMarketItems`、`logActivity`、`earnNT`、`spendNT` 合并进去。
