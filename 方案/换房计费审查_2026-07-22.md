# 换房计费逻辑审查与优化方案

> 2026-07-22 · 审查结论：通过 · 2 项优化建议

---

## 一、当前逻辑审查

### 换房流程

```
POST /api/accommodation/checkin
  │
  ├─ 1. 检测已有 active tenancy？
  │   ├─ 是 → 旧房 checkout
  │   │   ├─ debt > 0 且余额够 → 扣款 + 回流 pool + 写账本 (debt_settlement)
  │   │   ├─ debt > 0 且余额不够 → 写账本 (debt_unpaid)，不扣款
  │   │   └─ debt = 0（清零）
  │   └─ 否 → 跳过
  │
  ├─ 2. 检查目标房间满员？
  │   └─ 满 → 400
  │
  └─ 3. 创建新 Tenancy
       ├─ checkin_date = now
       ├─ debt = 0
       ├─ last_deducted = NULL
       └─ status = active
```

### daily_tick 扣费流程

```
POST /api/system/daily-tick
  │
  ├─ pool.last_tick_date == today？ → skip（幂等）
  │
  └─ 遍历所有 active tenancy
       ├─ t.last_deducted == today？ → skip
       ├─ 查 room_id 对应 BED_RATES[room_id]
       ├─ 用户余额 >= rate → 扣款 + 回流 pool + 写账本
       ├─ 用户余额 < rate → t.debt += rate（累计欠费）
       └─ t.last_deducted = today
```

### 审查结论：✅ 通过

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 旧债是否结算 | ✅ | 换房时自动结算 debt，余额够则扣款，不够则记录 debt_unpaid |
| debt 是否清零 | ✅ | `old.debt = 0`，不残留在已 checkout 的 tenancy 上 |
| 新房计费起点 | ✅ | `checkin_date = now`，`last_deducted = NULL`，次日 daily_tick 开始扣 |
| 无重复扣费 | ✅ | 旧房已 checkout 不参与 daily_tick；新房从当天起算 |
| 房间价格匹配 | ⚠️ | 见下方优化项 |

## 二、优化项

### O1 — BED_RATES 硬编码 → 数据库/config 化

**现状**：[nt.py:960](server/routes/nt.py#L960)
```python
BED_RATES = {"dorm101":20,"dorm102":30,"dorm103":30,"dorm104":60,"dorm105":30,"dorm106":35}
```

**问题**：房间价格硬编码在代码里，和客户端 `accommodations.pricePerBed`（localStorage 种子数据）不一致时，用户看到的是 UI 价格，实际扣的是硬编码价格。管理员也无法动态调价。

**方案**：在 `map_locations` JSON 中读取每个房间的价格作为权威源，或建立独立的 `room_pricing` 配置表。

**两种实现路径**：

| 路径 | 改动量 | 说明 |
|------|--------|------|
| A（最小） | ~5行 | 从环境变量读 JSON：`BED_RATES = json.loads(os.environ.get("BED_RATES", '{...}'))`，管理员改价修环境变量重启 |
| B（完整） | ~30行 | 在 CommunityPool 或新表存 room_pricing JSON，提供 `PUT /api/admin/room-pricing` 端点，管理员在线调价，daily_tick 实时读取 |

**建议**：当前阶段用 A（环境变量），Phase E 后走 B。

**验证**：修改 BED_RATES 环境变量 → 重启 → daily_tick 按新价格扣费

### O2 — 换房时客户端 tenancy 同步

**现状**：[app.js](nantang-mobile/js/app.js#L1316) `_confirmCheckin` 换房分支
```javascript
if (oldRoom && oldBed) {
  var oidx = oldRoom.tenants.indexOf(oldBed);
  if (oidx >= 0) oldRoom.tenants.splice(oidx, 1);
}
```

**问题**：客户端只是 splice 了本地 `room.tenants` 数组，但如果 `API.request('/accommodation/checkin')` 失败（网络问题），本地已删旧记录但服务端未换房——数据不一致。且旧房间的 `debt` 未在客户端同步清零。

**方案**：
1. `doCheckin` 内先调 `API.request`，成功后再改本地数据
2. 失败时回滚（恢复旧 tenancy 数据）
3. 换房成功后调 `API.syncAll()` 或至少调 `GET /api/nt/sync` 刷新 all_tenancies

**验证**：断网换房 → toast 提示失败 → 本地数据不变；联网换房 → 成功 → 两房间入住人正确更新

**改量**：~15行
