## 议题一：公约——入住时要不要展示全文？

### 现状

入住确认弹窗只有一句话：`入住即表示同意遵守社区公约`。没有展示公约的实际内容。

全貌页有一个 📜 社区公约入口（`_openCovenantOverlay()`），但只展示**定价标准**（劳动定价、清洁定价等），不是真正的社区公约文本（行为准则、权利义务、退出机制等）。

### 问题

1. 公约文本本身不存在——代码里没有任何地方定义了公约的正文内容
2. 签署时用户看不到任何条款
3. 签署后信箱没有任何通知

### 建议

1. 在 `app.js` 或 `data.js` 中定义公约正文（结构化文本，可分章节）
2. 入住确认弹窗改为两步：
   - 第一步：展示公约全文（可滚动阅读）
   - 第二步：确认"我已阅读并同意"
3. 入住后向信箱发送一条通知：`📜 你已签署南塘社区公约 · 2026-07-22`

## 议题二：数据初始化——部署时要不要清空所有数据？

### 现状

- 用户数据存在服务器 SQLite 文件 + 客户端 localStorage
- 部署是增量更新（`git push → Render auto-deploy`）
- 数据库文件（`nantang_fresh.db`）不会被部署过程清空
- 客户端 localStorage 持久化所有操作数据（任务、校核、发现、流水）

### 问题

1. 反复测试后，数据库和 localStorage 积累大量垃圾数据
2. 无法区分"代码 bug"和"脏数据导致的异常行为"
3. 公测前的测试数据会影响正式用户的初始体验
4. 多人测试时互相看到对方的测试数据

### 阶段性策略建议

```
┌─ 当前阶段（内部测试）──────────────────────────────────┐
│ 每次部署 → 手动清空数据库 + 清除浏览器 localStorage    │
│ 方式：在 Render 上执行 SQL 命令或重新创建数据库         │
│                                                       │
│ 临时方案：加一个 /api/admin/reset-db 端点（仅 admin）  │
│           POST 后清空所有用户数据、任务、流水、校核     │
│           保留 admin 账户                              │
└───────────────────────────────────────────────────────┘
                         ↓
┌─ 公测阶段 ────────────────────────────────────────────┐
│ 关闭 reset 端点，用户数据正式保留                      │
│ 部署不再清数据（只做数据库迁移）                       │
│ 加数据库备份（Render 自带 Postgres 备份）              │
└───────────────────────────────────────────────────────┘
```

## 议题三：刷新后用户信息丢失（变成重新注册）

### 诊断

**根因：Cookie `secure` flag 缺失。**

```python
# auth.py:38 — 当前
response.set_cookie("nt_rt", token, httponly=True, samesite="lax", max_age=7*86400, path="/")
# ← 缺少 secure=True
```

Render 生产环境是 HTTPS。浏览器在 HTTPS 下**拒绝**不带 `secure` flag 的 cookie。所以 `nt_rt`（refresh token cookie）永远不会被浏览器保存。

**刷新页面的流程：**

```
刷新 → autoLogin() → silentRefresh()
       → fetch /api/auth/refresh (credentials:'include')
       → 浏览器不发送 nt_rt（cookie 被拒绝）
       → 服务端返回 401（无 refresh token）
       → silentRefresh 回调 ('expired', null)
       → 显示登录页 → 用户以为"重新注册了"
```

**这也是为什么 Render 部署后一切都不正常的原因之一——用户状态无法持久化。**

### 建议

改 Cookie 设置：
```python
response.set_cookie("nt_rt", token, httponly=True, secure=True, samesite="lax", max_age=7*86400, path="/")
```

有了这条修复，刷新后 silentRefresh 能正常工作，用户自动恢复登录状态。

## 议题四：南塘时间线——放哪里？

### 现状

`📜 南塘时间线` 在社区副本页（overlayCommunity）底部。

### 用户意见

应该放到档案室（archive room），社区页应该只展示营地和活动。

### 分析

- 档案室已有 activity_log 系统（`ui-archive.js`），记录了所有操作日志
- 时间线的数据来源应该统一到档案室的 activity_log
- 社区页的时间线目前显示的是校核记录（discoveries），不是真正的时间线

### 建议

1. 从社区页删除时间线 section
2. 在档案室中强化时间线视图（按时间排序的 activity_log）
3. 社区页只保留：营地列表 + 活动公告

## 议题五：营地权限——管理者应该看到所有信息

### 现状

营地代码（`ui-camp.js`）中多处权限检查硬编码了 `admin`：
```javascript
var isBuilder = role==='builder'||role==='admin';  // 共建者+管理员
if (role==='admin') return true;                     // 仅管理员
```

但 `builder`（共建者/营地管理者）没有被授予查看所有营地信息的权限。

### 问题

1. 营地创建者（可能不是 admin）看不到自己营地的完整管理面板
2. 角色权限不统一：有些地方 admin+builder，有些地方只要 admin
3. `getUsers()[name].role` 在离线模式可能返回过期的角色

### 建议

1. 统一营地权限：营地创建者 + admin + builder 都能完整管理
2. 权限判断集中到一个函数：`canManageCamp(campId, user)`
3. 权限检查逻辑：`ctx.created_by === user.id || user.role === 'admin' || user.role === 'builder'`

## 议题六：5 个校核始终存在——是不是硬编码？

### 诊断

**不是硬编码。** 校核来自用户操作：

| 触发操作 | 代码位置 |
|---------|---------|
| 收割作物 | app.js:634 field_harvest |
| 清理容器 | app.js:996 daily_container |
| 放入物品 | app.js:1879 stock_in |
| 消耗物品 | app.js:1883 stock_out |
| 打扫房间 | app.js:1976 cleaning |
| 劳动上报 | ui-cardroom.js:1070 labor_report |
| 田间操作 | app.js:2373 field_action |
| 卡片室校核 | app.js:2347 |

**这些是真实用户操作产生的，不是硬编码种子数据。** 它们被存入 `AppData._data.pendingVerifications` → localStorage → 跨会话持久化。

### 为什么看起来"始终存在"

1. localStorage 持久化——不清除就一直在
2. 如果未校核（pending），它们会一直显示
3. 反复测试产生大量未校核记录

### 建议

1. 测试阶段定期清除 localStorage 或使用隐私模式
2. 部署时如果有 reset-db 端点，一并清客户端数据
3. 加一个"一键清理测试数据"按钮（开发阶段专用）

## 议题七：测试策略——已经上线 + 自动部署，怎么安全测试？

### 当前约束

- Render 自动部署（push → 1-2分钟 → 线上生效）
- 有真实用户可能访问
- 没有 staging 环境
- 数据库和线上是同一个

### 建议方案

```
┌─ 短期（本周）──────────────────────────────────────┐
│ 1. 关闭 Auto-Deploy：Render Settings → Off         │
│ 2. 改为手动部署：测完确认无误 → Manual Deploy        │
│ 3. 本地用私密窗口 + http://localhost:8080 测试      │
│ 4. 部署前清一次数据库（reset-db）                    │
└─────────────────────────────────────────────────────┘
                         ↓
┌─ 中期（公测前）────────────────────────────────────┐
│ 1. 建第二个 Render 服务作为 staging                  │
│ 2. staging 连独立数据库（或 SQLite 副本）            │
│ 3. 测试通过后手动 merge → deploy production          │
└─────────────────────────────────────────────────────┘
```

**最小可行方案**：先把 Auto-Deploy 关了，改为手动部署。每次部署前在本地验证完再推。
