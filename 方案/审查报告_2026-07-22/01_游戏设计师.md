## 🔴 严重（4 条）

### 1. resolve_task: zero escrow_amount orphans pool.task_escrow
{file: `server/routes/nt.py`, line: 661-665}

`resolve_task` 读 `escrow = task.escrow_amount or 0`。若 `escrow_amount` 为 0（DB 迁移、手动编辑、或之前部分操作残留），`pool.task_escrow -= 0` 为无操作。但创建时注入的托管金永久锁定。`if escrow <= 0: raise HTTPException` 守卫只拦负数，0 穿过。

**建议**：`escrow_amount <= 0` 时 raise 409/400，强制管理员排查。

### 2. verify_task reject (3+): same zero-escrow orphan risk
{file: `server/routes/nt.py`, line: 513-526}

同上——`task.escrow_amount` 为 0 时 `pool.task_escrow -= 0` 无操作。创建时从发布者扣的托管金留在 `pool.task_escrow` 中永久锁定。

**建议**：加 `if task.escrow_amount <= 0: raise HTTPException(409, "任务无托管金额")`。

### 3. No check preventing poster from accepting their own task
{file: `server/routes/nt.py`, line: 433-454}

`accept_task` 校验任务状态和满员但不检查 `task.poster == user.id`。发布者可自领→提交→审核（设自己为 reviewer 或勾结审核人）。净 NT 流为零和（拿回自己冻结的），但获得 **+8 trust score**（+3 poster + +5 assignee）。可信任度刷分。

**建议**：`if task.poster == user.id: raise HTTPException(400, "不能认领自己发布的任务")`。

### 4. Client-server verifyTask reject escrow behavior mismatch
{file: `nantang-mobile/js/nt-core.js`, line: 246-256 vs `server/routes/nt.py`, line: 508-526}

客户端 `verifyTask(taskId, verifierId, false)` 立即设 status="disputed" 并退款所有 escrow。服务端 rejection 设为"退回修改"保持 escrow 冻结——只在第 3 次 rejection 才退款（status 变"已取消"）。file:// → HTTP 过渡时 escrow 状态冲突。

**建议**：对齐客户端行为——reject 时保持冻结，加 `reject_count`，只在第 3 次释放。

## 🟡 重要（7 条）

### 5. "待提交" state is dead — never set by any endpoint
{file: `server/models.py` TASK_STATUSES + `server/routes/nt.py`, line: 466}

状态"待提交"存在于 `TASK_STATUSES` 且在 `verify_task` 白名单中，但无代码路径设置为该状态。`submit_task` 只在所有 assignee 提交后才过渡到"待审核"。部分提交时状态不变。

**建议**：要么实现部分提交态，要么从状态词汇表中移除。

### 6. CampTask and NTTask are two disconnected task systems
{file: `server/routes/camps.py`, line: 64-71; `server/routes/tasks.py`, line: 80-121}

营地创建产 `CampTask` 行（status="draft"），无端点推进到 `NTTask`。营地结算/报表筛选 `status="已结算"` 永远为空。

**建议**：(a) 加端点将 CampTask 提升为 NTTask，或 (b) 用 `NTTask + camp_id` 外键替代 CampTask。

### 7. camp_balance accumulates but is never consumed for task rewards
{file: `server/routes/camps.py`, line: 83-89; `server/routes/nt.py`, line: 456-528}

营地创建向 `pool.camp_balance` 注资，但任务 verify 永远从 `pool.task_escrow` 付款（由 NTTask 创建时从 user balance / pool.balance 注入）。营地预算滞留 `camp_balance`。

**建议**：营地关联的 NTTask 创建时从 `pool.camp_balance`（非 `pool.balance`）扣款。或营地结算时扫回未用资金。

### 8. delete_task lacks explicit `poster=='社区'` branch
{file: `server/routes/tasks.py`, line: 149-176 vs `server/routes/nt.py`, line: 543-554}

`cancel_task` 有显式 `poster=="社区"` 分支退款到 `pool.balance`。`delete_task` 无——它查 `User.id=="社区"` 返回 None，巧合性地 fall through 到 `pool.balance`。若某天创建了 id="社区" 的 User 行，退款会错误流入该用户。

**建议**：像 cancel_task 一样加显式分支。

### 9. verify approve vs reject use different escrow calculation bases
{file: `server/routes/nt.py`, line: 481 vs line: 516}

Approve: `pool.task_escrow -= task.reward * len(assignee_ids)` + `-= unclaimed` → `reward * slots`。
Reject (3+): `pool.task_escrow -= task.escrow_amount`。
若 `escrow_amount != reward * slots`（手工编辑、未来代码改动），会计静默破裂。

**建议**：escrow 操作前加断言或 log warning。

### 10. No per-assignee verification for multi-slot tasks
{file: `server/routes/nt.py`, line: 457-528}

`verify_task` 对多人任务全有或全无。3 个 assignee 中 2 个做好 1 个做差，审核人只能全批（多付）或全退（惩罚好的）。evidence map 已按用户存——基础设施存在。

**建议**：verify 接受 per-assignee 裁决 `{user_id: true/false}`。

### 11. Hall mode: client caches unfiltered tasks after initial fetch
{file: `server/routes/tasks.py`, line: 54-60; `nantang-mobile/js/core.js`, line: 284-355}

服务端 hall 模式返回"进行中"任务（非 onsite 过滤 `is_system_generated`）。客户端合并到本地缓存后，后续 `filterQuests()` 操作全量缓存——状态变化（如取消）的任务仍出现在 hall 中，直到下次服务端拉取。

**建议**：客户端的 hall 模式 filter 应与服务端过滤器一致，或每次打开 hall 重新拉取。

## 🟢 建议（3 条）

### 12. nt-core.js cancelTask: '社区' poster auto-registers phantom user
{file: `nantang-mobile/js/nt-core.js`, line: 261-281}

`cancelTask` 调用 `_getUser(t.poster)` 自动注册未知 ID。若 `t.poster === '社区'`，创建名为"社区"的用户。Escrow 退款流向该幽灵用户非社区池。

### 13. nt-core.js submitTask: only checks single assignee for multi-slot tasks
{file: `nantang-mobile/js/nt-core.js`, line: 194-202}

客户端 `submitTask` 只看 `t.assignee`（单列），多人任务存 `t.assignees`。file:// 离线模式下多人任务 reward 计算错误。

### 14. submit_task: no user feedback when partial multi-slot submission succeeds
{file: `nantang-mobile/js/data.js`, line: 319-322}

多人任务部分提交时 `doSubmit` 在记录当前用户 evidence 后提前返回，任务未过渡到"待审核"——无 toast 告知用户需等其他人。
