## 🔴 阻塞（5 项 · ~2h）

### R1 — `_applyStayConfirm` 回调不存在，入住流程断裂
**文件**：[app.js:1283](nantang-mobile/js/app.js#L1283)
**问题**：`showConfirm` 回调调用 `_applyStayConfirm()`，该函数从未定义。`doCheckin` 在另一个不可达分支。同时 `showConfirm` 的第三个参数被当作回调——实际 `showConfirm(msg, onOk)` 只接受两个参数。
**改**：
1. 删除 `_applyStayConfirm()` 调用分支
2. 将 `doCheckin` 闭包逻辑移入 `showConfirm` 的 onOk 回调
3. 签约公约 quest 移到 `doCheckin` 内部，确认后才标记
**验证**：点击入住→选房间→选床位→填日期→确认公约→toast 显示入住成功→房间卡片显示入住人

### R2 — T7 迁移 `claimants` 列不存在，数据静默丢失
**文件**：[database.py:75](server/database.py#L75)
**问题**：CampTask→NTTask 迁移 INSERT 引用 `claimants` 列，NTTask 模型没有该列（只有 `assignees`）。SQLite 报错被 `except: pass` 吞掉。
**改**：
1. INSERT 语句移除 `claimants`，CampTask 的 `claimants` JSON → 解析为 name 数组 → 写入 `assignees` JSON
2. 迁移前后加日志（至少 `print`），不再静默吞错
**验证**：清空 DB→重启服务器→检查 `nt_tasks` 表有 `camp_ref_id` 列→旧 `camp_tasks` 表已删除

### R3 — `settle_camp` 查询自相矛盾 + 整数除法蒸发 NT
**文件**：[camps.py:146-164](server/routes/camps.py#L146)
**问题**：
- (a) 查询 `status=="已结算" AND settler_id==None`——`settle_task` 设置 status="已结算" 时必然同时设 settler_id，条件互斥永远返回空
- (b) `share = t.reward // max(len(assignees), 1)` 整数除法截断余数，NT 永久蒸发
**改**：
- (a) 查询改为 `status=="待结算" AND camp_ref_id==camp_id`，结算后设 `status="已结算"`
- (b) 余数分配给第一个 assignee：`share = t.reward // len(assignees); first_share = t.reward - share * (len(assignees)-1)`
**验证**：创建营地→添加任务→完成任务→settle_camp→camp_balance 减少量 = 用户 balance 增加量之和

### R4 — `verify_task` 营地任务导致 task_escrow 负数
**文件**：[camps.py:88-96](server/routes/camps.py#L88) + [nt.py:490](server/routes/nt.py#L490)
**问题**：营地任务创建时不设 escrow_amount，verify 对所有任务统一执行 `pool.task_escrow -= task.reward * len(assignees)`。营地任务 verify 后 task_escrow 变负。
**改**：`verify_task` 加 `scope=='camp'` 分支——从 `pool.camp_balance` 扣款而非 `task_escrow`。`settle_camp` 同步处理。
**验证**：营地任务 verify→task_escrow 不变→camp_balance 减少→assignee 收到 NT

### R5 — 硬编码年份 2026 + 退房角色降级误伤 admin
**文件**：[app.js:1176](nantang-mobile/js/app.js#L1176) + [accommodation.py:88-95](server/routes/accommodation.py#L88)
**问题**：
- 日期计算用 `new Date('2026-'+...)` 硬编码年份，2027 年后全部失效
- 退房角色降级不检查 admin/builder，管理员退房后降级为 visitor
**改**：
- 日期改为 `new Date().getFullYear() + '-' + ...`
- 退房降级前加 `if user.role in ("admin", "builder"): return` 跳过

## 🟡 高（9 项 · ~3h）

### R6 — 前端 accommodations ↔ 服务端 Tenancy 双存储不同步
**文件**：[app.js:1267-1273](nantang-mobile/js/app.js#L1267) + [accommodation.py](server/routes/accommodation.py)
**问题**：客户端读写 `AppData._data.map_locations.accommodations`（localStorage），服务端读写 `Tenancy` 表。checkin/checkout 时两端独立操作，不同用户的入住数据不互相同步。
**改**：
- `_applyStay` 先调 `POST /api/accommodation/checkin`，成功后再写 localStorage
- `_checkoutBed` 先调 `POST /api/accommodation/checkout`，成功后再更新 localStorage
- 新增 `_syncAccommodations()`——从服务端拉取所有 tenancy 数据覆盖本地
- `_showStaySheet` 优先用服务端数据（在线），fallback 本地（离线）
**验证**：两个用户分别 checkin 同一房间不同床位→刷新→双方都看到正确占用

### R7 — 池操作全面加锁（13 处 `_get_pool` 缺 `lock=True`）
**文件**：[nt.py](server/routes/nt.py) + [tasks.py](server/routes/tasks.py) + [accommodation.py](server/routes/accommodation.py)
**问题**：spend/topup/verify/cancel/resolve/create/delete/checkout 共 13 处调用 `_get_pool(db)` 缺少 `lock=True`，并发可能双花池资金。
**改**：所有修改 pool 字段的 `_get_pool(db)` 替换为 `_get_pool(db, lock=True)`，并在同事务中对用户行加 `with_for_update()`
**验证**：并发脚本同时创建 10 个社区任务→pool.balance 正确递减，无超发

### R8 — 客户端 `claimants`[{name}] ↔ 服务端 `assignees`["id"] 格式断层
**文件**：[core.js:864](nantang-mobile/js/core.js#L864) + [data.py:400](server/routes/data.py#L400)
**问题**：客户端存 `claimants: [{name, submission, submittedAt}]`，服务端存 `assignees: ["user1", "user2"]`。syncAll 后新建任务写死 `claimants:[]`，已有人认领的服务端任务在客户端显示为空。
**改**：`_mergeSyncData` 新建任务时将 `data.tasks[i].assignees` 数组映射为 `claimants: assignees.map(name => ({name}))`。更新路径 Object.assign 不覆盖已有 `claimants`（避免丢失 `submission` 字段）。
**验证**：用户 A 认领任务→用户 B syncAll→用户 B 看到 A 在认领列表中

### R9 — 信箱 badge 双系统竞争 + 地图 badge 不更新
**文件**：[ui-archive.js:176](nantang-mobile/js/ui-archive.js#L176) + [index.html:439](nantang-mobile/index.html#L439)
**问题**：`mobile-bundle.js` 和 `ui-archive.js` 各有一个 `updateInboxBadge`，后加载的覆盖前者。`ubInboxBadge`（地图内）不被任何函数更新。
**改**：
- 删除 `mobile-bundle.js` 中的重复 `updateInboxBadge`
- `updateInboxBadge` 同时更新 `#inboxBadge` 和 `#ubInboxBadge`
**验证**：收到消息→地图和村口两处 badge 同步显示/消失

### R10 — `doPublish` 离线 NT 创建死代码 + `_guardOnline` 误杀离线发布
**文件**：[core.js:642-665](nantang-mobile/js/core.js#L642)
**问题**：`_guardOnline` 在函数开头 return，后续离线 NT 创建代码永远不可达。离线用户完全无法发布任务。
**改**：`doPublish` 和 `publishDraft` 中，`_guardOnline` 只拦截 HTTP 调用部分，离线 NT 路径移到 guard 之前或不 guard。
**验证**：离线模式→发布任务→NT 余额扣减→本地任务显示

### R11 — `reviewTask` approve 客户端不改服务端状态
**文件**：[data.js:385](nantang-mobile/js/data.js#L385)
**问题**：审批通过时调 `API.request('POST','/api/nt/tasks/'+id+'/verify',{approved:true})` 但没 await 结果。不管服务端成功与否，客户端直接改 `status='待结算'`。
**改**：
- API 调用改为 `await` 或增加回调处理
- 服务端返回非 ok 时回滚本地状态 + showToast
**验证**：审核通过→服务端 task_escrow 扣减→assignee 收到 NT→客户端 status 同步

### R12 — 退房欠费未清时 debt 不归零
**文件**：[accommodation.py:84-87](server/routes/accommodation.py#L84)
**问题**：余额不足退房时写 `debt_unpaid` 账本条目但不设 `t.debt = 0`。欠费残留在已 checkout 的 tenancy 上，永不回收。
**改**：无论是否还清，退房后 `t.debt = 0`（未清部分记录在账本，不挂在 tenancy）
**验证**：余额 0 的用户退房→tenancy.debt=0→账本有 `debt_unpaid` 记录

### R13 — `delete_camp` 不退还 camp_balance
**文件**：[camps.py:204-223](server/routes/camps.py#L204)
**问题**：删营地时 NTTask 级联删除，创建时注入 `camp_balance` 的 NT 永久丢失。
**改**：删除前 `pool.camp_balance -= camp_total`（回退入 pool.balance），写账本。
**验证**：创建营地（注入 100 NT）→删除→pool.balance 恢复 100 NT

### R14 — syncAll 缺 `camp_ref_id` + `syncTask` 缺 `req_photo/req_file` 写入
**文件**：[data.py:396-406](server/routes/data.py#L396) + [tasks.py:105-110](server/routes/tasks.py#L105)
**问题**：syncAll 任务序列化缺 `camp_ref_id`、`location_id`。task create 不写 `req_photo/req_file`。
**改**：data.py 任务序列化补 `camp_ref_id`、`location_id`。tasks.py create 补 `req_photo/req_file` 写入。
