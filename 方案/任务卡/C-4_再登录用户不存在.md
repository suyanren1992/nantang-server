## 目标
SQLite（Render 临时磁盘，重启丢库）→ Neon 免费 Postgres（0 元、0.5GB、100 CU-hour/月）。数据从此不丢。

## ① 监察勘察（Kimi Code，只读）
1. 通读 `server/database.py`、`server/models.py`、`server/main.py`、各 `routes/*.py`，列出所有 SQLite 方言依赖点（如 `INSERT OR REPLACE`、AUTOINCREMENT、JSON 存字符串的字段、datetime 默认值写法）
2. 列出 requirements/构建命令现状（缺 `web3` 的老问题顺带记录，不在本卡修）
3. 勘察结论写 BUG_TRACKER.md：需要适配的点清单

## ② 施工（Claude Code，按勘察结论）
1. **连接层**：`database.py` 改读环境变量 `DATABASE_URL`（无此变量时回落 SQLite，本地开发零影响）。Postgres 用 `postgresql+asyncpg://`，requirements 加 `asyncpg`
2. **方言适配**：按①的清单逐条改成跨库写法（upsert 用 SQLAlchemy 通用写法或分库分支，尽量通用）
3. **建表**：启动时 `create_all` 逻辑保留（Neon 新库自动建表）
4. **数据迁移**：当前线上库是全新库、无真实用户——**不做数据迁移**，直接新库起步，在 BUG_TRACKER.md 注明
5. **Neon 侧操作**（砚仁手动，监察给步骤）：neon.com 用 GitHub 账号一键注册 → 建项目（区域选新加坡/美西均可）→ 复制连接串 → Render 环境变量加 `DATABASE_URL`

## ③ 额度监控（本卡附带，Claude Code）
1. 前端余额轮询从 30s 降为 **60s**（省 CU-hour，用户无感）
2. `server/main.py` 加 `/api/health` 若已有则不动；新增一个轻量日志：每天首次请求时打一条启动日志即可（不做复杂计量）
3. 在 BUG_TRACKER.md 写"每月 25 日人工查一次 Neon 控制台用量"的提醒条目（正式告警后续再议）

## ④ 监察校验（Kimi Code）
- 本地：不设 DATABASE_URL 启动 → SQLite 正常工作
- 线上：设好 DATABASE_URL 部署后 → 注册新用户 → Render 手动重启 → 用户还在、余额还在
- 冒烟：上报劳动→校核→到账；充值申请提交
- 结论写 BUG_TRACKER.md
