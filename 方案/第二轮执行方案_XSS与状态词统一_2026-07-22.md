## W1 — XSS 修复：`esc()` 无效 + 全局 onclick → addEventListener

### 问题
`esc()` 只转义 HTML 实体（`&<>'"`），但在 `onclick="...' + esc(name) + '..."` 中无效——浏览器先 HTML 解码再给 JS，`&#39;` 被还原为 `'`。用户名设为 `'); alert(1)//` 即可 XSS。

### 改法

**阶段 A：快速止血 — `encodeURIComponent` 替代 `esc()`（15min）**
**原理**：`esc()` 输出 HTML 实体（`'` → `&#39;`），浏览器 HTML 解码属性值后又还原为 `'`，注入成功。`encodeURIComponent` 输出 `%27`，HTML 解码不会触动 `%` 编码。
**文件**：全部 `nantang-mobile/js/*.js` + `index.html`
全局搜索 `onclick="...esc(name)..."` → 替换为 `encodeURIComponent(name)`。
接收端（函数参数）加 `decodeURIComponent` 还原。
```bash
grep -rn "esc(" nantang-mobile/js/*.js nantang-mobile/index.html | grep onclick
```
**示例**：
```javascript
// 改前（不安全）：esc() 的 &#39; 被 HTML 解码还原
onclick="openTask(\'' + esc(name) + '\')"

// 改后（安全）：%27 不会被 HTML 解码
onclick="openTask(\'' + encodeURIComponent(name) + '\')"
// 接收端：
function openTask(name) { name = decodeURIComponent(name); ... }
```
**防御原理**：`');alert(1)//` → `encodeURIComponent` → `%27);alert(1)//%27` → 浏览器不解码 `%` → JS 字符串安全。

**阶段 B：长期方案 — data 属性 + 事件委托（2h）**（阶段 A 之后执行）
全局搜索 onclick 注入点，逐一改写为 `data-action` 属性 + 统一 `click` 事件委托。

示例——改前：
```html
<div onclick="openTask(' + esc(name) + ')">任务</div>
```
改后：
```html
<div data-action="openTask" data-name="${encodeURIComponent(name)}">任务</div>
```
```javascript
document.body.addEventListener('click', function(e) {
  var btn = e.target.closest('[data-action]');
  if (!btn) return;
  var action = btn.dataset.action;
  var name = decodeURIComponent(btn.dataset.name || '');
  if (action === 'openTask') openTask(name);
  // ... 其他 action
});
```

**影响面预估**：~20 处 onclick 需改写，涉及 core.js/data.js/app.js/index.html。

### 验证
- 注册用户名为 `<img src=x onerror=alert(1)>` → 不弹窗
- 注册用户名为 `'); alert(1);//` → 不弹窗
- 所有任务卡片/用户列表/成员选择器正常点击

## W2 — 状态词统一

### 问题
三套状态词汇并存：
- 服务端：`TASK_STATUSES`（`进行中→待审核→待结算→已结算`）
- 客户端 data.js：额外加了 `待提交`、`已完成`、`草稿`
- 客户端 nt-core.js：英文 `pending→active→completed→verified→settled`

没有任何映射层。跨边界操作靠运气。

### 改法

**阶段 A：审计当前状态（10min）**
**文件**：全部
统计每种状态词的使用位置和频率：

| 状态词 | 出现位置 | 说明 |
|--------|---------|------|
| `进行中` | nt.py, tasks.py, data.js, core.js | 服务端标准 ✅ |
| `待审核` | nt.py, data.js, core.js | 服务端标准 ✅ |
| `待结算` | nt.py, data.js, core.js | 服务端标准 ✅ |
| `已结算` | nt.py, data.js, core.js | 服务端标准 ✅ |
| `已取消` | nt.py, data.js | 服务端标准 ✅ |
| `已争议` | nt.py, data.js | 服务端标准 ✅ |
| `退回修改` | nt.py, data.js | 服务端标准 ✅ |
| `待提交` | data.js, core.js | 客户端独有 ⚠️ |
| `已完成` | data.js | 客户端独有 ⚠️ |
| `草稿(draft)` | data.js, core.js | 客户端独有 ⚠️ |

**阶段 B：统一决策（5min）**
- 保留 `待提交`（多人任务中一人提交后的中间态）
- 合并 `已完成` → `待结算`（语义等价）
- `草稿` 保留为客户端本地状态（不同步服务端）
- 服务端 `TASK_STATUSES` 加 `待提交` 键

**阶段 C：修改文件（1h）**

| 文件 | 改动 |
|------|------|
| `models.py:6-14` | TASK_STATUSES 加 `"pending_submit": "待提交"` |
| `data.js` | `已完成` → `待结算`，`doSubmit` 状态设 `待提交` |
| `core.js` | 所有 `已完成` → `待结算` |
| `nt-core.js` | 顶部加状态映射表：`{active:'进行中', completed:'待结算', verified:'已结算', pending:'待提交'}` |
| `nt.py:101` | `frozen_statuses` 引用 `TASK_STATUSES` 值而非硬编码 |

### 验证
- 创建任务 → 进行中 → 提交 → 待审核 → 审核 → 待结算 → 结算 → 已结算
- 多人任务一人提交 → 待提交（不触发审核）
- 客户端 nt-core.js 操作后状态与服务端一致
