## 🟡 高优大改（4 项 · ~2.5h）

### R6 — 前端 accommodations ↔ 服务端 Tenancy 双存储统一
**文件**：[app.js](nantang-mobile/js/app.js#L1079-L1289) + [accommodation.py](server/routes/accommodation.py) + [app-data.js](nantang-mobile/js/app-data.js)

**现状**：客户端读写 `AppData._data.map_locations.accommodations`（localStorage JSON），服务端读写 `Tenancy` 表（SQLite）。checkin/checkout 两端独立操作，互不同步。不同用户的入住数据在 local storage 不可见。

**方案**：
1. 新增 `API.getAccommodations()` → 调 `GET /api/accommodation/all`（返回所有 active tenancy 列表）
2. 服务端新增 `@router.get("/all")` → 返回 `select(Tenancy).where(Tenancy.status == "active")`
3. `_showStaySheet` 在线模式下先调 `API.getAccommodations()` 同步本地 `accommodations` 再渲染
4. `_applyStay` 和 `_checkoutBed` 改为：先调服务端 → 成功后再更本地
5. 离线模式保持纯本地操作
6. 每 5 分钟或切换页面时自动刷新

**验证**：用户 A checkin → 用户 B 打开住宿面板 → 看到 A 的床位已占用

**改量**：~60 行（新增端点 15 行 + 客户端 45 行）

### R8 — 客户端 `claimants`[{name}] ↔ 服务端 `assignees`["id"] 格式桥接
**文件**：[core.js](nantang-mobile/js/core.js#L857-L866) + [data.py](server/routes/data.py#L396-L406)

**现状**：
- 服务端存 `assignees: ["user1", "user2"]`（字符串数组）
- 客户端存 `claimants: [{name, submission, submittedAt, status}]`（对象数组）
- `syncAll` 新建任务写死 `claimants:[]`，服务端已认领的任务在客户端显示空

**方案**：
1. `_mergeSyncData` 新建任务分支：
   ```javascript
   // 现有：
   if (!dup) AppData._data.tasks[t.id] = { name:t.id, ..., claimants:[], ... };
   // 改为：
   var serverAssignees = (t.assignees || []).map(function(id){ return {name:id}; });
   if (!dup) AppData._data.tasks[t.id] = { name:t.id, ..., claimants: serverAssignees, ... };
   ```
2. `data.py` syncAll 返回的 `assignees` 已经是 `_safe_assignees(t)` 解析后的 list，直接可用
3. 更新路径 `Object.assign` 不覆盖已有 `claimants`（保留客户端 `submission` 字段）：
   ```javascript
   else { var keepClaimants = dup.claimants; Object.assign(dup, {...}); dup.claimants = mergeClaimants(keepClaimants, t.assignees); }
   ```

**验证**：用户 A 认领任务 → 用户 B syncAll → B 看到 A 在认领列表中

**改量**：~25 行（`_mergeSyncData` 两个分支 + 一个 `mergeClaimants` 小函数）

### R12 — 退房欠费未清时 debt 不归零 + 相关边界
**文件**：[accommodation.py:73-87](server/routes/accommodation.py#L73) + [nt.py:888](server/routes/nt.py#L888)

**现状**：
- (a) 余额不足退房时，`debt_unpaid` 写账本但 `t.debt` 不清零，残留在已 checkout 的 tenancy 上永不回收
- (b) `NIGHTLY_RATE` 无 `>=1` 校验，设为 0 或负数可倒转资金流

**方案**：
- (a) 退房后无论如何设 `t.debt = 0`（未清部分已记录在账本，不挂在 tenancy）
- (b) `NIGHTLY_RATE = max(1, int(os.environ.get("NIGHTLY_RATE", "10")))`

**验证**：余额 0 的用户退房 → tenancy.debt=0 → 账本有 `debt_unpaid` 记录

**改量**：~5 行
