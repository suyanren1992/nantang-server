# C-4 决策奏折：数据库持久化两条路

> 2026-07-24 · 请陛下朱批其一 · 背景：SQLite 存在 Render 临时磁盘上，每次部署/重启数据库即清空——注册的用户、余额、账目全部归零。这是当前唯一的架构级炸弹。

---

## 路 A：Render 挂持久磁盘（Persistent Disk）

**做法**：Render 服务设置里挂一块磁盘（如 `/var/data`），把 SQLite 数据库文件指到挂载路径。代码改动 ≈ 1 行（数据库路径改读环境变量）。

| 维度 | 评估 |
|---|---|
| 费用 | **约 $1/月**（1GB 起步价，够用很久） |
| 改动量 | 极小：环境变量 + 路径，半天内上线 |
| 风险 | 低：SQLite 不动，现有代码零重构 |
| 短板 | ① SQLite 单文件，多人同时写入会排队（20 人社区规模完全够）；② 磁盘和这台 Render 服务绑定，以后换服务商要手动搬文件；③ 免费 Hobby 套餐**是否支持挂盘需先确认**——若不支持，需升级套餐（$7/月起） |

## 路 B：迁移 PostgreSQL

**做法**：Render 开 Postgres 实例，代码改用 Postgres 连接（项目已是 SQLAlchemy 异步，迁移主要是连接串 + 少量方言适配），一次性把 SQLite 数据导过去。

| 维度 | 评估 |
|---|---|
| 费用 | Render 免费档 Postgres **90 天后到期需付费**（约 $7/月起）；或接外部免费 Postgres（如 Neon/Supabase 免费档，长期免费但有用量上限） |
| 改动量 | 中：连接配置 + 方言适配 + 数据迁移 + 回归测试，约 1-2 天 |
| 风险 | 中：迁移期要小心，但迁完后顾之忧少 |
| 长板 | ① 真正的多人在线数据库，并发不再是问题；② 不绑死 Render，随时可搬；③ 20 人→200 人都撑得住 |

## 丞相建议

**先 A 后 B**：现在花 $1 挂盘止血（当天生效，数据不再丢），社区跑起来、人数逼近上限或要换服务商时，再开一张卡迁 Postgres。两步不冲突，A 是创可贴，B 是正规手术——现在社区还在打磨期，不值得直接上手术台。

**例外**：如果 Hobby 免费套餐不能挂盘、必须升 $7 套餐——那不如直接走路 B（接 Neon 免费 Postgres），同样的钱办更大的事。
