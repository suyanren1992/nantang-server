# 社区资金池 · 多钱包 + 结算冻结 + 收支设计

> 2026-07-22 · 目标：20 人社区自我循环 · 基于现有代码的增量设计

## 一、现状诊断

### 当前池结构（单层混合）

```
CommunityPool（单行）
├── balance              ← 混合：收入沉淀 + 支出源 + 缓冲储备
├── task_escrow          ← 任务托管金（已分离 ✅）
├── camp_balance         ← 营队资金（已分离 ✅）
├── contribution_pool    ← CV 磨损汇集（未使用）
└── total_issued         ← 发行总量
```

**问题**：`balance` 承担三件事却没有区分——spend 回流、住宿费收入、校核支出、任务支出全混在同一口井里。出问题时无法判断是收入不足还是支出失控。

### 资金流向（全部 18 条路径）

**流入 pool.balance 的 7 条路径：**

| # | 来源 | 触发 | 频率 |
|---|------|------|------|
| 1 | spend 回流 | 用户消费 NT | 低频 |
| 2 | topup community_pool | admin 手动注资 | 按需 |
| 3 | 住宿费 daily_tick | 每人每天 | 日频 |
| 4 | 池补填 daily_tick | pool<300 时 | 日频 |
| 5 | cancel 退款（发布者删除） | 任务取消 | 低频 |
| 6 | resolve 退款（用户删除） | 争议仲裁 | 极低频 |
| 7 | checkout 欠费结算 | 退房时 | 低频 |

**流出 pool.balance 的 5 条路径：**

| # | 去向 | 触发 | 频率 |
|---|------|------|------|
| 8 | 校核 doer 奖励 | approve_verification | 日频 |
| 9 | 校核 verifier 奖励 | approve_verification | 日频 |
| 10 | 社区任务创建 | poster='社区' | 低频 |
| 11 | 营地结算回流 | camps settle | 低频 |
| 12 | 离线 earn 同步 | _drainPendingEarns | 低频 |

### 会计等式

```
sum(users.nt_balance) + pool.balance + pool.task_escrow + pool.camp_balance == pool.total_issued
```

## 二、目标架构：四钱包模型

不改会计等式，在**现有 CommunityPool 表上扩展字段**（零迁移风险，向前兼容）：

```
CommunityPool（单行，新增 3 列）
├── balance              ← 运营池：日常收支（继承当前 balance 语义）
├── reserve              ← 储备池：缓冲储备，仅补填和手动注资写入 [NEW]
├── task_escrow          ← 任务托管（不变）
├── camp_balance         ← 营队资金（不变）
├── contribution_pool    ← CV 磨损（不变）
├── total_issued         ← 发行总量（不变）
└── frozen               ← 冻结资金：争议/异常锁定 [NEW]
```

### 各钱包职责

| 钱包 | 收入来源 | 支出去向 | 冻结规则 |
|------|---------|---------|---------|
| **balance**（运营池） | 住宿费、spend 回流、checkout 欠费 | 校核 doer+verifier、社区任务、营地回流 | 无 |
| **reserve**（储备池） | 每日补填、admin topup、链上充值 | **不允许内部支出**——仅转入 balance 或链上提现 | 无 |
| **task_escrow**（托管） | 任务创建时冻结 | 任务完成→释放给执行者；取消→退还发布者 | 任务期间锁定 |
| **camp_balance**（营队） | 营地创建注资 | 营地结算发放 | 营地期间锁定 |
| **frozen**（冻结） | 争议仲裁暂扣、异常交易冻结 | admin 手动解冻或裁决后释放 | 永久冻结直到 admin 操作 |

### 会计等式（不变）

```
sum(users.nt_balance) + pool.balance + pool.reserve + pool.task_escrow + pool.camp_balance + pool.frozen == pool.total_issued
```

验证端点 `/api/nt/verify` 同步更新加 `reserve` 和 `frozen` 两项。

## 三、收支分配规则

### 收入 → 分配目标

> **6 Agent 审查修正**：补填进 balance（非 reserve）——运营池是唯一日常支出口，补填应直接输血。
> reserve 的收入来源为：链上充值（ChainScanner）+ admin topup 手动注资（默认进 reserve）。
> balance 盈余超过安全线时，日终自动转入 reserve（见第六节）。

| 收入事件 | 入 balance | 入 reserve | 理由 |
|---------|-----------|-----------|------|
| 住宿费 daily_tick | ✅ 100% | — | 直接进运营池，支撑日常校核支出 |
| spend 回流 | ✅ 100% | — | 用户消费直接回运营池 |
| checkout 欠费 | ✅ 100% | — | 追缴欠款回运营池 |
| 每日补填 | ✅ 100% | — | 运营池输血（20人方案M2：pool.balance<300时+20） |
| 链上充值（ChainScanner） | — | ✅ 100% | 外部资金进储备，不直接用于支出 |
| admin topup community_pool | 可指定 | 默认 100% | admin 可选紧急注资(运营)或长期储备 |
| cancel/resolve 退款 | ✅ 100% | — | 回退到运营池 |
| balance 盈余划拨 | — | ✅ surplus | balance>1000时超出部分转入reserve |

### 支出 → 扣款来源

| 支出事件 | 扣 balance | 扣 reserve | 规则 |
|---------|-----------|-----------|------|
| 校核 doer+verifier | ✅ | — | 直接扣运营池 |
| 社区任务创建 | ✅ | — | 移入 task_escrow；若balance不足但balance+reserve充足→即时划拨后再扣 |
| earn_sync 离线同步 | ✅ | — | |
| **提现（withdraw）** | — | ✅ | 两阶段：先冻入frozen→admin多签确认后从frozen扣 |
| 运营池不足时 | — | reserve→balance 划拨 | balance<150时从reserve转入min(reserve, max(200, 缺口+100)) |

**关键设计**：补填进运营池（直接可用），充值进储备池（长期存储）。运营池盈余自动转入储备池（balance>1000时）。提现从储备池走两阶段冻结（先冻入frozen，admin确认链上转账完成后才销毁），防止资金在链上确认前消失。

### 支出 → 扣款来源

| 支出事件 | 扣 balance | 扣 reserve | 规则 |
|---------|-----------|-----------|------|
| 校核 doer+verifier | ✅ | — | 直接扣运营池 |
| 社区任务创建 | ✅ | — | 移入 task_escrow |
| earn_sync 离线同步 | ✅ | — | |
| 运营池不足时 | — | reserve→balance 划拨 | **自动划拨**：balance < 100 时从 reserve 转入 min(reserve, 200) |

**关键设计**：reserve 是 balance 的"水库"——收入先进运营池花，补填进储备池存着。运营池不够时自动从储备池调水。人工 topup 默认进储备池（防滥用），紧急时可指定进运营池。

## 四、冻结/托管机制

### task_escrow（已有 ✅）

保持不变。任务创建时 `balance -= reward*slots`, `task_escrow += reward*slots`。验证通过后释放给执行者。

### frozen（新增）

用于争议场景和异常保护：

| 冻结场景 | 触发条件 | 冻结金额 | 解冻条件 |
|---------|---------|---------|---------|
| 争议仲裁 | `/api/nt/tasks/{id}/dispute` | `task.escrow_amount` | admin `/resolve` 后释放 |
| 异常交易 | admin 手动标记 | 指定金额 | admin 手动解冻 |
| 校核异常 | 3 次驳回后 | doer 对应 nt_amount | admin 审核后决定释放或没收 |

**实现**：
- `resolve_task` 中将 `pool.task_escrow -= escrow` 改为 `pool.frozen += escrow; pool.task_escrow -= escrow`
- 新增 `POST /api/admin/unfreeze` 端点：`pool.frozen -= amount; pool.balance += amount`

## 五、NTLedger type 扩展

新增 type 值以区分不同钱包间流转：

```
现有 type: transfer, earn, spend, topup, cashout, task_freeze, task_reward, task_refund, task_cancelled, task_settled, accommodation_fee, pool_refill, debt_repayment, debt_settlement

新增 type:
  pool_transfer    — balance ↔ reserve 划拨
  pool_freeze      — balance → frozen 冻结
  pool_unfreeze    — frozen → balance 解冻
```

每条 type 在 ledger 中完整记录了 from → to 的多钱包路径，审计时可追踪每一分钱的跨钱包移动。

## 六、日终结算（Daily Settlement）

在 `daily_tick` 末尾新增结算逻辑：

```python
# 3. 运营池盈余 → 储备池划拨（balance > 1000 时超出部分转入 reserve）
if pool.balance > 1000:
    surplus = pool.balance - 500  # 留 500 运营底线
    pool.balance -= surplus
    pool.reserve += surplus
    lid = _ledger_id()
    await _add_ledger(db, lid, "balance", "reserve", surplus,
                      "pool_transfer", f"盈余划拨 {today}", status="settled")

# 4. 储备池 → 运营池划拨（balance < 150 时自动调水）
if pool.balance < 150 and pool.reserve > 0:
    transfer = min(pool.reserve, max(200, abs(pool.balance) + 100))
    pool.reserve -= transfer
    pool.balance += transfer
    lid = _ledger_id()
    await _add_ledger(db, lid, "reserve", "balance", transfer,
                      "pool_transfer", f"日终划拨 {today}", status="settled")

# 5. 会计等式 drift 告警（>5 NT 或 >total_issued*0.1% 时打 ERROR）
total_user = (await db.execute(select(func.sum(User.nt_balance)))).scalar() or 0
total_system = total_user + pool.balance + pool.reserve + pool.task_escrow + pool.camp_balance + pool.frozen
drift = total_system - pool.total_issued
threshold = max(5, int(pool.total_issued * 0.001))
if abs(drift) > threshold:
    logger.error(f"ACCOUNTING DRIFT: {drift} NT (threshold={threshold})")
    results["drift"] = drift

# 6. 链上对账（仅 cron.py tick_daily 执行，RPC 10s 超时）
if PLATFORM_WALLET:
    try:
        chain_bal = await asyncio.wait_for(
            asyncio.to_thread(_query_chain_balance), timeout=10.0
        )
        results["chain_balance"] = chain_bal
        if chain_bal < pool.total_issued - 5:
            shortage = pool.total_issued - chain_bal
            logger.error(f"CHAIN SHORTFALL: {shortage} NT")
            results["chain_shortfall"] = shortage
    except (asyncio.TimeoutError, Exception) as e:
        logger.warning(f"chain audit skipped: {e}")
```

**注意**：链上对账使用 `asyncio.to_thread` 避免阻塞事件循环。仅 cron.py 每天执行一次，`daily_tick` HTTP 端点不触发 RPC 调用。

## 七、迁移计划

### Phase 1（即期 · 45min）**：加列 + 改 daily_tick + auto-transfer**

- `models.py` CommunityPool 加 `reserve`, `frozen` 两列（默认 0）
- `database.py` init_db 加 ALTER TABLE 迁移 + 全新池行加 `reserve=0, frozen=0`
- `nt_helpers.py` _get_pool 加 reserve/frozen 的 NULL 守卫（`if pool.reserve is None: pool.reserve = 0`）
- `daily_tick` 末尾加：balance>1000 → reserve 盈余划拨 + balance<150 → reserve→balance 自动调水（**Phase 1 必须包含 auto-transfer，否则补填进 balance 后 reserve 无法回馈运营池**）
- 补填保持进 balance（与20人方案M2一致，不做方向改变）
- `/api/nt/pools` 返回加 reserve, frozen 字段
- `/api/nt/verify` 等式加 reserve, frozen
- **回滚**：`ALTER TABLE DROP COLUMN reserve; DROP COLUMN frozen;` 恢复 daily_tick 旧逻辑

**Phase 2（短期 · 1h）**：自动划拨 + frozen 接入
- daily_tick 末尾加 balance<100 时 reserve→balance 划拨
- resolve_task 改为冻结而非直接释放
- 新增 `POST /api/admin/unfreeze`
- ledger type 加 pool_transfer, pool_freeze, pool_unfreeze

**Phase 3（中期 · 30min）**：客户端面板
- `/api/nt/pools` 返回 5 钱包完整数据
- 管理面板显示各钱包余额
- 池预警阈值从 balance 改为 balance+reserve（总可用资金）

## 八、收支平衡验证（20 人模型）

**日流量**：

| 事件 | 钱包 | 金额 |
|------|------|------|
| 住宿费 12人×28NT | balance | +336 |
| spend 回流 | balance | +10 |
| 补填 | **reserve** | +20 |
| 校核 doer 15人×1.5×9NT | balance | -203 |
| 校核 verifier 15人×1.5×2.25NT | balance | -51 |
| 周期任务 | balance | -14 |
| **balance 日净** | | **+78** |
| **reserve 日净** | | **+20** |

运营池日均 +78 盈余。月积累 ~2,300 NT 进运营池。储备池月积累 600 NT。

**低谷场景（10 人住）**：balance 日净 +48。仍盈余。  
**高峰场景（16 人住）**：balance +190 日盈余。月积累 5,700。超额时可手动将 balance 盈余转入 reserve。

**设计意图**：运营池轻度盈余 + 储备池稳定增长 = 两池合计月 +2,900 NT。这个盈余覆盖营地创建（50-200 NT/次）、月度活动、意外支出，不会通胀（total_issued 不变，只是池内部分配）。
