## 发现清单（10 项）

### ISSUE 1a [BUG] — file:// 模式下 fetchTasks 是死代码

`core.js:826-829`

HTTP 路径 `_finishEnter` 调 `_startPolling()` + fetch balance。file:// 路径有 `API.fetchTasks()` 但这需要 API.token（永远没有）。

### ISSUE 1b [CONSISTENCY] — enterVillage 双模分支脆弱

`core.js:786-830`

HTTP 和 file:// 是复制粘贴的两套初始化逻辑，未来改一个容易忘记另一个。

### ISSUE 1c [BUG] — deleteTask 用原始 XHR 绕过 API.request

`app-data.js:138`

直接 `new XMLHttpRequest()` 不发 API.request() 的离线检查和 token。

### ISSUE 1d [BUG] — syncTask 返回值是 Promise，调用方当同步判

`api.js:106-115`、`app-data.js:123`

`var srvId = API.syncTask(task)` → srvId 永远是 Promise 对象，不是 task ID。

### ISSUE 1e [CONSISTENCY] — syncActivity/syncDiscovery fire-and-forget

`api.js:133-142`

HTTP 正常，file:// 模式静默失败。

### ISSUE 2a [CRITICAL] — 双存储无冲突解决

在 HTTP 模式下 localStorage 和服务端 DB 同时写。只增不修，冲突不解决。

### ISSUE 2b [CRITICAL] — getUsers() HTTP 模式下始终空

`auth.js:10` 读 `nt_users` localStorage key。HTTP 注册走服务端，此 key 不填充。→ 所有权限检查、角色 UI、admin 按钮失效。

### ISSUE 2c [DUPLICATION] — pendingTransactions vs DepositIntent

`app-data.js:20` + `core.js:1116` 的 pendingTransactions 与 `models.py:259` 的 DepositIntent 目的相同但未同步。

### ISSUE 2e [CRITICAL] — approveTx 只改客户端 NT

`core.js:1164`

管理员审批调用 NT.topUp()（客户端内存），从不调 API.topUp()（服务端）。充值换设备即丢失。

### ISSUE 3a [HIGH] — _processedTxIds 无限增长

`nt-core.js:464` ponytail 注释标注上限但未实现 TTL。

### ISSUE 4c [BUG] — HTTP 模式角色 UI 隐藏全挂

ISSUE 2b 的级联效应。getUsers 空 → role 取不到 → admin 按钮隐藏、营队 chip 隐藏。

### ISSUE 5b [TYPE MISMATCH] — 钱包地址三地存储

nt_users[name].wallet vs AppData.me().wallet vs 服务端 wallet_address。充值流程读第一个（HTTP 模式为空）。

### ISSUE 5c [SCHEMA DRIFT] — invite_code 客户端发但服务端忽略

`api.js:41` POST 含 invite_code，`auth.py:16` RegisterRequest 不接受此字段。
