- 功能还没做的写设计 spec

**已修正**：Phase D 全部重写，每项带具体方案。6 项移入 Phase B+。

## 反馈 2：校核制双路径架构

**问题**：校核制和卡片式任务是两套独立系统，用户无法选择贡献路径。

**用户设计**：
- 做一件事时，用户可选：
  - **卡片式**（委派）：发布任务 → escrow NT → 有人认领 → 提交 → 发布者审核
  - **校核制**（自主贡献）：做完 → 公开求验证 → peer 确认 → 社区池发 NT
- 大扫除天然走校核制（公开可见）
- 已写入 D2：双路径贡献架构

## 反馈 3：入住 → 身份转变 → 新手引导

**核心问题**：
1. 选不了床位、登记不了住宿
2. 只有入住的人才是「在地村民」
3. 新手引导应在入住后触发，不是登录后

**用户设计**：
- 入住时签署公约（不是入住后）
- 签署公约 → role → resident（在地村民）→ 新手引导开始
- 未入住 = visitor = 只能看地图 + 发个人委托 + 查看工作台（看自己的任务）
- 退房 → role → visitor

**已写入**：Phase E · 身份与入住系统

## 反馈 4：新手引导 + 社区发配任务

**用户设计**：
- 签署公约后，社区自动分配劳动任务给新人
- 任务出现在**校核室**（不是任务大厅）——不用猜，公开确认
- 校核人 = 管理员（社区主理人）
- 管理员后台显示待校核的新人任务
- 不止 3 个任务（实际设计了 5 个：打扫/整理库存/浇水/翻堆肥/擦台面）

**已写入**：E3.4、E3.5、E3.8

## 反馈 6：卡片室（猜）+ 校核室（确认）双栏设计

**核心纠正**（多次对话迭代）：

### 第一版理解（错误）
卡片式 = 任务认领。校核式 = 公开验证。

### 用户纠正后的正确理解
- **卡片室** = 「猜」——有人做了事，系统生成**匿名卡片**，大家猜是谁做的。猜对有奖。
- **校核室** = 「确认」——公开求验证，doer 名字可见。确认者得验证奖励。
- 两者是**社区内部**空间，**不替代任务大厅**
- 任务大厅是公开招募，不改

### 任务大厅 ≠ 卡片室/校核室
- **任务大厅**：公开招募，任何人可发/接。不改。
- **卡片室 + 校核室**：社区内部，并排两栏。新增。
- **工作台**：个人中心。访客可查看（看自己发的任务）。改。

### 防作弊
- 校核人 + 猜对者 → 都和 doer 绑定记录
- `{ doer, verifier, guesser, action, timestamp }` 存入 ledger
- 同对频繁互确认/互猜 → 异常标记

**已写入**：E2 流程图 + E3.11

## 反馈 7：周期性任务自动生成

**用户设计**：
- 大扫除、日常清理、日常维护 → 自动变成任务卡片
- 出现在卡片室/校核室（社区内部），不是任务大厅
- 周期性：每日（日常清扫）、每周（大扫除）、每月（库存盘点）

**已写入**：E3.9

## 反馈 8：赏金池

**用户设计**：
- 玩闹性质小任务（打苍蝇、喂猫、浇水）
- 金额小（1-5 NT），频率高，周期性/不定时
- 资金从 COMMUNITY_POOL 支出（赏金池 = 社区池子用途，不新建独立池）
- 可指定校核人，或去卡片室发，或找人公开校核

**状态**：MVP 不做。等周期性任务（E3.9）跑通后扩展。

**已写入**：E3.10
