## 调查 4：Bug 排查

### Bug 4.1：放入物品保存不了
**根因**：`_submitKitchenEntry()`（app.js:1987）只调 `addVerification('stock_in')`，不写 `AppData._data.inventory`。
**影响**：物品在校核队列里但不在库存列表里。

### Bug 4.2：打扫卫生保存不了
**根因**：`_submitCleanEntry()`（app.js:2050）只调 `addVerification('cleaning')`，不更新 `dirtiness`、不写 `cleaning.log`、不调 `_save()`。
**影响**：刷完房间仍然显示脏。

### Bug 4.3：信箱首次点击延迟
**根因**：`_computeInboxMessages()`（index.html:552-640）同步遍历全部历史数据。首次点击叠加 DOM 创建。
**影响**：移动端阻塞数百毫秒。不是功能性 bug，是性能问题。

### Bug 4.4：登录点不进去
**状态**：未复现。已写排查清单（DevTools Console / Network / localStorage / 服务端状态）。

## 调查 5：入住/住宿系统

**结论**：三套并行的住宿数据模型，互不可见，住宿费永不会扣到任何人。

### 三套系统

| 系统 | 位置 | 用途 | 问题 |
|------|------|------|------|
| A: INN_ROOMS | `index.html:987` + `ui-phase4.js:2` | 硬编码 4 栋楼×6 间房 | 两份重复定义，与 B/C 不同步 |
| B: accommodations | `app-data.js:225` + `app.js:1070` | `_applyStay()` 写 `tenants[]` | 唯一可用的选床 UI |
| C: MGMT_DATA.stay | `app.js:825` | 单房间 legacy | 读硬编码，不是 B 的数据 |

### 关键 Bug
- `_deductAccommodation`（app-data.js:415）读 `room.tenant`（字符串），B 系统写的是 `room.tenants[]`（数组）→ **住宿费永远不会扣到任何人**
- 入住审批路径（ui-phase4.js:434）读 A 的 INN_ROOMS → 看不到 B 的真实入住状态
- 入住不改变用户角色（没有 resident role）

### 缺失
- 入住 → 角色变更：无代码路径
- 退房 → 角色降级：无代码路径
- 入住时支付住宿费：未实现
- 服务端入住 API：不存在（只有 `/map_locations` JSON blob 端点）
