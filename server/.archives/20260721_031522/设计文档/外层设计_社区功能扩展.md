## 整体愿景

南塘云村的社区互动围绕三个空间展开。社区功能扩展的核心变化是：**任务大厅从纯「手动发任务」扩展为三种内容来源**，同时卡片室和校核室作为社交信任层。

```
┌──────────────────────────────────────────────────┐
│                   任务大厅                         │
│   ┌─────────────┐ ┌─────────────┐ ┌────────────┐ │
│   │ 手动委托     │ │ 周期性任务   │ │ 赏金任务    │ │
│   │ 任何人可发   │ │ 系统自动生成 │ │ 随机发放    │ │
│   │ 所有人可见   │ │ 仅在地成员   │ │ 仅在地成员  │ │
│   └─────────────┘ └─────────────┘ └────────────┘ │
│          认领 → 做 → 提交 → 审核 → 得 NT          │
└──────────────────────────────────────────────────┘

┌─────────────────────┐  ┌─────────────────────┐
│   卡片室（猜）        │  │  校核室（确认）       │
│   匿名卡片 · 猜谁做   │  │  公开验证 · 名字可见  │
│   猜对有奖           │  │  确认者得 NT         │
│   数据源：已完成行为   │  │  数据源：自主贡献报备  │
└─────────────────────┘  └─────────────────────┘
        ↑   社区内部空间（社交+信任层）   ↑
```

**三层对比**：

| 空间 | 可见性 | 内容来源 | doer 身份 | 参与方式 |
|------|--------|---------|-----------|---------|
| 任务大厅 | 手动委托所有人 / 周期+赏金仅在地成员 | 手动 + cron自动 + 随机 | 公开 | 认领→做→提交→审核 |
| 卡片室 | 社区内部 | 已完成行为匿名化 | **隐藏**（匿名） | 猜是谁→猜对得 NT |
| 校核室 | 社区内部 | 自主贡献报备 | **公开** | 确认→得验证奖励 |

**在地成员定义**：`User.role IN ('resident', 'builder', 'admin', 'manager')` **且** `User.location IS NOT NULL`（有住宿位置记录）。注意：`resident` role 由 Phase E 入住系统创建，当前代码中尚不存在。过渡期用 `role IN ('builder', 'admin')` + location 非空替代。

**在地成员的定义依赖 Phase E（入住系统）**。在 Phase E 就位前，在地成员 = 当前有住宿记录的用户（`accommodations[].tenants[]` 包含该用户名），过渡期实现在服务端读 accommodations 数据。

### 任务大厅内部信息架构

任务大厅按 **scope** 分 tab（个人/营队/社区），「社区」tab 内按**来源**加二级筛选 chips：

```
任务大厅
 ├─ tab: 个人委托
 ├─ tab: 营队任务
 └─ tab: 社区  ← 默认显示三个来源
      ├─ [全部] [手动委托] [周期性] [赏金]  ← filter chips
      └─ 任务卡片列表
```

- `poster='社区'` + `is_system_generated=true` → 周期性/赏金任务，仅在地成员可看到此 chip
- `poster='社区'` + `is_system_generated=false` → 管理员手动发的社区任务
- 非在地成员：社区 tab 不显示「周期性」「赏金」chips

## 项目 A：统一贡献系统（卡片室+校核室+双路径）

> 来源：用户反馈 2、6 · Phase D D2 + Phase E E3.11
> 成熟度：🟡 概念阶段 · 原项目 1 和项目 5 合并（审查发现：它们是同一系统的服务端和客户端两侧）
> 改动文件：`ui-cardroom.js`（`renderCardRoom()` 等）、`app-data.js`（`verifyAction()`）、`nt.py`（新增 earnFromPool）、`data.py`（verification sync）

### 需求

用户做一件事时，可以选择贡献路径：

- **卡片式（委派）**：发布任务 → escrow NT → 有人认领 → 提交 → 发布者审核 → 完成后行为进卡片室（可被猜）
- **校核制（自主贡献）**：做完 → 公开求验证 → peer 确认 → 社区池发 NT → 完成后行为进卡片室（可被猜）

两条路径对应不同的场景：
- 委派适合「需要特定人做事」（如请人修水龙头），NT 由发布者承担
- 校核适合「已经做完了只需确认」（如打扫完求验证），NT 从 COMMUNITY_POOL 出

**路径一旦选择不可逆**：选委派则 NT 已 escrow，不可切换到校核制。无人认领的委托 72 小时后自动取消，escrow NT 退还。发布者可手动取消未认领的委托。

### 当前状态

- 校核制：`addVerification`/`verifyAction` 在 `app-data.js:299-386`，8 种校核类型，含 4 条防作弊规则（非自校核、1h 冷却、日上限 10 次、3 次驳回永久拒绝）。纯客户端运行，服务端 pool 不同步。
- 卡片式：`createTask`/`verifyTask` 在 `nt-core.js` + `nt.py`，已有完整 pipeline
- `ui-cardroom.js` 已实现：`renderCardRoom()`（主渲染函数）、`openDiscoveryDetail()`（猜测详情）、`confirmDiscovery()`/`denyDiscovery()`（确认/拒绝）、`_submitSelfReport()`（自报劳动——**当前同时写入 cardDiscoveries 和 pendingVerifications 两个数据源**）
- 关键代码位置：`CardDiscovery` 表（`models.py:161-175`，已有 `expired_at`/`guesser`/`guessed_person`/`doer_confirmed_at`/`doer_denied_at`）

### 设计决策

#### 已决定

2. **复用 `ui-cardroom.js`**：不新建系统。在现有 `renderCardRoom()` 中加标签页切换。
3. **标签页 > 双栏**：移动端双栏不实用。标签页切换更适合小屏幕。
4. **防作弊**：复用现有 4 条规则（见下方）——**不新增 ledger 字段**。
5. **现有代码已同时写入两个数据源**：`_submitSelfReport()`（`ui-cardroom.js:842-888`）当前同时调用 `addVerification()`（→ `pendingVerifications[]`）和手动构造 `cardDiscovery` 对象（→ `cardDiscoveries[]`）。这恰好符合「同一行为同时喂给卡片室和校核室」的目标。**保留此行为，不需要重新设计**。

#### 校核室防作弊（复用现有规则，不改 ledger）

## 项目 B：社区管理员角色

> 来源：用户反馈 5 · Phase E · E3.8
> 成熟度：🟡 概念阶段 → 待确认权限粒度

### 需求

管理员可以：
1. 发布社区任务（`poster='社区'`，NT 从社区池出）
2. 审核新人任务（管理员后台显示待校核的新人任务）

### 当前状态

- 单一 `admin` 角色，`builder` 可看成员列表
- `adminNames = ['砚仁']` 硬编码在 `mobile-bundle.js:30`
- Phase D·D12 计划将 adminNames 迁移到服务端 role

### 设计决策

#### 已决定

1. **复用 `builder` 角色**：`builder.isMember=true, canClaimCamp=true`。社区管理员 = builder + 社区权限。避免新建角色类型。
2. **服务端存储**：D12 将 adminNames 迁移到服务端 `users` 表的 `role` 字段后，社区管理员也从服务端读取。
3. **权限用层级模型，不用独立 Boolean flags**（审查发现：当前社区规模个位数管理员，独立 flag 过度工程）。
   - `admin` → 全部社区权限
   - `builder` → 可发布社区任务 + 审核新人任务
   - 其他人 → 无社区管理权限
   - 未来社区超 20 人且需要差异化权限时，再细化为独立 Boolean flags。
4. **砍掉 `canManageTemplates` 权限**（审查发现：模板放配置文件，不需要管理 UI）。Phase 1 模板在代码配置中，admin 修改 = 改配置 + 部署。Phase 2（社区超 30 人、模板频繁调整）才迁入 DB + 管理 UI。

#### 待用户确认

- **「新人任务」的判定规则**（不在本文档范围，但项目 B 引用了此概念）：建议定义为「任务创建者在 7 天内入住的用户」或「任务带 `source='newbie'` 标记」。在新手引导系统设计文档中定义，本文档留接口占位。

### 服务端改动

不需要新增 `community_permissions` JSON 字段。改为在权限检查处直接判断 role：

```python

# 审核新人任务：role in (admin, builder)
# 模板管理（Phase 2 才需要）：role == admin
```

服务端鉴权检查点（审查要求补充）：

| 操作 | 端点 | 鉴权 |
|------|------|------|
| 发社区任务 | `POST /api/tasks` with `poster='社区'` | `user.role IN (admin, builder)` |
| 审核新人任务 | `POST /api/tasks/{id}/verify` (newbie) | `user.role IN (admin, builder)` |

## 项目 C：自动任务生成系统（周期性任务 + 赏金任务）

> 来源：用户反馈 7、8 · Phase E · E3.9、E3.10
> 成熟度：周期性 = 🟡 方案已定 / 赏金 = 🔴 远期
> 审查决定：合并原项目 3 和项目 4（两者共享模板引擎，差异仅在触发策略和金额）

### 需求

- **周期性任务**：大扫除、日常清理、日常维护自动变成**任务大厅里的任务卡片**，仅在地成员可见/可接，多人可一起做（多槽位）
- **赏金任务**（远期）：玩闹性质小任务（打苍蝇、喂猫、浇水），随机出现在任务大厅，仅在地成员可见/可接

### 当前状态

- `_checkDailyContainers()` — 生成每日容器清理 verification（客户端定时器，`app.js:926`，每 60 分钟）
- `_dailyTick()` — 脏污度增长、池补填、住宿费（客户端定时器）
- `_tickDirtiness()` — `app-data.js:512-524`，60 分钟间隔
- `_dailyPoolRefill()` — `app-data.js:526-535`，pool < 500 时补充 50 NT
- `_deductAccommodation()` — `app-data.js:372-403`
- **无周期性任务卡片生成器**，无常驻服务端 cron

### 🚨 技术决策：服务端 cron vs 客户端触发

审计发现 #18 已明确：**客户端触发不可行**。

| 方案 | 可靠性 | 重复风险 | 成本 |
|------|:---:|:---:|:---:|
| 客户端 `setInterval`（当前） | 无人登录=不生成 | 多人登录=重复 | 零 |
| 服务端 cron + 幂等键 | ✓ | 零（幂等键去重） | 需 cron runner |
| 服务端 + 访问时检查 | ✓ | 低（时间窗口内并发） | 低（无独立进程） |

**选型：服务端 cron**（APScheduler 或 asyncio loop），每条任务生成时带幂等键 `{template_id}:{date}`，`INSERT OR IGNORE`（SQLite）或 `ON CONFLICT DO NOTHING`（Postgres）。

### 客户端→服务端迁移机制

**问题**：Phase C 期间客户端定时器和新的服务端 cron 可能同时运行，产生重复任务。

**方案**：sync 端点响应中新增 `cron_active: true` 标志位。客户端读到 `true` 时跳过以下任务生成逻辑（保留 UI 更新和池显示）：
- `_checkDailyContainers()` → 不生成 verification
- `_dailyPoolRefill()` → 不执行池补充（服务端 cron 负责）
- `_tickDirtiness()` → 仅在客户端缓存显示，状态变更以服务端数据为准

迁移步骤：
1. Phase C 完成 → 服务端 cron 上线，`cron_active: false`
2. 验证 cron 稳定 → 设 `cron_active: true`
3. 客户端检测到 `cron_active: true` → `_dailyTick`/`_checkDailyContainers` 降级为仅 UI 更新
4. 稳定一周后 → 移除客户端定时器中的任务生成逻辑（保留 UI 层）

### 设计决策

#### 已决定

1. **出现位置**：**任务大厅的「社区」tab**。周期性任务 + 赏金任务以任务卡片形式自动发布。
   - 用户纠正：「放入大厅里更好，赏金任务随机发放，周期性任务可以大家一起做」
2. **可见性**：仅**在地成员**可见/可接。云村民和访客看不到这些任务卡片。
   - 服务端 `GET /api/tasks?scope=社区` 时过滤 `poster='社区'` + `is_system_generated=true` 的任务：非在地成员不返回。
   - **鉴权与可见性分开**：claim/submit/verify 操作也需服务端校验调用者是在地成员。
3. **任务形式**：任务卡片（`NT.createTask()`），走正常任务大厅流转：认领→做→提交→审核。
4. **多槽位**：`slots > 1`（`NTTask.slots` 已存在，`models.py:55`）。大扫除 3 槽，日常清扫 1 槽。
5. **资金**：从 COMMUNITY_POOL escrow。`poster='社区'` → 服务端扣 `COMMUNITY_POOL` 而非用户 NT。
6. **模板系统**：周期性任务和赏金任务共享模板引擎。差异仅在触发策略（cron 定时 vs 随机概率）和金额（3-10 vs 1-5 NT）。
7. **模板存储**（审查修正）：Phase 1 模板在配置文件（JSON/YAML），砍掉 `canManageTemplates` 权限。Phase 2（社区超 30 人）迁入 DB。
8. **不存储 `visible_to` 字段**：`visible_to` 由 `poster == '社区'` + `is_system_generated == true` 推导，不需要独立字段。
9. **不新建 `cron_state` 表**：幂等键 `{template_id}:{period_start_date}` + tasks 表 UNIQUE 约束即可防重复。`last_run` 从 tasks 表推导：`SELECT MAX(created_at) FROM tasks WHERE is_system_generated=true AND template_id=X`。

### 服务端 cron 执行流程

```
每日/每周 tick:
  for each template matching current period:
    1. 计算 period_start_date（daily=今天日期，weekly=本周一）
    2. 构造幂等键: "{template_id}:{period_start_date}"
    3. 检查 COMMUNITY_POOL.balance >= reward * slots
       若不足 → 跳过此模板 + 写 WARN 日志
    4. INSERT OR IGNORE → nt_tasks 表
       (poster='社区', is_system_generated=true, idempotency_key=<key>)
    5. COMMUNITY_POOL -= reward * slots
       TASK_ESCROW += reward * slots
```

**cron 分频**（审查建议）：
- 高频 tick（每 10 分钟）：仅检查赏金任务（概率触发，远期）
- 每日 tick（00:05）：检查所有 daily 模板
- 每周 tick（周日 00:10）：检查所有 weekly 模板

**并发安全**：使用 `INSERT OR IGNORE` + tasks 表 `idempotency_key` UNIQUE INDEX。两个 cron worker 同时触发时，第二个的 INSERT 被静默忽略（constraint violation 降级为 DEBUG 日志）。

**错过周期**：不补发。服务器宕机错过周期 = 该周期任务被跳过。降低 cron 运维压力。

### 退房处理

## COMMUNITY_POOL 可持续性方案

> 审查发现：新增周期任务流出 ~302 NT/月 + 赏金 ~90 NT/月，叠加现有月度赤字，池寿命以天计。需可持续方案。

### 当前收支（参考经济审查 v2）

| | 月估算 |
|---|:---:|
| 池日流入（交易磨损 75 + 标签费 180 + 捐献 100） | ~355 NT |
| 现有义务（新手任务、验证奖励、现有任务） | ~600-2100 NT |
| 新增·周期任务（4 个模板） | ~302 NT |
| 新增·赏金（远期） | ~90 NT |

**当前社区池初始余额**：**0 NT**（客户端 `nt-core.js:34` 初始 `COMMUNITY_POOL=0`，服务端 `models.py:77` `CommunityPool.balance` 默认 0）。历史上曾有 2000 NT 注资，但当前代码已统一为 0。**服务端和客户端的 COMMUNITY_POOL 完全独立，互不同步**（客户端 `_dailyPoolRefill` 在 pool<500 时补 50 NT，仅改客户端变量；服务端 `daily_tick` (`nt.py:697`) 同样逻辑但仅改服务端 DB）。两端在运行数天后必然分歧。

### 缓解措施（Phase 1）

1. **🔴 首笔注资（P0 · 必须在 cron 上线前完成）**：初始余额为 0 → cron 不会生成任何周期任务。需要**社区创建者在服务端手动执行一笔 pool 注资**（如 500 NT）。服务端已有 `POST /api/nt/topup` 端点可直接使用。
2. **客户端/服务端池统一**：Phase C 后客户端 `COMMUNITY_POOL` 变量改为从服务端 `GET /api/nt/pools` 同步读取，不再独立维护。客户端 `_dailyPoolRefill` 降级为仅显示（不再写 localStorage）。
3. **`poster='社区'` 的任务池不足时跳过**（cron 流程第 3 步）：cron 生成前检查余额，不足 → 跳过 + 写 WARN 日志 + sync 响应中设 `community_pool_status: "low"`。
4. **模板 reward 可调**：模板在配置文件，运行 2 周后根据实际领取率和池余额调整金额。
5. **服务端 `daily_tick` 统一管理池补充**：服务端 cron 负责 `_dailyPoolRefill`（当 COMMUNITY_POOL < 阈值时自动补充 50 NT），替代客户端独立印钞。
6. **池不足用户通知**：客户端在任务大厅「社区」tab 顶部显示横幅：「⚠️ 社区资金池不足，部分功能暂停。请联系社区管理员。」当 `community_pool_status === "low"` 时展示。不推送，仅 UI 层。

## 玩法打磨（Agent 审查建议 · 非用户需求）

> 以下来自第二轮 Agent 审查（游戏设计师+安全工程师+QA）的建议，非用户反馈直接提出。
> 不阻塞执行，但影响上线后的玩家体验。在 spec 阶段细化。

### 冷启动保护

**问题**：卡片室 0 张卡片 → 玩家第一印象「这个功能坏了」→ 不再打开。

**方案**：
1. 卡片数 < 4 时，显示一张特殊占位卡（扑克牌背面 + 问号）+ 文案：「第一张好事卡即将出现…在社区做一件好事，它会匿名出现在这里」
2. 前 3 个报备不需要多人确认，1 人确认即可进卡片室（打破「没人确认→没卡片→没人来」的死亡螺旋）
3. 系统在社区首次启动时自动「种植」一张示范卡：「社区 NPC 已完成：欢迎新成员入住」——纯教学用途，无 NT

### 确认超时

**问题**：报备提交后无人确认怎么办？委派有 72h 自动取消，报备没有对应超时。

**方案**：报备提交后 72h 无确认 → 自动过期。退回给报备者通知「暂时无人确认，要不要分享到群里邀请确认？」。不扣 NT（NT 还没发），不计入贡献统计。

### 多槽位独立结算（或门）

**问题**：大扫除 3 槽，第 1 人做完提交后第 2/3 人能否继续？

**方案**：每个槽位独立结算。每个认领者独立提交→独立审核→独立获得 `reward / slots` 向上取整的 NT。已完成者不需要等其他人。72h 后未认领的槽位自动关闭。

```
大扫除 reward=12 NT, slots=3 → 每槽 4 NT（12/3 向上取整=4）
总发放 ≤ 12 NT。若只认领 2/3，池只扣 8 NT。
```

### 管理员 self-claim 禁止

**问题**：管理员发社区任务后自己认领→自己审核→从池套取 NT。

**方案**：`poster='社区'` 的任务，**admin/builder 不能认领**。`POST /api/tasks/{id}/claim` 检查：`if task.poster=='社区' and user.role in ('admin','builder') → 403`。社区任务 reward 硬上限 20 NT（防止超额任务）。

### admin 审核 SLA

**问题**：新人提交任务后唯一管理员离线半天→任务卡住→新人退出。

**方案**：新人前 3 个任务提交后 4h 无审核 → 自动通过（渐进信任）。第 4 个任务起正常审核。提交时显示预估审核时间：「管理员会在 4 小时内审核，超时自动通过。」
