# 创意总监审查 · 架构与双模式

> 审查范围：双模式一致性、数据主权、技术债、角色权限
> 文件：全量 server + client 核心文件

## 发现 19 项（含 1 项架构债）

### 1. 🔴 状态词汇：三套不兼容词汇（server 中文 / nt-core 英文 / AppData 另一套中文）
**影响文件：** server/routes/nt.py, server/routes/tasks.py, nt-core.js, core.js
Server 用 `"进行中"/"待审核"/"待结算"/"已取消"/"已争议"`。nt-core 用 `"pending"/"active"/"completed"/"verified"/"settled"/"cancelled"/"disputed"`。AppData 用 `"进行中"/"待提交"/"待审核"/"退回修改"/"已完成"/"待结算"/"已结算"`。三个独立状态机，转换不同步。
**修复：** 统一为中文。nt-core.js 改用 server 状态字符串。

### 2. 🔴 字段名不匹配：poster vs publisher, taskId vs name, nt vs reward, type vs category
**影响文件：** models.py, nt-core.js, core.js:208-210,553,697-701,889, app-data.js:96-99, api.js:101-109
每层用不同字段名。8+ 处手动翻译 → 一处遗漏 = 静默数据损坏。
**修复：** 建单一 `_normalizeTask(task, source)` 函数。长期统一字段名。

### 3. 🔴 客户端 verifyTask 后余额被 server sync 覆盖 → 离线操作静默丢弃
**文件：** nt-core.js:197-242, core.js:738
`NT.verifyTask` 更新本地余额 → 进入村庄时 server balance 覆盖 → 离线 earn/spend/verify 全部丢失。
**修复：** 余额同步改为 merge/max 或排队未同步操作并重放。

### 4. 🔴 服务端 earn()/spend() 无 admin 要求 → 任何用户可铸造/提取 NT
**文件：** server/routes/nt.py:143,174
仅要求 `get_current_user`（非 `require_admin`）。任何认证用户可调 earn 抽干社区池。
**修复：** 加 `require_admin` 或删除端点（客户端不用它们）。

### 5. 🔴 客户端 transfer 允许 admin 代转，服务端不允许
**文件：** nt-core.js:491-498, server/nt.py:113
客户端检查 admin → 可代转。服务端始终用认证用户作 sender → 拒绝 admin 代转。
**修复：** 服务端 transfer 接受可选 `from_user` 参数。

### 6. 🟡 客户端 earnFromPool 更新 totalEarned/experienceValue；服务端无 totalEarned 列
**文件：** nt-core.js:446-448, server/nt.py:162-164
客户端跟踪 `totalEarned`/`totalPaid`。服务端无对应列。同步时这些字段不保持。
**修复：** 加列或从 ledger 在渲染时推导。

### 7. 🟡 服务端 spend 始终退社区池；客户端 spendToPool 支持 camp pool
**文件：** server/nt.py:187-188, nt-core.js:464-468
Camp-scoped spend → 服务端始终回社区池。客户端回 camp pool。池余额偏差。
**修复：** 服务端 spend scope camp 时路由到 `pool.camp_balance`。

### 8. 🟡 服务端 earn pool 路由脆弱 — 对无 pool 字段模型查 hasattr
**文件：** server/nt.py:153
`hasattr(req, 'pool')` 始终 False → 死代码。暗示未完成迁移。
**修复：** 删掉或显式实现 camp pool 支持。

### 9. 🟡 服务端与客户端会计等式验证不一致
**文件：** nt-core.js:546-588, server/nt.py:236-263
客户端等式不含 `PUBLIC_CV_POOL` → transfer CV 磨损产生永久漂移。服务端含 `contribution_pool`。
**修复：** 两端对齐。要么都含，要么都不含。

### 10. 🟡 deposit()/withdraw() 是含完整实现的死代码
**文件：** nt-core.js:102-125
早期 return 后含完整函数体。仍在全局作用域中定义但永不执行。
**修复：** 删掉函数体。

### 11. 🟡 _batchSettle_dead 是 70 行永不执行的代码
**文件：** nt-core.js:310-380
含净额结算算法。应归档至 docs/ 并从源码中删除。

### 12. 🟡 旧 nt.js 与 nt-core.js 重复计算
**文件：** nt.js:45-58,148-161
`calcNtTotal`/`calcFrozenBalance` 与 nt-core.js 重复。可能返回不同值。
**修复：** 确认无调用方后废弃 nt.js。

### 13. 🟡 transfer CV 规则永久销毁 25% CV → 随时间趋向零
**文件：** server/nt.py:132-133,162-164, nt-core.js:503-505
每次 transfer 净销毁 25% CV（进 PUBLIC_CV_POOL，无分发机制）。CV 随时间趋向零。
**修复：** 如有意则文档化。否则将 25% 分发回用户。

### 14. 🟡 require_admin 使用不一致 — 混用内联检查和依赖注入
**文件：** 多处
有时用内联 `user.role == "admin"`，有时用 `require_admin` 依赖。混乱。
**修复：** 统一为一种模式。

### 15. 🟢 初始余额差异 — 服务端首个用户得 200 NT，客户端得 0
**文件：** server/auth.py:59-60, nt-core.js:743-748, core.js:751
Client 的 `registerUser(n)` 无初始 deposit → 0 NT。进村时 sync 修复但此前可能误报余额不足。

### 16. 🟢 轮询无限制累积任务 → AppData._data.tasks 无限增长
**文件：** core.js:203-217
轮询仅添加任务。无清理或淘汰 → 数周后内存无限增长。
**修复：** 修剪仅本地且陈旧的任务。

### 17. 🟢 get_current_user 返回 None 而非抛 401 → 14+ 端点重复检查
**文件：** server/auth.py:36-41
每个端点必须手动 `if not user: raise 401`。一处遗漏 = AttributeError。
**修复：** get_current_user 直接抛 401。

### 18. 🟢 _saveState(300ms) 与 _saveShared/_savePrivate(200ms) 防抖间隔不同
**文件：** nt-core.js:64, app-data.js:217,236,244
不同间隔 → 崩溃时可能丢失 NT 状态但保留 AppData。
**修复：** 统一间隔，或 AppData save 前先 flush NT。

### 19. 🔴 三套并行 NT 模型 — 架构债（无单项修复）
服务端（SQLAlchemy）、客户端 NT core（nt-core.js）、客户端 AppData（app-data.js）各自独立建模 User/Task/Balance。每次同步需 8+ 处手动翻译。
**建议：** ① 统一状态词汇 ② 统一字段名 ③ 将 nt-core.js 变为 API 代理。
