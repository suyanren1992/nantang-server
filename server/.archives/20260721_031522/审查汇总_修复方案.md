### S1. index.html:718 — `];` 孤行语法错误 → 整个内联脚本崩溃，页面完全不可用
**源：** UI 程序员 #1

**修复：** 删除第 716-718 行（注释 + 悬空对象 + `];`）。`CAMP_MOCK` 已在 `data.js:517` 定义。

### S2. 住宿 overdueNT 几何级增长 — 第 N 天欠费 = 150 × 2^N
**源：** 经济设计师 #2
**文件：** `app-data.js:432`
**问题：** `totalDue` 已包含旧 `userOverdue`，而 `unpaid`（也包含旧欠费）又被 `+=` 累加到 `user.overdueNT`。第3天：应欠 450，实际欠 1050。第7天：应欠 1050，实际欠 19050。
**修复：** `user.overdueNT = unpaid;`（不是 `+=`）

### S3. `_dailyPoolRefill` guard 页面刷新丢失 → 刷 N 次 = N×50 NT 注入
**源：** 系统设计师 #1
**文件：** `app-data.js:218,566`
**问题：** `_lastPoolRefill` 不在 `_saveShared` 的序列化 payload 中。页面刷新后 guard 为 `undefined`，pool 再次 refill。
**修复：** 在 `_saveShared` payload 中加入 `_lastPoolRefill: this._data._lastPoolRefill`，并在 `_dailyPoolRefill` 末尾调用 `this._saveShared()`。

### S4. `_publishing` 锁在错误路径上永久死锁 → 所有后续发布被阻断

### S21. `unclaimTask` 双花 — 退款给 poster 后任务可被重新接取
**源：** 游戏设计师 #1
**文件：** `nt-core.js:273-282` · 无 server 端 unclaim
**问题：** `unclaimTask` 退 `TASK_ESCROW` 给 poster 并重置为 `pending`。重新 accept + verify 会再次扣减 TASK_ESCROW → 付给新 assignee 但钱已退回。净效果：poster 保留 NT + assignee 获得报酬 = 凭空印钞。
**修复：** (a) 不从 escrow 退款 — 仅清空 assignee 并保留冻结；或 (b) 如退款则设标志阻止重接。同时需加 server 端 unclaim 端点。

### S22. `update_task` (PUT /api/tasks/{id}) 绕过所有状态守卫
**源：** 游戏设计师 #2 · 安全工程师 #11
**文件：** `server/routes/tasks.py:95-126`
**问题：** 任何状态值可被直接设入，无验证。任务可从 `进行中` 直接跳到 `已结算`，跳过 escrow 释放。破坏会计等式。
**修复：** 加状态转换白名单。禁止直接设状态 — 强制通过专用端点（cancel/submit/verify）。

### S23. `task.escrow_amount` 从不递减 → 双重退款
**源：** 游戏设计师 #3, #10, #14
**文件：** `server/routes/nt.py:293,385-390` · `server/routes/tasks.py:143-151`
**问题：** `escrow_amount` 创建时设一次，verify/cancel 后从不更新。cancel/delete 读陈旧值 → 双重退款 → `pool.task_escrow` 变负。
**修复：** 每次 escrow 释放后将 `task.escrow_amount = 0`。`cancel_task`/`delete_task` 中屏蔽 `已取消` 状态。

### S24. `_mergeSyncData` 永不更新已存在任务 → 客户端状态永久陈旧
**源：** 系统设计师 #3 · UI 程序员 #6
**文件：** `core.js:699-702`
**问题：** 合并逻辑 `if (!dup)` 仅插入新任务。server 状态变更（pending→settled）不传播到已有客户端任务。用户看到永久陈旧状态。
**修复：** 去重后，对已存在任务应用 `Object.assign` 更新 status/assignee 等字段。
