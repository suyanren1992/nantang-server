## 执行协议（每轮必须）

```
方案 → 审查方案 → 执行 → 差分审查 → 断言验证 → 反角色抽查 → commit
                          ↑ 三步骤              ↑
                          │                    │
                          └── 不通过 → 修 → 重跑（最多 2 次）──┘
                              第 3 次仍不通过 → 暂停，人工介入
```

**刹车规则：** 同一轮重试上限 = 2。第 3 次任何步骤不通过 → 停止本轮，保留当前 diff 和失败日志，等人工判断。防止「修 A 引入 B → 修 B 引入 C → …」无限循环烧 token。

**成本估算（单轮 · 一次通过）：**

| 步骤 | 消耗 |
|------|------|
| 执行（写代码） | ~5000 tokens |
| ① 差分审查 | ~2000 tokens |
| ② 断言验证 | ~0（bash） |
| ③ 反角色抽查 | ~3000 tokens |
| **合计** | **~10000 tokens** |

最坏情况（2 次重试）= ~30000 tokens。9 轮 × 10000 = ~90000 tokens 全流程，约等于之前 8 Agent 审查的 70%。

### 步骤 1：差分审查（防副作用）

每轮执行完、commit 前，将本轮 `git diff` 喂给一个 Agent：

> 审查这个 diff。每处修改问三个问题：
> 1. 是否引入了新崩溃路径（空指针/未初始化/语法错误）？
> 2. 是否改变了相邻代码的假设（变量名/返回值/时序）？
> 3. 修改的代码路径是不是有调用方依赖旧行为？

Agent 用 `general-purpose`，无 schema，纯文本审查。输出 ≤5 条发现或「无新问题」。

### 步骤 2：断言验证（防功能回退）

每轮方案末尾写的验证命令 **必须实际执行**，不是可选项。未通过 → 不回滚、不前进，停在当前轮修到通过为止。

### 步骤 3：反角色抽查（30 秒快速审计）

每轮随机抽一个**反角色** Agent，给本轮改过的文件列表，让它 30 秒内找新 bug：

| 本轮角色 | 反角色抽查者 |
|---------|------------|
| 🔴 安全修复 (R7) | 🧪 QA 测试 |
| 🔴 经济修复 (R8) | 🔒 安全工程师 |
| 🔴 状态机修复 (R9) | 💰 经济设计师 |
| 🟡 同步/数据层 (R10) | 🎮 游戏设计师 |
| 🟡 权限/范围 (R11) | ⚙️ 系统设计师 |
| 🟡 客户端修复 (R12) | 🖐️ 交互设计师 |
| 🟡 autoLogin/UX (R13) | 🖥️ UI 程序员 |
| 🟢 清理/纵深 (R14) | 🎬 创意总监 |
| 🔵 连通性 (R15) | 🔒 安全工程师 |

Prompt 模板：

# create_access_token — 加 token_version 参数：
def create_access_token(user_id: str, role: str, token_version: int = 0) -> str:
    expire = datetime.utcnow() + timedelta(minutes=ACCESS_TOKEN_EXPIRE_MINUTES)
    to_encode = {"sub": user_id, "role": role, "jti": str(uuid.uuid4()), "type": "access", "version": token_version, "exp": expire}

# create_refresh_token — 加 token_version 参数：
def create_refresh_token(user_id: str, token_version: int = 0) -> str:
    expire = datetime.utcnow() + timedelta(days=REFRESH_TOKEN_EXPIRE_DAYS)
    to_encode = {"sub": user_id, "jti": str(uuid.uuid4()), "type": "refresh", "version": token_version, "exp": expire}

**R7-7e. auth.py logout** — 手动读 token 避免与 R11-8 冲突：
```python
@router.post("/logout")
async def logout(request: Request, response: Response, db: AsyncSession = Depends(get_db)):
    # 手动读取 token，不做强制认证（无 token 用户也要能删 cookie）
    auth_header = request.headers.get("authorization", "")
    if auth_header.startswith("Bearer "):
        payload = decode_token(auth_header[7:])
        if payload and payload.get("sub"):
            u = (await db.execute(select(User).where(User.id == payload["sub"]))).scalar_one_or_none()
            if u and payload.get("version") == u.token_version:
                u.token_version += 1
                await db.commit()
    response.delete_cookie("nt_rt", path="/")
    return {"ok": True}
```
> ⚠️ 不能用 `Depends(get_current_user)`——R11-8 之后 get_current_user 对无效/过期 token 直接抛 401，无 token 用户永远删不掉 cookie。改为手动读 header，token 无效时静默跳过 version 递增。

> S27 原始需求含「密码修改时递增 token_version」。当前 auth.py 无修改密码端点，此项延后。添加端点时在密码更新逻辑中加 `user.token_version += 1`。
