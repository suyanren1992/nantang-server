## 发现

### 数据层一致性

```
🟡 | app-data.js:223 | _saveShared 的服务器同步 payload 中 `inventory: this._data.items` 应为 `this._data.inventory`
后果：`this._data.items` 是按用户索引的物品字典（`{user1:[...], user2:[...]}`），而 `this._data.inventory` 才是 init 中初始化为 `{}` 的共享库存。由于服务器 `/api/data/sync_shared` 目前不处理 inventory 字段，数据被丢弃，暂无实际影响。
建议修复：将 line 223 改为 `inventory: this._data.inventory`；或若服务器将来接收此字段，当前即为逻辑错误。
```

```
🔴 | app-data.js:9-42 (init) vs app-data.js:215 (_saveShared) | `campRmb` 保存在 _saveShared 中但 init 从未初始化
后果：`_data.campRmb` 在首次保存前为 `undefined`，`JSON.stringify` 会静默忽略该键。若用户在 campRmb 被外部代码设置前刷新页面，该值丢失。core.js line 70-71 读取 `AppData._data.campRmb` 时得到 `undefined`（替代期望的 0）。
建议修复：在 init 中添加：`if (this._data.campRmb === undefined) this._data.campRmb = 0;`
```

```
🔴 | app-data.js:547 (_tickDirtiness) | 使用 `new Date().toISOString().slice(0,10)` 获取今日日期，而未使用 `Clock.today()`
后果：与 `_deductAccommodation` (line 406，使用 Clock.today()) 不一致。(a) 当 Clock 冻结在虚拟时间进行测试时，脏污度仍按真实时间每日 tick。(b) `toISOString()` 返回 UTC 日期，与 Clock.today() 的 UTC+8 日历日可能差一天（在 UTC+8 的 0:00-7:59 期间）。同样问题出现在 `_dailyPoolRefill` (line 560)。
建议修复：两个函数统一使用：`var today = (typeof Clock !== 'undefined' ? Clock.today() : new Date().toISOString().slice(0,10));`
```

### API 通信层

```
🟡 | api.js:90-98 (syncTask) | 客户端创建任务时未发送 `req_photo` 和 `req_file` 字段
后果：服务器 `TaskCreate` 模型定义了这两个字段（默认值为 0），doPublish (core.js:548) 在客户端设置了 `reqPhoto`/`reqFile` 到任务对象的 data 中，但这些值在 `syncTask` 中从未传给服务器。服务器端照片/文件提交要求始终为 0。
建议修复：在 api.js syncTask 的 data 对象中添加：`req_photo: task.reqPhoto || 0, req_file: task.reqFile || 0`
```

### Bridge & 路由

```
🟡 | data.js:500 (closeOverlay) | closeOverlay 无条件执行 `document.getElementById('villagePage').classList.remove('hidden')`
后果：当从 Game bridge（line 100: `closeOverlay('overlayMap'); openQuestHallPage()`）调用时，villagePage 被短暂显示后再被 overlay 覆盖。产生视觉闪烁。在 hash 路由包裹层（index.html:1330）中同样存在：`closeOverlay` 设置 hash='' 后又紧接其他 open 函数设置新 hash，产生两次 hash change。
建议修复：可考虑将 villagePage 的显示逻辑移出 closeOverlay，交由各 open 函数自行决定；或增加参数控制是否展示 villagePage。
```

```
🟡 | core.js:103 (Game.confirm) vs core.js:1267 | Game.confirm 使用原生 `confirm()` 弹窗但永不回复 iframe；message handler 中的 confirm 通过 postMessage 回复
后果：两套 confirm 机制行为不一致。Game.confirm 是同步弹窗（`setTimeout(onConfirm, 0)` 无 postMessage 回调），iframe 无法收到确认结果；message handler 中的是异步消息回复。地图 iframe 应统一使用 message 通道。
建议修复：在 Game.confirm 中增加 callback ID 追踪，或让地图只使用 message 通道。
```

### 数据存储

```
🔴 | app-data.js:38-42 (init) vs app-data.js:238 (_savePrivate) | `_savePrivate` 保存 `myItems` 和 `items` 两个字段，但 init/switchUser 加载后，只有 `items` 被实际读取（`myItems()` line 145 读的是 `items[user]`）
后果：`myItems` 字段被加载但从未被使用——它是死数据。`_savePrivate` 中保存 `myItems` 浪费存储空间。同时 `_saveShared` 的服务器同步 payload 错误地将 `items`（用户字典）作为 `inventory` 发送，混淆了 `items` 与 `inventory` 的语义。
建议修复：明确 `items` 的语义（按用户索引的私有物品）与 `inventory`（共享社区库存）的区别。考虑移除未使用的 `myItems` 字段，或若它表示"当前用户物品"，则在 switchUser 中正确填充而非留空。
```

### 服务端数据完整性

```
🟡 | data.py:108-111 (get_verifications) | API 返回字段不完整——Verification 模型有 17 列，但仅返回 7 个字段
后果：缺少：`verifier`（校核人）、`verified_at`（校核时间）、`reject_reason`（拒绝理由）、`rejected_by`、`rejected_at`、`retry_count`、`detail`。当前客户端未调用此端点（校核系统纯客户端运行），但若将来有管理面板或跨设备同步需要此接口，返回数据不完整。
建议修复：补全返回字段至与模型一致。
```
