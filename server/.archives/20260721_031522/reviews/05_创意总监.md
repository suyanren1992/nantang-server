## Dual-Mode Consistency (双模式一致性)

```
🔴 | nt-core.js:102-124 | deposit()/withdraw() return null unconditionally
设计影响：这两个函数名与topUp/cashOut表达不同语义（保证金 vs 充值/提现），但都被导出到window.NT中。调用者期待保证金操作但永远得到null，易产生静默bug。
建议修复：从NT导出对象中移除deposit/withdraw，或让它们委托给topUp/cashOut并标注语义差异。

🔴 | nt-core.js:274-347 | batchSettle() 标注为 DEAD CODE 但仍被导出
设计影响：任何人调用batchSettle会导致双重扣款（verifyTask已完成了转账），直接破坏会计等式。
建议修复：在batchSettle入口处加 early return + console.error，或从导出对象中移除。

🔴 | nt.py:81-85 vs core.js:862-879 | 服务端NT操作不更新contribution_value，客户端更新
设计影响：HTTP模式下用户完成任务的CV不会增长，而file://模式下CV正常增长。两种模式下的用户画像数据不同步。
建议修复：nt.py的earn端点加 `user.contribution_value += req.amount`；spend端点加 `user.contribution_value = max(0, user.contribution_value - req.amount)`。

🔴 | nt.py:134-155 vs nt-core.js:354-437 | 服务端earn/spend不支持camp_pool，只走community_pool
设计影响：三池体系（社区池/营队池/任务托管）在HTTP模式下退化为单池。营队预算注资会向camp_pool充值，但用户端无法从camp_pool收支。
建议修复：nt.py增加camp_pool字段到CommunityPool模型，earn/spend端点支持pool参数。

🔴 | nt.py:109-131 vs nt-core.js:449-467 | 服务端transfer不实现CV转移规则（75%/25%分润）
设计影响：HTTP模式下的转账等同于直接打款，缺失CV磨损机制和公共池积累。
建议修复：nt.py transfer端点增加CV转移逻辑，与客户端对齐。

🟡 | api.js:76,86 | getLedger/getPools有服务端端点但在离线模式静默失败
建议修复：调用点（如refreshUserUI）在收到_offline时应fallback到本地NT数据。
```

## Design Intent Alignment (设计意图对齐)

```
🔴 | nt.js:192-194 | recordTransaction() 创建tx对象但不存储——空操作
设计影响：函数名和签名让人觉得它在记录交易，实际是no-op。
建议修复：要么删除此函数，要么让它调用NT._addLedger()完成实际记录。

🔴 | nt-core.js:28 vs core.js:9 | LEDGER变量名冲突——nt-core.js用LEDGER[]存流水，core.js注释"FIX-15: LEDGER 已废弃"
设计影响：两个不同文件对LEDGER有相反的理解。
建议修复：core.js的FIX-15注释应明确说明"旧UI中的_ledgerOpen/旧LEDGER对象已废弃，改用NT.getLedger()"。

🟡 | nt.js:45-58 | calcNtByScope() 基于tx.scope字段过滤，但_addLedger不写scope字段
🟡 | nt.js:1-13 | nt.js头部注释说"依赖 data 全局对象"，这些在当前架构中可能不存在
🟡 | index.html:711-713 | 残骸数据——旧营队种子数据孤行，不属于任何变量/数组
建议修复：删除该孤立对象。
```

## Functional Symmetry (功能对称性)

```
🔴 | nt-core.js:172-184 vs 缺失 | acceptTask有实现，但没有对应的unclaimTask/releaseTask
设计影响：用户接取任务后无法放弃。NT会被永久冻结在TASK_ESCROW中。
建议修复：增加unclaimTask(taskId, assigneeId)。

🔴 | nt-core.js:263-268 | disputeTask有实现，但没有resolveDispute
设计影响：一旦任务被标记为disputed，没有任何API可以将其恢复。托管NT无法释放。
建议修复：增加resolveDispute(taskId, resolution)。

🔴 | 服务端nt.py | 缺少cancel_task、dispute_task、submit_task端点
设计影响：HTTP模式下，这三个操作客户端存在但服务端无对应API。
建议修复：在nt.py增加POST /tasks/{id}/cancel、POST /tasks/{id}/dispute、POST /tasks/{id}/submit端点。

🟡 | app-data.js:148-158 | addItem有实现，但没有deleteItem
🟡 | nt-core.js | cashOut不检查_totalIssued是否足够
```

## Old/New Version Confusion (新旧版本混淆)

```
🔴 | nt-core.js + nt.js + server/models.py | 三套并行的NT数据模型同时存在
设计影响：(1) nt-core.js有自己的USERS/TASKS/LEDGER (2) nt.js假定了data.finance和data.deposits (3) server端有数据库模型。三个系统各自记账，没有统一真相源。HTTP模式下客户端和服务器NT状态存在漂移。
建议修复：明确宣布哪一个是唯一的真相源。HTTP模式以server为准（客户端NT仅作缓存/离线fallback），file://模式以nt-core.js为准。

🔴 | core.js:1002-1010 vs index.html:1221-1224 | _avatarStyles/_avatarSeedPool/fillSeedPool 被定义两次
设计影响：core.js先定义并调用fillSeedPool()。index.html再次定义并调用fillSeedPool()。
建议修复：删除index.html中的重复定义。

🔴 | core.js:17-19 vs index.html:1218 | _secFold定义了两次
设计影响：同一变量在core.js:197和index.html:1218各定义一次。
建议修复：删除index.html中的重复定义。

🟡 | nt.js:18-20 vs core.js:3-4 | ROLE_TITLES 与 roleIcon()/roleName() 功能重叠
🟡 | core.js:10 vs index.html:505 | TASKS别名的赋值时机有竞争
```

## 总结

| 严重度 | 数量 | 关键主题 |
|--------|------|----------|
| 🔴 严重 | 9 | 三套并行NT模型、缺少3个关键端点、batchSettle死代码、server earn无CV、server无camp_pool支持、transfer无CV、重复定义、deposit/withdraw静默失败 |
| 🟡 重要 | 13 | 角色系统不一致、数据主权违规、技术债、旧数据残骸、TASKS别名竞争 |
| 🟢 通过 | 4 | ntBalance委托NT、角色过滤、admin权限、类型检查防护 |

最严重的问题（建议优先修复）：
1. 三套并行的NT数据模型没有统一真相源
2. 服务端NT缺少cancel/dispute/submit端点
3. batchSettle死代码暴露且可触发双重扣款
4. 服务端earn不更新CV、不支持camp_pool——HTTP模式下经济系统两个子系统完全失效
5. 任务生命周期缺少unclaimTask和resolveDispute
