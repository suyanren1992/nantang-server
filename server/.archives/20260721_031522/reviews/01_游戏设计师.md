## 发现

### 1. 任务生命周期完整性

```
🔴 | data.js:322 | doSubmit 无条件设 status='待审核'，阻断多人任务其他认领者提交
后果：3人任务，第1人提交后任务变为"待审核"，第2/3人无法提交（UI 不显示提交按钮）。
      所有人的成果必须同时审核，但未提交的人被永久卡住。
建议修复：doSubmit 检查 slots > 1 且 claimants 未全部提交时，只更新 claimant 状态
      不改变 task.status。仅当全部 claimants 都提交后才将 task.status 改为"待审核"。
```

```
🔴 | data.js:333-338 | claimTask 不检查 slots 上限
后果：t.slots=1 的任务可被无限人认领。UI 隐藏了认领按钮（renderTaskCard line 301），
      但 toggleQuestCard（line 369）同样无 slots 检查。客户端可绕过 UI 直接调用。
建议修复：claimTask 开头加 `if((t.claimants||[]).length >= (t.slots||1)) { showToast('名额已满','error'); return; }`
```

```
🔴 | server/tasks.py:98 | update_task 审批评审权限仅检查 poster==user 或 admin，不检查 reviewer 字段
后果：任何人都可以设定 reviewer，但 reviewer 字段完全没有服务端强制力。
      发布者可以指定审核人，但实际审批操作不需要 reviewer 参与。
      这使 reviewer 字段沦为装饰，破坏审核机制的公信力。
建议修复：update_task 审批操作加 `if task.reviewer and task.reviewer != user.id and user.role != 'admin': raise 403`
```

```
🟡 | core.js:489 | openPublishTask 默认设 reviewer=CURRENT_USER（自己审核自己）
后果：发布者默认审核自己的任务，审核人机制形同虚设。只有用户主动修改才会换人。
建议修复：reviewer 默认为空，提示"请选择审核人"。或默认设为第一个 admin 用户。
```

```
🟡 | data.js:348-357 | unclaimTask 创建 unclaimRequest 但无任何处理器消费它
后果：认领者申请取消认领后，发布者看不到这个请求，也无法批准/拒绝。认领者被永久锁定。
      这是单向孤儿状态——请求发出后无人处理。
建议修复：在发布者的任务管理界面增加"待处理的取消申请"列表，允许发布者批准/拒绝。
```

### 2. 任务状态机

```
🔴 | core.js:259,979 | "已完成" 状态在 UI 过滤器中被引用但从未被任何转换路径生成
后果：状态机路径为 进行中→待审核→待结算→已结算，中间不存在"已完成"。
      如果通过服务端 PUT 手动设为"已完成"，无后续转换逻辑（不可结算）。
      过滤器中包含它只是为了不丢失数据——属于垃圾回收状态。
建议修复：要么从过滤器中移除"已完成"（统一使用"待结算"→"已结算"），
      要么定义"已完成"的明确用途及其转换路径。
```

```
🟡 | core.js:376-384 + data.js:359-384 | 审核打回可无限循环
后果：审核人打回 → 认领者重新提交 → 再次打回 → 再次提交… 无次数上限，
      无 NT 惩罚，审核人和认领者可形成死循环。
建议修复：加 rejectCount 字段，超过3次打回任务自动关闭并退还托管 NT。
```

```
🟡 | core.js:7 | isTaskOverdue 不触发任何状态变更
后果：逾期任务仅显示"⚠️逾期"标签，但任务状态永不改变。逾期任务永远留在"进行中"
      或"待审核"中。无自动清理、无自动退回、无自动关闭。
建议修复：增加定时检查或页面加载时的逾期处理——逾期"进行中"任务自动标记为"已过期"，
      逾期"待审核"任务自动退回给认领者（减 NT 作为惩罚）。
```

```
🟡 | server/tasks.py:115-116 | completed_at 在状态改为"待审核"时设定，打回后不清除
后果：提交→审核打回→重新提交→再次打回，任务可能被打回多次，
      但 completed_at 始终指向第一次提交的时间。数据不准确。
建议修复：打回时清除 completed_at，仅在最终通过时设置。
```

### 3. 多人任务 (slots/claimants)

```
🔴 | data.js:322, 359-364 | 多人任务审核通过时，未提交的认领者状态不同步
后果：3人任务，2人提交1人未提交，审核通过后：2人标记 completed，1人保持 in_progress。
      结算时只有 completed 的人获得 NT，in_progress 的人被遗忘。
      未提交者的席位被浪费。
建议修复：审核前检查是否所有 claimants 都已提交。如果未全部提交，
      提示"还有 N 人未提交，确认通过将放弃他们的认领"。
```

```
🟡 | core.js:301-303, 369 | renderTaskCard 和 toggleQuestCard 的认领按钮不检查剩余名额
后果：虽然 slots 在 UI 显示，但认领按钮的可见性仅由 status 决定（status==='进行中'）。
      满额任务的认领按钮依然可见（只是后续 claimTask 也不检查）。
建议修复：按钮条件加 `&& (t.claimants||[]).length < (t.slots||1)`
```

### 4. 营地系统

```
🟡 | ui-wizard.js:52-54, 60-69 | 创营向导仅验证 Step 1 和 Step 5，其他步骤无验证
后果：Step 2 可跳过日期设置，Step 3 可设负预算，Step 4 可无任务，
      Step 6 可不发确认请求直接跳过。
建议修复：Step 2 验证 startDate 不为空；Step 3 验证 builder>=1；
      Step 4 验证 tasks.length>0（或允许为空但警告）。
```

```
🟡 | ui-wizard.js:557-560, server/camps.py:77-82 | 营地 NT 总池只计入 lodging+meal，不含 extraExpense
后果：建营预算中有额外支出项（场地租金、画材采购、讲师酬劳等），但这些不纳入 NT 总池计算。
      如果营队任务 NT 来自额外支出项，总池会偏低，导致超支检测误报。
建议修复：将 extraExpense 中标记为 NT 的项目也纳入总池计算；或明确约定 NT 池仅用于 lodging+meal。
```

```
🟢 | server/camps.py:35-90 | Camp 创建接口接收裸 dict，无 Pydantic 模型校验
后果：任何格式错误的 JSON 都能写入数据库。budget 字段可为任意结构，
      schedule 格式不校验。前端重构时容易产生数据脏写。
建议修复：至少定义 CampCreate Pydantic 模型校验必填字段。
```

```
🟢 | server/camps.py:64 vs models.py:85 | Camp 服务端模型缺少 season 字段
后果：客户端 launchCamp 传入 season（"第四期"等），但服务端 Camp 模型无此列。
      数据存入数据库时会丢失 season 信息。
建议修复：服务端 Camp 模型添加 season 列并写入 camps.py create/update 逻辑。
```

### 5. 审核人机制

```
🔴 | server/tasks.py:76, 98 | reviewer 字段服务端零校验 —— 可设为空字符串或任意不存在的人
后果：reviewer="" 时，UI 不显示审核按钮（core.js line 353 检查 t.reviewer 存在性），
      但 update_task 不检查 reviewer 匹配，任何 admin 都能审批。
      实质上是：reviewer 字段没有任何约束力。
建议修复：创建时校验 reviewer 为有效用户；审批时校验 user.id==task.reviewer || admin。
```

### 6. 前后端字段映射

```
🔴 | core.js:204-206, 689-693 | 服务端返回 location_id 但客户端 sync 不映射为 locationId
后果：服务端 POST/PUT 存储了 location_id 关联地图建筑，但客户端从服务端拉取任务列表后
      丢弃了该字段。任务失去地点关联，地图交互失效。
建议修复：sync 代码加 `locationId: t.location_id || ''`
```

```
🟡 | core.js:204 vs server/tasks.py:53 | 客户端 type 映射到 server category，值域不兼容
后果：客户端 type="主线"/"支线"/"在地任务"/"日程任务"/"多人任务"/"单人任务"
      服务端 category="other"（默认）且无枚举约束。
      同步后客户端将 server 的 category 值当作 type 渲染，可能导致未知类型显示异常。
建议修复：统一任务类型枚举，服务端用 category 字段存储枚举值，文档化映射关系。
```

```
🟡 | core.js:204 | 客户端 sync 不映射服务端的 assignee 和 evidence 字段
后果：服务端有 assignee（认领人）和 evidence（提交证据）字段，
      但客户端使用 claimaints 数组管理多人认领。完全独立的两套数据模型。
      如果有人通过 API 直接修改 assignee，客户端不感知。
建议修复：明确客户端 claimsants 与服务端 assignee 的关系——二选一或做双向同步。
```

```
🟢 | server/tasks.py:53-58 | list_tasks 返回 created_at 和 settled_at 但不返回 accepted_at 和 completed_at
后果：客户端无法知道任务被领取和完成的时间戳。时间线（core.js line 334-343）依赖客户端
      本地字段 claimedAt/completedAt/reviewedAt，与服务端数据不一致。
建议修复：list_tasks 返回 accepted_at 和 completed_at。
```

```
🟢 | core.js:204 vs models.py:44 | 服务端 NTTask 默认 status="pending"，但 create_task 硬编码 "进行中"
后果：如果任何其他代码路径创建 NTTask 而不显式设 status，会得到 "pending" 状态，
      客户端不认识此状态，不会在任何过滤器中显示——任务隐身。
建议修复：将模型默认值改为 "进行中"，或在所有创建路径显式设置 status。
```

## 总结

| 严重度 | 数量 | 关键主题 |
|--------|------|----------|
| 🔴 严重 | 6 | 多人任务提交阻断、slots 无上限检查、reviewer 无服务端校验、location_id 丢失、"已完成"死状态、self-review 默认值 |
| 🟡 重要 | 8 | 无限打回循环、逾期无自动处理、unclaimRequest 孤儿、type/category 不一致、creator 缺验证、NT池只含住宿餐饮、completed_at 不清除 |
| 🟢 次要 | 5 | 模型默认值不一致、season 字段缺失、accept 时间戳不返回、Camp创建无schema校验、assignee/evidence 不映射 |

最紧急修复项：`doSubmit` 对多人任务的阻断（1），`claimTask` 缺少名额检查（2），以及服务端 reviewer 权限校验缺失（3）。这三项直接导致核心 gameplay 流程可被破坏。
