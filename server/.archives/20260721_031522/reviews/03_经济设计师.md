# 经济设计师审查报告

> 审查日期：2026-07-21
> 审查人：💰 经济设计师 Agent
> 审查范围：NT 经济系统 / 会计等式 / 池子 / 校核制

---

## 1. 会计等式完整性

**Client-side (`nt-core.js`):** 等式 `sum(user.ntBalance) + COMMUNITY_POOL + sum(CAMP_POOLS) + TASK_ESCROW == _totalIssued` 在所有核心路径中正确维护（createTask, verifyTask, cancelTask, topUp, cashOut, earnFromPool, spendToPool, depositToCommunityPool）。

**Server-side (`nt.py`):** 等式 `sum(user.nt_balance) + pool.balance + pool.task_escrow + camp_pool_balance == pool.total_issued` 有维护，但 `camp_pool_balance` 从 ledger 动态计算而非存储字段，脆弱。

```
🟡 | nt.py:213-217 | camp_pool_balance computed from ledger scan, not stored field
会计影响：If a ledger row is corrupted or missing, the verify check silently passes with wrong camp_pool_balance.
建议修复：Add a `camp_balance` column to CommunityPool, or validate that the ledger-derived camp_pool_balance is >= 0.
```

---

## 2. spend 函数 — scope 参数和池子路由

```
🔴 | nt-core.js:358-360 | spend() always routes to 'community' pool, scope parameter is misleading
会计影响：The wrapper `spend(userId, amount, reason, scope)` calls `spendToPool(userId, amount, reason, 'community', scope)`. scope 只影响 ledger type 标签（如 "camp_spend" vs "personal_spend"），但从不改变钱去哪个池子。scope='camp' 的 spend 仍然进 COMMUNITY_POOL。
建议修复：Either route camp-scoped spends to CAMP_POOLS, or remove the pool parameter from the wrapper.

🔴 | nt.py:175 | Server spend scope→ledger mapping is inverted and inconsistent with earn
会计影响：Spend writes `to_user="community_pool"` when `scope=="personal"`, and `to_user=None` otherwise. But earn writes `from_user="community_pool"` when `scope=="camp"`. 条件不一致，ledger-based camp_pool_balance 计算不可靠。
建议修复：Unify the scope→ledger mapping.
```

## 3. earn 函数 — CV 和 XP

```
🔴 | nt.py:147-148 | Server earn increments XP but NOT CV
会计影响：Client `earnFromPool` adds both `contributionValue` and `experienceValue` by `amount`. Server `earn` only adds `experience_value`. HTTP 模式下用户 CV 永不增长，影响 CV-gated 功能。
建议修复：Add `user.contribution_value += req.amount` to server earn endpoint.

🟡 | nt-core.js:428 | Client spendToPool decrements CV but server spend does not
建议修复：Decide whether CV should be affected by spend at all, then align both implementations.

🟡 | nt-core.js:213-216 | verifyTask (NT task system) does not add CV or XP
建议修复：Decide: should task completion grant CV/XP like other earns?
```

---

## 4. topUp vs deposit

```
🟡 | nt-core.js:102-111 | deposit() is dead code with accounting hazard
会计影响：The function returns null immediately (deprecated), but the dead code below would increment `depositBalance` and `_totalIssued` without incrementing `ntBalance`. 如果函数被重新启用，会计等式会断裂。
建议修复：Delete the dead code entirely.

🟡 | nt.py:topup | Server has topup but no cashout endpoint
建议修复：Add `/api/nt/cashout` endpoint with admin-only access.
```

---

## 5. cashOut vs withdraw

```
🟡 | nt-core.js:113-124 | withdraw() is dead code with same hazard as deposit()
建议修复：Delete dead code.

🟡 | nt-core.js:383 | cashOut increments totalPaid
会计影响：`totalPaid` 语义上是任务付款，不是提现。混合数据使 `totalPaid` 计数器无意义。
建议修复：Either remove totalPaid increment in cashOut, or rename to totalSpent.
```

## 6. verifyTask — TASK_ESCROW 处理

```
🟢 | Both client and server | Approved and rejected paths correctly balance TASK_ESCROW.

🟡 | nt.py:331-332 | Rejected verify with missing poster puts escrow into community pool without ledger entry
会计影响：Poster 已删除时退回的资金没有 ledger 记录，钱从用户空间消失进入池子但无迹可查。
建议修复：Add a ledger entry for the fallback path (type: "escrow_absorbed", to: "community_pool").
```

## 7. deleteTask / cancelTask — 托管一致性

```
🟡 | nt-core.js:244 vs tasks.py:130 | Status gates for cancellation differ between client and server
会计影响：Client cancelTask 只允许 'pending' 和 'active'；Server delete_task 允许 'completed'。
建议修复：Align status gates.

🟢 | Both client and server | Escrow return logic is identical: `TASK_ESCROW -= reward; poster.ntBalance += reward;`
```

---

## 8. 住宿费自动扣费

```
🔴 | app-data.js:417-431 | overdueNT is informational only -- never collected
会计影响：`totalDue = roomPrice * daysPassed` 只收当天租金，`room.overdueNT` 历史欠费从不加入 totalDue。欠费无限滚雪球。
建议修复：Include `room.overdueNT || 0` in the `totalDue` calculation.

🟡 | app-data.js:428 | overdueNT is stored on room, not user
会计影响：租客搬到其他房间后，欠费留在旧房间，新房客继承债务。
建议修复：Store `overdueNT` on the user object (via NT system).
```

## 9. verifyAction 双重 earn

```
🟢 | app-data.js:378-383 | Two separate earn calls for two different users -- not a double charge

🟡 | app-data.js:379 | scope='camp' passed to NT.earn but never affects pool routing
会计影响：`NT.earn(vfy.doer, vfy.ntAmount, vfy.action, 'camp')` 看起来像从营队预算支出，但实际从 COMMUNITY_POOL 扣。
建议修复：Either route camp-scoped earns to CAMP_POOLS, or rename scope values.
```

---

## 10. 营地预算注资

```
🔴 | camps.py:82-87 | camp_total is added to total_issued but camp_pool is never funded as a spendable balance
会计影响：营队预算写入 ledger 但从不进入可消费的 camp_pool。server earn 永远从 community pool 取钱。
建议修复：Either add a `camp_balance` field to CommunityPool and modify earn to support camp-pool withdrawals.

🔴 | nt-core.js | CAMP_POOLS on the client is NEVER funded
会计影响：客户端 CAMP_POOLS 永久为空，earnFromPool('camp:xxx') 永远失败。
建议修复：Either add `depositToCampPool()` function, or remove camp pool infrastructure.
```

## 额外发现

```
🔴 | nt-core.js:47 | _loadState has fallback chain sp→cmp that conflates community pool with a legacy field
会计影响：Old state with `sp` is loaded into COMMUNITY_POOL, but `_totalIssued` is from `s.ti`. If `sp` and `ti` were from different snapshots, accounting mismatch from load.
建议修复：Remove the legacy `sp` fallback or validate that the equation holds after loading.
```
