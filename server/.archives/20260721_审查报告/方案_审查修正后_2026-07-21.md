### A9. 登录点不进去 ✅ 已修复（2026-07-21 复现定位）

**根因**：`api.js` `asyncAuth(type,name,password,role,seed,inviteCode,callback)` 是 7 参函数，`core.js` 登录调用只传 6 个实参（漏了 inviteCode）→ 回调被挤到 inviteCode 位置、`callback=undefined` → 成功/失败/异常三个分支都不调回调 → 按钮完全无响应。注册传 7 参正常，所以只有登录坏。服务端本身无恙（实测 /api/auth/login 正常返回业务错误）。

**修复**：`core.js` 登录调用补第 6 参 `null`。

**现象**（历史记录）：用户报告登录按钮无响应。

**已知可能原因**：
1. HTTP 模式下服务端未启动 → `API.asyncAuth` 静默失败，错误未 surfaced
2. `_finishEnter` 中某步抛异常（如 `_saveLocalUser` 写 localStorage 失败）
3. `index.html:68` 的 `onclick="enterVillage()"` 依赖全局函数——如果 JS 加载失败则不存在

**排查步骤**：
1. 打开 DevTools Console，点登录，看是否有红色报错
2. 检查 Network 标签——`/api/auth/login` 请求是否发出，状态码
3. 检查 `localStorage` 是否已满（QuotaExceededError）
4. 如果服务端未启动 → 启动服务端；如果是 JS 错误 → 修

## Phase B · 🟡 功能正确性（10 项）

### B1. 多槽任务 escrow 退款 × slots + 台账金额

**根因**：createTask 冻结 `reward × slots`，cancel 和 rejected 只退 `reward × 1`。**AND** 台账 `task_freeze` 只记 `reward` 不记 `reward × slots`——会计等式校验时 escrow 与台账对不上。

**改**：
- `nt-core.js:247` 和 `:229` — `TASK_ESCROW -= t.reward` → `t.reward * (t.slots \|\| 1)`
- `nt-core.js:158` — 台账 `task_freeze` 金额 `t.reward` → `t.reward * slots`
- `server/routes/tasks.py:93` — 同上，服务端台账金额修正
- **🆕 `server/routes/nt.py:414`** — verify 通过时 `pool.task_escrow -= task.reward * len(assignee_ids)` → `pool.task_escrow -= task.escrow_amount`。原因：部分填充任务（5 名额只有 2 人认领→验证），残留 escrow = `reward × (slots - len(assignee_ids))` 永不被释放，等式左侧永久偏高

### B2. disputed 加退出路径

**根因**：disputed 后不能重验、取消、重接——永久死锁。

> ⚠️ 客户端 `nt-core.js:248-268` cancelTask **已允许 disputed**（守卫不拦截它）。服务端 `nt.py:459` 封锁 `已争议`——争议任务只能走 admin resolve，不能自助取消（合理设计）。

**改**：
- ~~`nt-core.js` cancelTask~~ — 无需修改，已完成
- `nt.py` 加 `POST /api/nt/tasks/{id}/resolve`：admin 强制结束争议，escrow 退还给发布者。Request: `{ action: "refund_poster" | "release_assignee" }`。Response: `{ task_id, new_status: "已取消" | "待结算", ledger_entries: [...] }`（注意真实状态值，不是 `resolved`）

### B3. 统一状态词汇 + UI 不丢任务 ⚠️ 临时

> **审计修正 D4**：B3 的部分状态修正将在 Phase D·D1 中被全面替换（D1 统一三层的 TASK_STATUSES 常量）。B3 是临时补丁——如果 D1 两周内执行，跳过 B3。

**根因**：钱包存在三处（nt_users、AppData、服务端），HTTP 模式充值读不到。

**改**：`showRechargeForm()` / `submitRecharge()` — 只读 `AppData.me().wallet`

### B5. _processedTxIds 加 TTL

## Phase C · 🔷 客户端-服务端信任架构

> 详见 `c:\Users\苏砚仁\.claude\plans\client-server-trust.md`（若不存在，本文 Phase C 已包含完整设计）

**目标**：服务端是唯一真相源。客户端只是只读缓存 + 操作提交器。

**原则**：
1. 所有 NT 写操作先过服务端 → 成功才更新本地
2. 客户端余额 = 上次同步到的服务端值
3. 客户端 ledger = 服务端 ledger 的只读缓存
4. 登录时全量同步覆盖本地状态

### 阶段 C0：加 sync 端点（30min）

`GET /api/nt/sync` — 返回余额 + CV + XP + 近期流水 + 任务 + 冻结余额 + **充值意图 + 校核记录 + 角色 + 入住状态**

> **审计修正 U1**：sync 载荷加 `deposit_intents`（用户本人的充值意图列表），解决 pendingTransactions 与 DepositIntent 双轨不同步。
> **审计修正 U2**：sync 载荷加 `role` + `accommodation`（当前用户的角色和入住状态），为身份系统提供服务端权威数据源。
> **审计修正 U3**：sync 载荷加 `pending_verifications` + `verification_log`，为校核室/卡片室提供数据源。

### 阶段 C1：登录强制全量同步（20min）

`enterVillage()` / `_finishEnter()`：登录后 await sync → 覆盖本地 → 渲染 UI

### 🏗️ 组 2：需要功能设计

#### D2. 双路径贡献架构：卡片式任务 ↔ 校核制

**设计问题**：当前卡片式任务（createTask→accept→submit→verify）和校核制（addVerification→verifyAction）是两套独立系统，用户无法选择。校核制覆盖了 8 种动作类型（cleaning/stock_in/stock_out/field_harvest/field_action/daily_container/labor_report/store_in），全部纯客户端运行，NT  payout 永远不同步到服务端。

**用户需求**：做一件事时，用户应能选择走哪条路径——
- **卡片式**：发布者出 NT 进 escrow → 有人认领 → 提交 → 发布者审核 → NT 释放。适合「我需要人帮我做 X」的委派场景。
- **校核制**：做完 → 提交公开校核 → 任何 peer 验证 → NT 从社区池发放。适合「我做了 X，大家都能看到，谁来验证一下」的自主贡献场景。大扫除天然属于此类。

**设计方案**：

```
用户想做一件事
    │
    ├─ 委派给别人做？
    │   → 卡片式任务：createTask(escrow) → accept → submit → verifyTask
    │   → NT 来源：发布者 escrow
    │   → 验证者：发布者（任务 owner）
    │
    └─ 自己做，要人验证？
        → 校核制：doAction → addVerification → peer verifyAction
        → NT 来源：社区池（COMMUNITY_POOL）
        → 验证者：任何非本人的 peer
```

**统一入口 UI**（`core.js` 新增 `showContributePanel()`）：

```
┌─ 我要贡献 ──────────────────────┐
│                                   │

│  详情：___________________        │
│                                   │
│  ○ 我自己做，找人校核  (校核制)   │
│     → 做完后公开求验证，NT 从     │
│       社区池发放                  │
│                                   │
│  ○ 我想发布任务请人做  (卡片式)   │
│     → 押 NT 进 escrow，有人认领   │
│       后完成后我审核              │
│                                   │
```

### 🔧 组 3：需要迁移脚本

#### D5. DateTime 从 String → SQLAlchemy DateTime

**问题**：`models.py` 中 `created_at`/`updated_at` 等字段用 `String` 存 ISO 格式，不是 `DateTime`。排序/比较靠字符串运气（ISO 8601 字典序 ≈ 时间序，但不保证时区正确）。

**方案**：三步迁移——
1. **代码**：`models.py` 改 `Column(String)` → `Column(DateTime, default=func.now())`
2. **数据迁移脚本**：
```python
# migrate_datetime.py
async def migrate(db):
    tables = ['nt_ledger', 'nt_tasks', 'users', 'deposit_intents']
    for t in tables:
        await db.execute(text(f"ALTER TABLE {t} ADD COLUMN _dt_tmp DATETIME"))
        await db.execute(text(f"UPDATE {t} SET _dt_tmp = datetime(created_at)"))
        # ... SQLite 不支持 DROP COLUMN 旧版，需重建表
```
3. ponytail：SQLite 没有真正的 DATE 类型，`DateTime` 列仍然存字符串。真正的收益是 SQLAlchemy 层可以做 `func.date()` 查询 + 自动序列化。等下次大版本迁移（如换 PostgreSQL）时一起做。

```
    公开                社区内部（在地可操作）           个人中心
                       云村民可查看，不可操作
```

**卡片室（猜）——社区内部**：
- 有人通过校核制完成动作 → 校核通过后 → 自动生成匿名卡片到卡片室
- 卡片不透露 doer 身份，只显示「有人做了 X」
- 其他人猜是谁做的 → 猜对获小奖励（1-2 NT）
- 猜对者 + doer 绑定记录——防两人长期互猜刷奖励

**校核室（确认）——社区内部**：
- 社区任务（包括新手引导的劳动任务）出现在这里——**不用猜**
- 卡片显示 doer 名字 + 做了什么，完全透明
- 任何人可点「确认」（通过）或「质疑」（驳回）
- 确认者获验证奖励，doer 获劳动 NT
- doer + verifier 绑定记录——防串通互确认

**防作弊记录**：
- 校核室的确认者 + 卡片室的猜对者 → 两条记录交叉验证
- `{ doer, verifier, guesser, action, timestamp }` 存入 ledger
- 同对频繁互确认/互猜 → 异常标记

## 九、D1 独立执行方案：三套状态词汇统一

> ⚠️ 本项需要全库搜索替换，风险高。单独成方案，不在日常迭代中执行。

### 问题现状

系统存在三套互不兼容的任务状态词汇：

| 层 | 位置 | 示例状态 | 语言 |
|----|------|---------|:--:|
| 服务端（真相源） | nt.py, tasks.py | `进行中`, `待审核`, `待结算`, `已结算`, `退回修改`, `已取消`, `已争议` | 中文 |
| 客户端 NT 核心 | nt-core.js | `pending`, `active`, `verified`, `settled`, `rejected`, `cancelled`, `disputed` | 英文 |
| 客户端 UI | core.js, data.js | `已完成`（死分支）, `已通过`, 混杂 | 中文·第三套 |

### 目标

以**服务端中文**为唯一真相源，客户端两套词汇全部对齐。

| 服务端（真相源） | nt-core.js（当前） | 统一后 |
|:--|:--|:--|
| `进行中` | `pending` / `active` | → `进行中` |
| `待提交` | — | → `待提交`（客户端无此状态，sync 后新增） |
| `待审核` | `verified` | → `待审核` |
| `待结算` | `settled` | → `待结算` |
| `已结算` | — | 保持（服务端独有） |
| `退回修改` | `rejected` | → `退回修改` |
| `已取消` | `cancelled` | → `已取消` |
| `已争议` | `disputed` | → `已争议` |
| — | `completed`（死分支） | 删除 |

> ⚠️ `待提交` 状态：当前客户端无对应英文状态，服务端 `submit_task`（nt.py:546）将状态设为 `待审核`（跳过 `待提交`）。若未来服务端补 `待提交` 中间态，D1 迁移映射已预留该行。在 `TASK_STATUSES` 常量中列明但标注 `# 预留，当前激活路径不经过`。

### 执行步骤

#### 步骤 1：定义常量（nt.py）
在 `nt.py` 中添加 `TASK_STATUSES` 字典（服务端已是中文，作为参考）：

```python
    "进行中": "进行中", "待审核": "待审核", "待结算": "待结算",

}
```

- `'pending'` → `'进行中'`
- `'active'` → `'进行中'`
