## 🔴 阻断：必须修复才能执行（5 项）

### B1 · `render()` 未同步修改 — 调用已删除的 `renderCarousel` 将崩溃

**来源**：🧪QA（S2 升级为阻断）
**位置**：[app.js:114](nantang-mobile/js/app.js#L114)

当前：`function render() { renderContent(currentIdx); renderCarousel(currentIdx); }`
方案 Step 5 删除 `renderCarousel()` 和 `bcTrack` DOM 后，`render()` 仍然调用它 → `_q('bcTrack')` 返回 null → `null.innerHTML` 抛 TypeError → **地图页白屏**。

**修**：`function render() { renderContent(currentIdx); }`

---

### B2 · `renderContent()` 缺 idx 越界保护

**来源**：🧪QA（S1）
**位置**：[app.js:116](nantang-mobile/js/app.js#L116)

方案代码：
```javascript
function renderContent(idx) {
  var b = getBuildings()[idx];
  if (b.id !== 'info') { openBuilding(idx); return; }  // b 为 undefined 时崩溃
```

**修**：第一行加 `if (!b) { renderInfoPage(); return; }`

---

### B3 · `closeOverlay('overlayMap')` 不关闭 `overlayBuilding` — 孤立的可见 overlay

**来源**：🧪QA（S2）、⚙️系统设计师（发现 3 相关）
**位置**：[index.html:428](nantang-mobile/index.html#L428) / [data.js:534](nantang-mobile/js/data.js#L534)

用户点 X 关地图 → `closeOverlay('overlayMap')` → overlayMap 隐藏。但 `overlayBuilding` 仍在 DOM 中且有 `.open` 类 → **建筑详情继续覆盖整个屏幕**。用户以为关了地图，实际建筑还挡着。

**修**：`closeOverlay('overlayMap')` 内加 `document.getElementById('overlayBuilding').classList.remove('open')`

---

### B4 · 方案将「动作快捷入口」混为「建筑导航卡片」

**来源**：💰经济（阻塞）、🎮游戏（发现 3）、🖐️UX（必须修 #1）
**位置**：方案步骤 4

方案说「把 `_renderQuickEntryCards()` 的 onclick 从 `goTo(idx)` 改为 `openBuilding(idx)`」。但该函数渲染的是**动作卡片**（放取物品/打扫卫生/田间管理），onclick 是 `_openKitchenQuick()` 等，根本没有 `goTo(idx)`。如果按方案执行，三个高频 1 步操作入口会丢失。

**修**：
- `_renderQuickEntryCards()` 保持不动（3 个动作卡片）
- 新建 `_renderBuildingNavCards()` 渲染建筑导航卡片网格
- 替代当前底部轮播的建筑导航功能

---

### B5 · `_renderBuildingRooms` / `_renderBuildingFAB` 未定义且 DOM scope 未切换

**来源**：⚙️系统设计师（发现 1）、🧪QA（N1）
**位置**：方案步骤 3

方案调用这两个函数但从未给出实现。关键问题：`renderContent` 内部用 `_q()` 操作 `overlayMap` 内的元素（photoImg, bldName, floorTabs），新 overlay 的元素 ID 不同（bldPhotoImg, bldOverlayTitle, bldFloorTabs, bldRoomsGrid）。如果直接提取代码不改 DOM 查询，函数操作的是 overlayMap 而非 overlayBuilding。

**修**：给出两个函数的完整实现，使用 `overlayBuilding.querySelector()` 或新建 `_qBld(id)` 查询函数操作正确的 DOM scope。`setFloor`/`selectRoom`/`closeRoom` 也需要对应调整。

## 🟡 功能缺陷（4 项）

### F1 · 建筑间切换退化：1 步 → 3 步

**来源**：🎮游戏（发现 1 阻塞）、🖐️UX（原则 1）
**影响**：用户看完建筑 A 想看到建筑 B，需要返回全貌→找卡片→点击（3 步），当前只需滑动轮播（1 步）。浏览 9 个建筑的体验严重退化。

**修**：在 `overlayBuilding` 的 photo 区域保留左右箭头，`openBuilding(idx-1)` / `openBuilding(idx+1)`。不增加代码复杂度，保留连续浏览能力。

---

### F2 · `closeBuilding()` 无条件打开 overlayMap

**来源**：⚙️系统设计师（发现 3 中）、🎬创意总监（条件通过）
**影响**：如果未来从任务大厅直接跳建筑，`closeBuilding()` 会错误地在任务大厅之上打开全貌页。

**修**：加 guard — 只有 `overlayMap` 当前 open 时才恢复：
```javascript
if (document.getElementById('overlayMap').classList.contains('open')) {
  // do nothing, already open
}
// 或者更简单：不做任何操作，让用户回到之前的状态
```

---

### F3 · `currentIdx = null` 导致全貌页箭头按钮崩溃

**来源**：🎮游戏（发现 2 阻塞）
**位置**：[app.js:176-177](nantang-mobile/js/app.js#L176-L177)

`renderInfoPage()` 仍渲染了 `‹ ›` 箭头，onclick 调用 `goTo(currentIdx-1)` / `goTo(currentIdx+1)`。若 `currentIdx = null`，`goTo(NaN)` → 取 `getBuildings()[NaN]` → undefined → 崩溃或静默失败。

**修**：两种选择——（A）保留箭头但改为 `openBuilding` 直接调用；（B）全貌页不需要左右箭头，删除。建议选 B（减少视觉噪音），导航走建筑卡片网格。

---

### F4 · `checkin_5` quest hook 需迁移到 `openBuilding`

**来源**：🎬创意总监（条件通过）
**位置**：[app.js:743-751](nantang-mobile/js/app.js#L743-L751)

`goTo()` 中有 visitedSpaces 追踪和新手 quest 进度钩子。方案将卡片 onclick 从 `goTo(idx)` 改为 `openBuilding(idx)` 后，钩子被绕过。在 `openBuilding` 中补 3 行：

```javascript
if (b.id && window.AppData) {
  if (!AppData._data.visitedSpaces) AppData._data.visitedSpaces = [];
  if (AppData._data.visitedSpaces.indexOf(b.id) === -1) {
    AppData._data.visitedSpaces.push(b.id);
    if (AppData._data.visitedSpaces.length >= 5 && typeof _completeNewbieQuest === 'function')
      _completeNewbieQuest(CURRENT_USER, 'checkin_5');
    AppData._save();
  }
}
```

## 总体判断

**方案方向正确，但不可直接执行。** 5 个阻断项全部修复后，方案才是可执行的状态。修复这 5 项预计新增 ~15 行代码（B4 建筑导航卡片 + B5 函数实现 + B1/B2/B3 守卫），净效果从「~35 行删减」变为「~20 行删减」，仍然显著简化。
