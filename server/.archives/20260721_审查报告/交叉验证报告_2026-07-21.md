### #3 canUserClaim 字段不匹配 ✅ 确认
- **源码验证**：`auth.js:203` 检查 `task.status !== 'active'`，但 TASKS 状态为中文 `'进行中'`
- **影响**：所有认领被阻断
- **根因**：本次会话在 `claimTask` 新加了 `canUserClaim` 调用，但该函数使用英文状态而 TASKS 使用中文状态
- **原则评估**：**少即是多**——是否真的需要 `canUserClaim`？已有的客户端检查（`claimTask` lines 344-347）已经覆盖了自领/满员/重复检查。新增的 auth 调用引入了不一致的状态系统。**建议回退本次新增的 canUserClaim 调用，保留原有内联检查。**

### #4 verifyTask 参数模式错误 ⚠️ 部分确认
- **源码验证**：服务端 `approved: bool = True, reject_reason: str = ""` 声明为查询参数（无 Body()），客户端 `API.request('POST', ..., {approved:true})` 发送 JSON body
- **FastAPI 行为**：对于 POST 请求中无 Body() 注解的简单类型参数，**FastAPI 默认从 query string 读取**，JSON body 中的值被忽略
- **通过**：`approved=True` 是默认值，碰巧正确
- **驳回**：`approved` 保持 True，`reject_reason` 保持 ""
- **修复**：服务端添加 Body() 注解，或客户端改为 query string。最小改动：服务端加 `from fastapi import Body` → `approved: bool = Body(True)`
- **原则评估**：本真实 bug，C2 改造时遗漏了参数模式对齐

### #5 校核 approve 客户端发空 body ✅ 确认
- **源码验证**：`app-data.js:381` `API.request('POST', '/api/nt/verifications/' + vfy.id + '/approve')` — 只有 2 参，无 body
- **影响**：服务端返回 422，HTTP 模式校核永远失败
- **修复**：补齐请求体：`{doer: vfy.doer, action: vfy.action, nt_amount: vfy.ntAmount, verifier_reward: vfy.verifierReward}`
- **原则评估**：C2 改造时写了服务端端点但客户端调用未对齐

### #6 校核 approve 无金额校验 ✅ 确认
- **源码验证**：`nt.py:656-684` 直接使用 `req.nt_amount` + `req.verifier_reward`，不查询 Verification 表验证
- **影响**：结合 #5 当前实际不可达（请求体为空→422）。但修复 #5 后此问题暴露
- **修复**：在服务端查询实际 verification 记录获取权威金额。最小改动：从 `Verification` 表读取 `nt_amount` 和 `verifier_reward`，忽略客户端传入的金额
- **原则评估**：**信任边界**问题——客户端输入不可信

### #7 _hashFromCode 死锁 ✅ 确认
- **源码验证**：`index.html:1331-1345`，`closeOverlay` 设 hash=''，若当前已为 '' 则不触发 hashchange → flag 永不重置
- **场景**：连续两次 closeOverlay（如先关 map 再关 community）
- **修复**：在设置 hash 前检查 `if (window.location.hash === '#map') _hashFromCode = true;` 或在 hashchange 处理末尾加 `_hashFromCode = false`
- **原则评估**：本次引入的 bug（B+2 改造）

### #8 daily tick 非原子 ✅ 确认
- **源码验证**：`nt.py:703` `_get_pool(db)` 未传 `lock=True`
- **修复**：`pool = await _get_pool(db, lock=True)`
- **原则评估**：一行修复，符合少即是多

### #9 checkin 非原子 ✅ 确认
- **源码验证**：`accommodation.py:38-50` SELECT COUNT 与 INSERT 之间无锁
- **修复**：`select(...).with_for_update()` 或在事务中加行锁
- **原则评估**：SQLite WAL 模式下 SELECT+INSERT 间隙仍可能并发

### #10 _ntTaskId 未设置 ✅ 确认
- **源码验证**：`core.js:577` HTTP 模式跳过 `NT.createTask` → `data._ntTaskId = null`

- **原则评估**：**全局最优**——_ntTaskId 和 _srvId 的双轨身份是整个系统的基础问题

### 1. 少即是多
- ✅ E3.1 删除 ~220 行死代码 — 正确
- ⚠️ `canUserClaim` 调用引入多余复杂性 — 原有的内联检查（自领/满员/重复）已足够

### 2. 复用优先
- ✅ E3.1 复用 `_showStaySheet()` 替代 INN_ROOMS 系统
- ✅ C2 复用已有服务端端点
- ⚠️ sync 端点返回 accommodation=None——已有 `GET /api/accommodation/status` 端点但未接入

### 3. 需求对齐
- ✅ 每项改动均可追溯到 scalable-stargazing-hippo.md
- ✅ 无「顺手优化」

### 4. 全局最优
- ⚠️ isMember 散落 4 处——未从全局视角统一
- ⚠️ _ntTaskId / _srvId 双轨——任务身份系统需统一
- ⚠️ 客户端+服务端双重扣费风险——需明确 HTTP 模式下哪些客户端逻辑应禁用

### 5. 长期主义
- ⚠️ Daily tick 不写账本——会计等式会随时间漂移
- ⚠️ 住宿欠费无上限、无追缴——长期债务积累
- ⚠️ `confirm()` 签署公约无持久化记录——未来公约版本更新时无法追溯
