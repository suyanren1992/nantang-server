## 🔴 HIGH

### 1. 角色权限裂脑：npc.isMember=true 与 UI 硬编码不一致

**发现**：`nt.js` 第 29 行将 `npc.isMember` 从 `false` 改为 `true`。但 `isMember` 的判断在代码库中以两种互不通信的方式存在：

| 位置 | 判断逻辑 | npc 结果 |
|---|---|---|
| `nt.js:29` ROLE_CAPABILITIES (权威源) | `npc.isMember = true` | true |
| `auth.js:123` `getIdentity()` | 读取 `ROLE_CAPABILITIES` | true |
| `auth.js:221` `canUserClaim()` | 调用 `userCan(user, 'isMember')` | true |
| `data.js:538` `renderCommunityHub()` | `role==='admin'||role==='builder'||role==='adventurer'` | false |
| `core.js:1049-1050` `showMy()` | `role==='admin'||role==='builder'||role==='adventurer'` | false |
| `ui-village.js:56` `showCampWindow()` | `role === 'admin' || role === 'builder' || role === 'adventurer'` | false |
| `ui-village.js:6` `enterCamp()` | `role==='admin'||role==='builder'||role==='adventurer'` | false |

**影响**：npc 现在可以通过 `canUserClaim` 领取营队任务（auth 层放行），但 npc 不能进入营地（`enterCamp` 硬编码拒绝）。

**建议**：所有 `isMember` 判断应统一走 `userCan(user, 'isMember')`。

### 2. 双模式余额窗口不一致：HTTP 模式下客户端 NT 未被冻结

**发现**：`core.js` `doPublish()` 和 `publishDraft()` 中，HTTP 模式跳过客户端 `NT.createTask()`，但服务端响应回来前客户端余额不变。30 秒轮询窗口内用户看到未冻结的旧余额。

| 场景 | file:// 模式 | HTTP 模式 |
|---|---|---|
| 发布任务 | 客户端立即冻结 NT | 客户端跳过冻结，30秒后同步 |
| 余额不足时发布 | `NT.createTask` 返回 null → 拒绝 | 客户端不校验，服务端可能稍后拒绝 |
| 连发两个任务 | 余额门可拦截 | 可能超额发布 |

**建议**：HTTP 模式也应执行客户端 NT 冻结（乐观冻结 + 轮询权威修正）。

### 3. 离线 NT 变更被静默丢弃

**位置**：`core.js` `_mergeNTSyncData()` — `ntUser.ntBalance = data.balance` 直接覆盖，无冲突检测。
