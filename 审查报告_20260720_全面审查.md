## 二、发现了什么

### 🔴 CRITICAL — 服务端安全（4 项）

#### C1. JWT 密钥硬编码在源码中

- **位置**：`server/auth_utils.py` 第 7 行
- **代码**：
  ```python
  SECRET_KEY = "nantang-cloud-village-secret-key-change-in-production"
  ```
- **影响**：任何拿到源码的人都能伪造任意用户（包括管理员）的有效 JWT token。密钥值本身就写了"上线前请更换"。
- **修复**：改为读取环境变量 `os.getenv("JWT_SECRET")`，部署时设置。

#### C2. 活动日志和公告写接口无认证——任何人都能写入

- **位置**：`server/routes/data.py` 第 46、254 行
- **代码**：
  ```python
  @router.post("/activity_log")
  async def add_activity_log(req: dict, db: AsyncSession = Depends(get_db)):
      # 没有 Depends(get_current_user)
  ```
- **影响**：不需要登录就能向 `activity_log` 和 `announcements` 表写入任意数据。虽然没有直接的经济损失，但可以污染社区的公共动态流。
- **修复**：给这两个端点加上 `user: User = Depends(get_current_user)`。

#### C3. NT 操作存在 TOCTOU 竞态条件——并发请求可导致余额变负

- **位置**：`server/routes/nt.py` 所有写操作（转账、出账、入账、任务冻结）
- **原理**：
  ```python
  if user.nt_balance < req.amount:   # ① 读取余额（旧快照）
      raise HTTPException(...)
  # ... 中间可能有其他操作 ...
  user.nt_balance -= req.amount      # ② 基于旧快照写入
  await db.commit()                   # ③ 提交
  ```
  两个并发请求同时读取到余额为 50，都通过了检查。第一个扣 30（余额变 20），第二个也扣 30（余额变 -10）。总支出 60，但账户只有 50。
- **触发条件**：同一用户几乎同时发起两笔转账（概率低但存在）。
- **修复**：使用 SQLite 的行级锁（SQLAlchemy `with_for_update()`）或应用层互斥锁。

#### C4. 缺少权限检查——任意登录用户可修改他人数据

| 端点 | 位置 | 问题 |
|------|------|------|
| `PUT /api/tasks/{id}` | `tasks.py:91` | 任何登录用户可以修改任何任务的标题、状态、执行人。不检查是不是发布者。 |
| `PUT /api/camps/{id}` | `camps.py:90` | 任何登录用户可以修改任何营地的名称、主题、状态。不检查是不是创建者。 |
| `POST /api/nt/tasks/{id}/verify` | `nt.py:284` | 任何登录用户可以验证任何任务——等于任何人都能释放托管中的 NT 给自己。不检查是不是指定的审核人。 |
| `PUT /api/data/card_discoveries/{id}` | `data.py:83` | 任何登录用户可以修改任何卡片发现的状态。不检查是不是猜测者或被猜者。 |
| `PUT /api/data/verifications/{id}` | `data.py:126` | 任何登录用户可以修改任何校核记录。不检查是不是校核发起人。 |

- **修复**：每个端点加上所有权检查。例如 `if task.poster != user.id: raise 403`。

### 🟠 HIGH — 逻辑 Bug（7 项）

#### B1. 地图房间详情全部空白——`buildRoomDetail()` 缺 `return`

- **位置**：`js/app.js` 第 677–739 行
- **原因**：函数构建了 60 多行 HTML（`body += ...`），但末尾没有 `return body;`。调用处 `$('ioBody').innerHTML = buildRoomDetail(room)` 得到 `undefined`。
- **影响**：地图上点击任何非田地的房间（厨房、画室、宿舍、厕所等），弹出的详情面板是空白的。
- **修复**：在函数末尾加上 `return body;`。

#### B2. 任务结算时间永不记录——状态覆盖顺序错误

- **位置**：`server/routes/tasks.py` 第 101、112 行
- **原因**：
  ```python
  if req.status:           # 第 101 行：先把 status 改成"已结算"
      task.status = req.status
  
  # 第 112 行：再检查 status 是不是 != "已结算"——已经被改了，永远是 False
  if req.status == "已结算" and task.status != "已结算":
      task.settled_at = datetime.utcnow().isoformat()
  ```
- **影响**：所有任务的 `settled_at` 字段永远是空的。结算报告里的时间线不完整。
- **修复**：在赋值前保存旧状态，或者把 `settled_at` 的赋值放在 `task.status = req.status` 之前。

#### B3. 中英文任务状态混用——删除守卫完全失效

- **位置**：`server/routes/tasks.py:129` + `server/routes/nt.py:256`
- **原因**：
  - `nt.py` 创建任务时状态是英文：`"pending"` / `"active"` / `"verified"` / `"settled"`
  - `tasks.py` 创建任务时状态是中文：`"进行中"` / `"待审核"` / `"已结算"`
  - 删除守卫只检查英文状态：
    ```python
    if task.status in ("verified", "settled"):
        raise HTTPException(...)
    ```
  - 中文状态的任务永远不会匹配，**任何阶段都可以被删除**，包括已结算的任务。
- **影响**：通过 CRUD 路由创建的任务，即使已经结算完成，任何人都可以删除它。已发放的 NT 不会被追回。
- **修复**：统一所有状态为中文或统一所有状态为英文，二选一。建议统一为中文（与前端一致）。

#### B4. 营地预算记账——会计等式被破坏

- **位置**：`server/routes/camps.py` 第 82–84 行
- **原因**：
  ```python
  if camp_total > 0:
      lid = _ledger_id()
      await _add_ledger(db, lid, None, "camp_pool", camp_total, "topup", ...)
      # 缺少：pool.total_issued += camp_total
      # 缺少：pool.balance += camp_total
  ```
  创建营地时产生了 `camp_total` 的 NT 记账，但 `CommunityPool.total_issued` 没有增加。会计等式 `sum(用户) + 池余额 + 托管 = total_issued` 被打破。
- **影响**：创建营地后，`NT.verify()` 会计验证失败，差额恰好等于营地预算总额。
- **修复**：在记账后加上 `pool.total_issued += camp_total`。

#### B5. 空日程模板导致营地预算为零

- **位置**：`server/routes/camps.py` 第 78 行
- **原因**：
  ```python
  days = len((schedule[0] or {}).get("cells", [])) if schedule else 8
  ```
  `schedule = [{}]` 时（前端传了一个空模板），`schedule` 是 truthy，进入分支。`schedule[0]` 是 `{}`，`.get("cells", [])` 返回 `[]`，`len([]) = 0`。营地总预算 `camp_total = 0`。
- **影响**：前端向导创建营地时如果日程还没填，后端算出的 NT 预算为零。
- **修复**：`if days == 0: days = 8`。

#### B6. 卡片室发现奖励——UI 说 +10 NT，实际只给 5 NT

- **位置**：`nantang-mobile/js/ui-cardroom.js` 第 800 行
- **原因**：UI 显示 `actionNT = 10`，但实际计算走 `_getLaborNT('_discovery')`，在 `_LABOR_NT_DEFAULTS` 中找不到，返回 fallback 值 `5`。
- **影响**：用户看到"可获得 +10 NT"，确认后只收到 5 NT。
- **修复**：在 `_LABOR_NT_DEFAULTS` 中加 `_discovery: 10`，或把 UI 的显示值改为 5。

#### B7. 卡片室猜错扣 0.1 NT——浮点数在服务端被强制转为 0

- **位置**：`nantang-mobile/js/ui-cardroom.js` 第 1031 行
- **原因**：`NT.spend(g.name, 0.1, ...)` 传入浮点数。服务端 `POST /api/nt/spend` 的参数类型是 `amount: int`，Pydantic 将 `0.1` 强制转为 `0`。扣款无效。
- **影响**：猜错的惩罚实际上不生效。
- **修复**：将 0.1 改为 1（整数），或者降低发现奖励来平衡（不需扣款）。

### 🟡 MEDIUM — 其他问题

#### B8. 账本 ID 在高并发下可能碰撞

- **位置**：`server/routes/nt.py` 第 38–40 行
- **原因**：`entry_id` 用微秒时间戳生成。两个请求在同一微秒内到达会产生相同 ID。`entry_id` 有唯一约束，第二个写入抛异常导致 HTTP 500。
- **修复**：在 ID 中加入随机后缀 `uuid4().hex[:6]`。

#### B9. JWT 无撤销机制

- **位置**：`server/auth_utils.py` 第 9 行
- **原因**：Token 有效期 7 天，没有黑名单、没有登出端点。一旦签发无法撤销。
- **修复**：加入 token 黑名单表，`/api/auth/logout` 将 jti 加入黑名单。
