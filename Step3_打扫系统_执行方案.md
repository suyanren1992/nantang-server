# Step 3：打扫系统

> 来源文档：`03_卫生脏污度与打扫审核.md`
> 前置依赖：Step 2（空间定义）
> 解锁：Step 4（NT三层扩展）、Step 6（时间线）

---

## 做什么

每个空间有脏污度（0-100），每天自动增长。在地成员看到脏了→拍照打扫前→打扫→拍照打扫后→提交→另一个成员复议通过→脏污度重置 + 发奖励。复议不通过→退回重扫。

## 关键设计决策

- 审核改为**复议制**（非管理员独审）：打扫人提交 → 任意另一在地成员复议通过 → 两人署名。出问题两人追责。避免管理员不在线时的审核积压，体现社区人人平等
- ⚠️ **复议通知**：有人提交打扫后，首页/村口必须显式展示「待复议」数量（badge 角标）。否则其他成员不知道有打扫等着复议，复议制就会卡住。MVP 不做推送通知，纯前端 badge 即可。
- 排班系统延后——MVP 3-5 人时口头协调更灵活
- 脏污度增长用**固定值**（非随机浮动），避免争议
- 防刷规则：前后照必须不同、脏污度<30 不可提交、不可自审（复议人≠打扫人）
- `ponytail:` 像素比较防刷是轻度启发式检查（Canvas API，差异 >5%），不是安全机制。熟人社区中复议制的社会压力才是主要防刷手段。人数 >20 时考虑加 GPS 校验。

## 脏污度增长规则

| 空间类型 | 每日增长 |
|---------|:---:|
| 卫生间 | +15 |
| 厨房 | +10 |
| 画室 | +8 |
| 走廊 | +8 |
| 卧室 | +5 |
| 储物间 | +3 |
| 洗衣房 | +5 |
| 户外 | +2 |

## 五级状态灯

| 脏污度 | 状态 | 地图显示 |
|--------|------|:--:|
| 0-30 | 干净 | 🟢 |
| 31-60 | 轻度脏污 | 🟡 |
| 61-80 | 需要打扫 | 🔴 |
| 81-100 | 严重脏污 | 🚨 |
| 打扫中 | — | 🔵 |

## 具体任务

| # | 任务 | 说明 |
|---|------|------|
| 3.1 | 脏污度数据模型 | `nt_cleaning` 结构，与 `nt_spaces` 关联 |
| 3.2 | 自动增长逻辑 | 页面打开时计算 daysPassed × dailyGrowthBase，min(100, ...) |
| 3.3 | 状态灯渲染 | 地图每个空间叠加 🟢🟡🔴🚨🔵 颜色底色 |
| 3.4 | 打扫流程 UI | 拍照前→打扫中（等待）→拍照后→提交→复议 |
| 3.5 | 复议机制 | 任意另一在地成员可复议通过，两人署名 |
| 3.6 | 防刷规则 | 前后照像素比较、脏污度<30 拒提交、不可自审 |
| 3.7 | CV/XP 奖励 | `ponytail:` 当前调现有 `recordTransaction`（NT），Step 4 后切三栏：+40 CV +40 XP（NT 月底结算） |
| 3.8 | 待打扫列表 | 按脏污度排序，显示逾期天数 |
| 3.9 | 待复议列表 | 显示待复议的打扫提交 |

## 验证

空间脏污度每天增长。脏了→拍照打扫→复议通过→脏污度归零+发奖励。复议不通过→脏污度不变，退回重扫。
