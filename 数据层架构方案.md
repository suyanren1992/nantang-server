# 数据层架构方案

> 2026-07-18 · 从补丁到架构：用户数据隔离 → 后端迁移 → 50人并发

---

## 一、现在的问题

### 1.1 数据散落

```
TASKS      → 全局变量，无用户隔离
LEDGER     → 全局变量，无用户隔离
MOCK_ITEMS → 全局变量
CURRENT_USER → 唯一区分用户的变量
NT 系统    → 按用户（✅ 但 UI 没完全接）
auth.js    → 按用户（✅ 但只存角色/密码）
saveAppData() → 12 处散落，容易遗漏
```

### 1.2 切换用户 = 只改一个字符串

```javascript
CURRENT_USER = '小云';  // 改了，但 LEDGER.balance 还是砚仁的
refreshUserUI();        // 头像换了，余额还是共享的
```

## 二、数据层架构：UserData

### 2.1 数据结构

```javascript
var AppData = {
  // 所有用户数据
  users: {
    '砚仁': {
      tasks: { ... },       // 我发的 + 我接的（TASKS 子集）
      items: [ ... ],       // 我的物品
      ntBalance: 0,         // NT 余额
      ntHistory: [ ... ],   // NT 流水
      profile: { ... },     // 个人资料
      drafts: [ ... ],      // 草稿
    },
    '小云': { ... },
    '朝林': { ... }
  },
  
  // 共享数据（所有用户可见）
  shared: {
    taskMarket: { ... },    // 任务大厅（所有发布的任务）
    itemMarket: [ ... ],    // 物品市集
    camps: [ ... ],         // 营队信息
  }
};
```

### 2.2 API

```javascript
// 当前用户数据
AppData.my('tasks')         // → 我的任务
AppData.my('ntBalance')     // → 我的余额
AppData.mySet('ntBalance', v) // → 写入 + 自动存盘

// 共享数据
AppData.shared('taskMarket')  // → 任务大厅
AppData.sharedSet('taskMarket', v)

// 切换用户
AppData.switchUser('小云')  // → 切换上下文 + 刷新全部 UI

// 存盘（自动 + 手动）
AppData.save()              // → 全量写 localStorage
AppData.saveDebounced()     // → 200ms 防抖存盘
```

### 2.3 迁移路径

```
现在（v0）            v1（单机多用户）     v2（后端上线）
─────────           ────────────────     ──────────────
localStorage           localStorage        REST API
全局数据                UserData           POST /api/tasks
CURRENT_USER            AppData.my()       GET /api/items
                       AppData.shared()    JWT token
```

**关键设计：API 签名一致。**

```javascript
// v0/v1: localStorage
AppData.my('tasks') → localStorage['user_砚仁'].tasks

// v2: 后端
AppData.my('tasks') → fetch('/api/tasks', {headers: {Authorization: token}})
```

**调用方 `renderMyTasks()` 不需要改。只改 `AppData.my()` 的实现。**

## 三、50 人并发分析

### 3.1 当前架构（单页 localStorage）

```
50 人并发 = 50 个浏览器 Tab
每个 Tab 独立 localStorage → 互不干扰
```

**问题：谁的任务是谁的？**

当前：一个浏览器只有一个用户。50 人测试 = 50 台设备各打开一个页面。每个人看到的是自己的数据（因为 localStorage 按浏览器隔离）。**但任务大厅需要看到所有人的任务——这做不到。**

### 3.2 后端架构

```
客户端              服务端                数据库
───────           ───────              ──────
POST /task     → 验证 token → 写入 tasks 表
GET  /tasks    → 验证 token → 查询 tasks 表
POST /settle   → NT 转账     → 写入 ledger 表
```

**50 人并发瓶颈：**
- 数据库读写：50 QPS → SQLite/PostgreSQL 轻松应对
- 无实时需求（不是聊天/游戏）→ 轮询 5 秒一次即可
- 登录并发：JWT token，无状态，零压力

### 3.3 延迟预期

| 操作 | 本地（现在） | 后端（v2） |
|------|------------|-----------|
| 发布任务 | 0ms | ~50ms |
| 认领任务 | 0ms | ~50ms |
| 结算 | 0ms | ~100ms（含 NT 转账） |
| 刷新任务列表 | 0ms | ~80ms |

**结论：50 人并发对后端没有任何压力。瓶颈在前端渲染，不是数据层。**

## 四、实施路线

### Phase A：UserData 层（现在，~2h）

1. 创建 `js/user-data.js` — AppData 单例
2. 迁移 TASKS → `AppData.my('tasks')` + `AppData.shared('taskMarket')`
3. 迁移 LEDGER → 从 NT 系统读（删除 LEDGER 全局变量）
4. 迁移 MOCK_ITEMS → `AppData.my('items')`
5. 删除 12 处 `saveAppData()` 散落调用 → AppData 自动存盘
6. `refreshUserUI()` 全量重读 AppData

### Phase B：后端 API（下月，需服务器）

1. Express/Fastify + SQLite
2. JWT 登录
3. Tasks CRUD API
4. NT 转账 API
5. AppData 切换到 fetch 实现

### Phase C：性能优化（按需）

1. 任务列表虚拟滚动（100+ 任务时）
2. 轮询改 WebSocket（实时通知）
3. 图片 CDN

## 五、与现有功能的冲突分析

| 现有功能 | 用 AppData 后 | 冲突 |
|---------|-------------|------|
| TASKS 全局 | 拆为 my.tasks + shared.taskMarket | ✅ 无冲突 |
| filterQuests() | 读 shared.taskMarket | ✅ 改数据源 |
| renderMyTasks() | 读 my.tasks | ✅ 改数据源 |
| confirmSettle | 写 NT + 更新 my.ntBalance | ✅ 改数据源 |
| 物品系统 | 读 my.items | ✅ 改数据源 |
| 村口卡片 | 读 my.profile | ✅ 已有 |
| 地图 Game 桥接 | Game.getUser() 读 CURRENT_USER | ✅ 无冲突 |
| 任务大厅 | 读 shared.taskMarket | ✅ 大厅看到所有人任务 |

**唯一有冲突的：发布委托表单。** 现在写入全局 TASKS。改造后写入 shared.taskMarket + my.tasks。

## 六、建议

**现在做 Phase A。** 这是所有后续工作的地基——用户隔离、后端迁移、50 人测试，全部依赖一个清晰的数据层。

**不做 Phase A 的后果：** 每加一个新功能，数据又在某个新地方存一份。三个月后，代码里有 5 种不同的数据存取方式，没人记得哪个函数写到哪里。
