# 数据层架构方案 v2 — 移动端专用 + PC 端复用 + 新数据

> 2026-07-18 · 基于 PC 版 10 表设计 + 移动端独有功能需求

---

## 一、设计原则

1. **与 PC 版对齐字段名** — 相同的概念用相同的列名，方便以后数据互通
2. **新增移动端独有数据** — 地图交互、活动日志、通知、营地报名
3. **模板与实例分离** — 营地任务有模板（每期复用）和实例（本期执行）
4. **NT 余额从流水汇总** — 不单独存数字，防篡改
5. **编辑即保存** — 2 秒防抖自动同步 localStorage

---

## 三、与现有代码的映射

| 现有变量 | 迁移到 | 
|---------|-------|
| `TASKS` | `AppData._data.tasks` |
| `LEDGER` | `NT.getLedger()`（NT 系统自管，AppData 不存） |
| `MOCK_ITEMS` | `AppData._data.items` |
| `DRAFTS` | `AppData._data.tasks`（status='draft'） |
| `CURRENT_USER` | `AppData._currentUser` |
| `NT.earn()` / `NT.spend()` | 保持不变——AppData 不包装金融操作 |
| `NT.getUser()` / `NT.getLedger()` | `AppData.ntBalance()` / `AppData.myTransactions()` 内部调用 |

## 四、API 设计

```javascript
// 用户
AppData.me()                          // → 当前用户完整对象
AppData.me('location')                // → 实时地图位置
AppData.switchUser('小云')             // → 切换上下文

// NT 余额/流水（从 NT 系统读，AppData 不自己算）
AppData.ntBalance()                   // → NT.getUser(me).ntBalance
AppData.myTransactions()              // → NT.getLedger({user: me})

// 任务
AppData.myTasks()                     // → 我的任务
AppData.taskMarket()                  // → 任务大厅（排除草稿）
AppData.addTask({...})                // → 写 tasks 表
AppData.updateTask(name, {...})       // → 更新任务
AppData.deleteTask(name)              // → 删除任务

// 物品
AppData.myItems(filter)               // → 我的物品
AppData.addItem({...})                // → 添加物品
AppData.updateItem(id, {...})         // → 更新物品

// 邀请码
AppData.validateInviteCode(code, campId)  // → {valid, reason, entry}
AppData.consumeInviteCode(code, userName) // → 标记已使用
```

## 六、关键决策记录

1. **NT 余额从 NT 系统读，AppData 不自己算。** NT 系统是金融唯一真相源——`sum(user.ntBalance) + SYSTEM_POOL + TASK_ESCROW === _totalIssued` 的会计等式 + `verify()` 自检。AppData 做包装而非替代。AppData 不提供 `addTransaction()`，所有 NT 变动走 `NT.earn()`/`NT.spend()`。
2. **timeline 用数组不用多表**——任务的时间线是只追加的日志，不需要单独表。
3. **map_interactions 推迟到 §2**——需要等地图打扫→NT 系统接上。在此之前，地图交互的 NT 变动直接走 `NT.earn()`。
4. **通知表推迟到 §3**——§0-§2 没有任何通知 UI。数据结构在没人读它的时候价值为零。
5. **邀请码是独立表**——§1 C2 的进入营地逻辑依赖邀请码验证和一次性使用检查，不能靠硬编码。
