## P0-1. `changePwd()` 是空壳函数 — 密码从未被修改

**来源**：🛡️安全（H4）、🧪QA（S1-Critical）、🎨UX（H3）

### 🛡️安全工程师原始报告（H4）

> **nantang-mobile.html lines 1639-1647**
>
> ```js
> function changePwd(){
>   var o=document.getElementById('profileOldPwd').value;
>   var n=document.getElementById('profileNewPwd').value;
>   var c=document.getElementById('profileCfmPwd').value;
>   if(!o||!n||!c){showToast('请填写完整');return}
>   if(n.length<6){showToast('新密码至少6位','error');return}
>   if(n!==c){showToast('两次密码不一致','error');return}
>   showToast('密码已修改！','ok');renderProfile('view');
> }
> ```
>
> The function **never verifies the old password** against the stored hash and **never writes the new password** to the users object or calls `saveUsers()`. It always shows "密码已修改!" regardless. This is a broken feature that creates a false sense of security.
>
> **Fix**: Verify old password, compute and store new hash, call `saveUsers()`.

### 🧪QA 测试原始报告（S1-Critical）

> **Steps to reproduce**: Open profile -> edit -> fill in old/new passwords -> click "确认修改密码". Observe toast. Log out, try logging in with new password -- fails.
>
> **Fix**: Add actual password verification and update logic:
> ```javascript
> function changePwd(){
>   var o=...; var n=...; var c=...;
>   // validate as before...
>   var users=getUsers(); var u=users[CURRENT_USER];
>   if(!u)return;
>   var oldHash=encodePassword(o, CURRENT_USER);
>   if(u.password !== oldHash){showToast('当前密码错误','error');return}
>   u.password=encodePassword(n, CURRENT_USER);
>   saveUsers(users);
>   showToast('密码已修改！','ok');
>   renderProfile('view');
> }
> ```

### 🎨UX 设计师原始报告（H3）

> Password change never validates old password — security & UX flaw. `changePwd()` checks that new password >= 6 chars and confirmation matches, but never verifies the old password against stored data. It just shows "密码已修改!" without actually changing anything (no `saveUsers()` call).
>
> **User impact**: False sense of security. If this is placeholder code, the user believes their password changed when it did not. If the validation is supposed to work, anyone can change any user's password by clicking through the UI.

## P0-3. `resetAllData()` 后 `TASKS` 与 `AppData._data.tasks` 断开

**来源**：🧪QA（S1-Critical）、🏗️主程、💰经济（Bonus Finding）

### 🧪QA 测试原始报告（S1-Critical）

> **nantang-mobile.html line 512 (init) and line 1329 (reassignment)**
>
> `var TASKS = (window.AppData) ? AppData._data.tasks : {};` creates a reference. But `resetAllData()` at line 1329 does `TASKS = { new literal }`, which replaces the reference with a new unconnected object. After this, `AppData._data.tasks` still holds the old (pre-reset) object, and all subsequent `TASKS[name] = x` assignments write to the detached copy. The `localStorage.clear()` on line 1320 wipes AppData from storage, but `AppData._data` (in-memory) is not re-synced.
>
> **Steps to reproduce**: Click "重置" on the login screen. Observe that `resetAllData` runs, then publish a new task. The task appears in `TASKS` but is invisible to `AppData.taskMarket()` and won't survive a page reload (since AppData._data.tasks still holds the cleared/empty object).
>
> **Fix**: After `resetAllData()`, sync back to AppData:
> ```javascript
> AppData._data.tasks = TASKS;
> AppData._save();
> ```
> Or better, populate via `AppData.addTask()` instead of direct assignment.

### 💰经济设计师原始报告（Bonus Finding: Reset Desync）

> **Severity: CRITICAL** | Lines: 1320-1346
>
> The reset function seeds `TASKS` with 8 tasks (lines 1329-1340) containing NT rewards, but **never calls `NT.createTask()`** for any of them. After reset:
> - NT core: `USERS` has one user (砚仁), balance = 0 (no initial deposit), `TASKS = {}`, `TASK_ESCROW = 0`
> - AppData/TASKS: 8 tasks with real NT values, `_ntTaskId` = undefined for all
> - LEDGER: all zero
>
> The UI shows real task rewards and frozen balances, but the NT core has no record of any of them. Any `NT.verifyTask()` call on these tasks will fail because `_getTask(taskId)` returns null. The system is in an inconsistent state from boot.
>
> **Fix**: Post-reset, iterate seed tasks and call `NT.topUp(publisher, nt, 'seed task funding')` then `NT.createTask(publisher, name, nt, 'other')` for each, storing the returned `_ntTaskId`.

## P0-5. `confirmSettle()` 双重记账 — NT 已转移却再次加余额

**来源**：🧪QA（S2-High）、💰经济（Critical #1 和 #4）

### 💰经济设计师原始报告（#1 Double-Counting, Critical）

> **Severity: CRITICAL** | Lines: 816-828
>
> `confirmSettle()` writes `LEDGER.personal.balance += t.nt` (line 824), but NT was already transferred from poster to assignee in `verifyTask()` during the review step (nt-core.js line 202-206). The settlement step adds the reward amount a second time to the settler's LEDGER balance, regardless of who the settler actually is or whether they already received the NT.
>
> ```javascript
> // Line 824: Adds NT to LEDGER.personal (always, regardless of scope)
> LEDGER.personal.balance += t.nt;
> LEDGER.personal.income += t.nt;
> // Line 826: Comment acknowledges the NT already moved in verifyTask
> // NT 托管已在 reviewTask → verifyTask 时释放给认领者，无需再 earn/spend
> ```
>
> **Economic impact:** Every settled task inflates `LEDGER.personal.balance` by the reward amount without corresponding NT core balance changes. The LEDGER UI will diverge from reality by the sum of all settled task rewards.
>
> **Fix:** Remove lines 824-825 entirely, or replace with a read-only sync from NT core. The LEDGER should reflect `NT.getUser(CURRENT_USER).ntBalance`, not maintain its own counter.

### 💰经济设计师原始报告（#4 Settlement Logic, Critical）

> The financial flow is:
> ```
> Publish → NT.createTask → poster.ntBalance -= reward, TASK_ESCROW += reward  (nt-core:126-127)
> Claim   → No NT change  
> Submit  → No NT change
> Review  → NT.verifyTask → TASK_ESCROW -= reward, assignee.ntBalance += reward (nt-core:202-206)
> Settle  → LEDGER.personal.balance += reward (HTML:824) ← BUG: NT already moved!
> ```
>
> The NT is already transferred during `verifyTask` (review step). Settlement should be a pure status change. But `confirmSettle()` adds the full reward amount to `LEDGER.personal.balance`, effectively double-counting.
>
> Also: `confirmSettle()` has no scope awareness. It always writes to `LEDGER.personal` regardless of whether the task scope is `营队` or `社区`.

### 🧪QA 测试原始报告（S2-High）

> Settlement unconditionally does `LEDGER.personal.balance += t.nt` and `LEDGER.personal.income += t.nt`. A camp-scope or community-scope task settlement incorrectly inflates the personal ledger. The camp and community ledgers are never credited.
>
> **Fix**: Route to the correct ledger bucket based on `t.scope`:
> ```javascript
> var bucket = t.scope==='营队' ? LEDGER.camp : t.scope==='社区' ? LEDGER.community : LEDGER.personal;
> bucket.balance += t.nt; bucket.income += t.nt;
> bucket.records.unshift({d:today().slice(5), note:t.name+' 结算', amt:t.nt, in:true});
> ```

> ⚠️ **注意**：QA 的修复建议与经济的修复建议矛盾——QA 建议纠正 scope 路由，经济建议整段删除（因为 NT 已转帐）。建议采纳经济的方案：**删除 lines 824-825**，LEDGER 应只读 NT 核心而非自己维护计数器。

## P0-6. 四套平行状态存储

**来源**：🏗️主程（Finding 4, HIGH）、💰经济（#2 Missing Transactions, Critical + #3 Balance Inconsistency, Critical）

### 💰经济设计师原始报告（#3 Balance Inconsistency, Critical）

> **Severity: CRITICAL** | Lines: 468-507, 1347-1390
>
> Three separate balance representations exist simultaneously:
>
> | Source | Where Shown | Updated By |
> |--------|------------|------------|
> | `NT.getUser(name).ntBalance` | Profile card (line 1370-1372), top bar | NT core operations |
> | `LEDGER.personal.balance` | Ledger cards (line 503) | Only `confirmSettle()` |
> | `LEDGER.camp.balance` / `LEDGER.community.balance` | Ledger cards (lines 504-506) | **Never written** |
>
> The `refreshUserUI()` function (line 1347) reads from both `NT.getUser()` for the main NT display AND `refreshLedgerCards()` which reads the stale LEDGER object. They will always show different numbers after any non-settlement NT activity.
>
> Additionally, `LEDGER.camp.balance` and `LEDGER.community.balance` are initialized to 0 at line 470-471 and NEVER modified anywhere in the codebase (confirmed by grep: no writes to `LEDGER.camp` or `LEDGER.community` beyond the initialization).
>

>

### 🏗️主程原始报告（Finding 4: Parallel state stores, HIGH）

> There are at least four independent state stores for overlapping data:
>
> | Store | Location | Persisted | What it holds |
> |-------|----------|-----------|---------------|
> | `TASKS` (inline global) | line 512, aliased to `AppData._data.tasks` | Yes (via AppData) | Task objects keyed by name |
> | `LEDGER` (inline global) | line 468 | No | Financial records per-scope |
> | `AppData._data` | app-data.js | localStorage `nt_app_v2` | Users, tasks, items, camps |
> | `NT` (nt-core.js) | localStorage `nt_core_v1` | Yes | Users, tasks, ledger, settlements |
> | `DRAFTS` (inline global) | line 513 | Yes (via AppData) | Draft tasks |
> | `MOCK_ITEMS` (inline global) | line 514 | No | Inventory items |
>
> `TASKS` and `AppData._data.tasks` are the same object reference (line 512), which is good -- but `LEDGER` is completely independent. `confirmSettle` (line 820-828) writes to `LEDGER` manually alongside calling `AppData.updateTask()`, and the NT system has its own separate ledger. The accounting equation `NT.verify()` will never match the UI-side `LEDGER` unless every write is manually mirrored in three places.
>
> **Fix**: Pick one source of truth per domain. The NT system (`nt-core.js`) should own all financial data. The UI should read from NT's ledger via `NT.getLedger()`, not maintain a separate `LEDGER` object. The inline `LEDGER` global at line 468 should be deleted; `openLedger()` and `refreshLedgerCards()` should call `NT.getLedger()`.

### 💰经济设计师原始报告（#2 Missing Transactions, Critical）

> Every NT flow writes to NT core's `LEDGER` array (via `_addLedger`) but NOT to the UI `LEDGER` object. The following 7 NT events produce zero UI ledger records:
>
> | Event | NT Core Line | UI LEDGER Written? |
> |-------|-------------|---------------------|
> | Task publish (freeze NT) | nt-core.js:158-159 | No |
> | Task cancel (refund NT) | nt-core.js:244 | No |
> | Review approve (release NT to assignee) | nt-core.js:209 | No |
> | Review reject (refund NT to poster) | nt-core.js:223 | No |
> | TopUp approval | nt-core.js:367 | No |
> | CashOut approval | nt-core.js:379 | No |
> | Task claimed/unclaimed | n/a | No |
>
> Only `confirmSettle()` writes to the UI `LEDGER.personal.records` (line 825).
>
> **Economic impact:** The UI ledger shows at most 1 record per task (settlement), when the actual financial history involves 3-7 events per task lifecycle. Users cannot audit their own NT movement.
>
> **Fix:** Either (a) make `refreshLedgerCards()` read from `NT.getLedger({user: CURRENT_USER})` and drop the UI LEDGER object entirely, or (b) add LEDGER writes at each of the 7 listed call sites.

## P0-7. XSS — `esc()` 函数存在但从未被调用

**来源**：🛡️安全（C1, Critical，独立发现）

### 🛡️安全工程师原始报告（C1: XSS via innerHTML with unescaped user data）

> **What:** User-supplied strings (names, task titles, notes, submissions, review comments, item details) are interpolated directly into HTML strings and injected via `innerHTML` without calling the `esc()` function that exists in `utils.js`.
>
> The `esc()` function IS defined at `js/utils.js:163`:
> ```js
> function esc(s) { return (s||'').replace(/&amp;/g,'&amp;amp;').replace(/&lt;/g,'&amp;lt;')...; }
> ```
> but it is **never called** in the HTML generation code within `nantang-mobile.html`. Every string concatenation building HTML is raw.
>
> **Affected locations (not exhaustive -- essentially every innerHTML call in nantang-mobile.html):**
>
> | Line(s) | Function | What is unsafely interpolated |
> |---------|----------|-------------------------------|
> | 630 | `toggleCard()` | `t.note` (task description) |
> | 642 | `toggleCard()` | `t.reviewNote` (rejection note) |
> | 644 | `toggleCard()` | `t.submission` (submitter's text) |
> | 654 | `openSub()` | `t.note`, `t.name` |
> | 658 | `openSub()` | `t.poster`, `t.reviewer` |
> | 661 | `openSub()` | claimant names via `.map()` |
> | 711-715 | `openSubmit()` | `t.name` interpolated into HTML template |
> | 830-844 | `settleTask()` / `renderRow` | `t.name`, done claimant names |
> | 980-984 | `questCard()` | `t.name`, `t.publisher` |
> | 1000 | `toggleQuestCard()` | `t.note` |
> | 1019 | `toggleQuestCard()` | `t.reviewNote` |
> | 1021 | `toggleQuestCard()` | `t.reviewer` |
> | 1058-1061 | `toggleQuestCard()` | `c.name` (claimant names) via DiceBear URL |
> | 1080 | `toggleClaimantSub()` | `cname`, `c.submittedAt`, `c.submission` |
> | 1106 | `filterPubTargets()` | user name `u` in search dropdown |
> | 1123 | `filterReviewers()` | user name `u` in search dropdown |
> | 1185-1195 | `publishTask()` | `name`, `note` in confirm dialog |
> | 1261 | `renderLoginUserList()` | `n` (user name) in chip |
>
> **Fix:** Call `esc()` on every user-provided value before interpolation, e.g., change `'+t.name+'` to `'+esc(t.name)+'`.
>
> ⚠️ **长期主义**：逐处手动加 `esc()` 是机械正确但长期脆弱的方案——下一个写代码的人 100% 会遗漏。建议同步封装一个 `html(strings, ...values)` 模板标签函数或 `safeInnerHTML(el, html)` 辅助函数（~10 行），一次封装、永久防漏，而非依赖每次手动记忆。

## H11. 冻结 NT 计算五个缺陷

**来源**：💰经济（#5 Frozen NT Calculation, High）

> **Severity: HIGH** | Lines: 1382-1390
>
> ```javascript
> var frozen=0;
> Object.values(TASKS).forEach(function(t){
>     if(t.status==='已结算')return;
>     var isClaimant=(t.claimants||[]).some(function(c){return c.name===u});
>     if(isClaimant)frozen+=(t.nt||0);
> });
> ```
>
> Five defects:
>
> 1. **退回修改 tasks counted as frozen** (line 1385): When a task is rejected, `NT.verifyTask(id, verifier, false)` refunds the NT to the poster (nt-core.js:221-222). But this code still counts the full `t.nt` as frozen for the claimant — NT that was already returned.
> 2. **Multi-claimant double-count** (line 1387): If a task has 3 claimants and reward is 15 NT, ALL THREE claimants see 15 NT frozen. Total frozen across users = 45 NT for a 15 NT task.
> 3. **Source of truth mismatch**: Frozen is calculated from `TASKS` (AppData), but the actual frozen NT lives in `NT._tasks` as `TASK_ESCROW`. On reset, `TASKS` is seeded with tasks (line 1329-1340) whose `_ntTaskId` is never set, so `NT.createTask()` was never called for them. > 4. **Status 'in_progress' claimant counted**: Tasks in `进行中` where the claimant hasn't submitted anything still show NT as frozen for that user.
> 5. **Refunded/canceled tasks in NT core not reflected**: If `NT.cancelTask()` refunds NT but `TASKS[name]` status isn't updated (the deletion happens in `withdrawTask()` line 791: `delete TASKS[name]`), OK. But if canceled via some other path, frozen calculation from TASKS is wrong.
>
> **Economic impact:** Users see inflated frozen balances, making their "available" NT appear lower than reality. The frozen bar visualization (line 1390) shows `frozen / (balance + frozen)` which will be misleading.
>
> **Fix:** Calculate frozen from NT core: `NT._tasks` filtered by status and user, or expose `NT.getFrozenBalance(userId)`. The comment at line 826 claims "NT handles frozen in createTask", so use that.
>
> 💡 **复用优先**：优先在 UI 层写纯函数从 `NT._tasks` 直接计算冻结合计，不必修改 nt-core.js 暴露新 API——`NT._tasks` 已包含所有冻结任务的完整数据，新增 API 是多余的维护面。

## H12. `NT.earn()` 允许 SYSTEM_POOL 无限为负 — 隐性通胀

**来源**：💰经济（#7 Currency Sinks vs Faucets, Medium）

> **Severity: MEDIUM** | Lines: nt-core.js:339-358, 362-381
>
> **Critical flaw:** `NT.earn()` (line 339-348) lets SYSTEM_POOL go negative without bound:
> ```javascript
> function earn(userId, amount, reason, scope) {
>     SYSTEM_POOL -= amount;   // No floor check! Can go negative
>     u.ntBalance += amount;   // User gets NT even if pool is empty
> }
> ```
> This breaks the accounting equation. If `SYSTEM_POOL = -1000`, then `sum(users) + (-1000) + TASK_ESCROW = _totalIssued`, meaning there's 1000 more NT in user hands than was ever issued — silent inflation.
>
> **Inflation risk:** If map iframe calls `ntEarn` via postMessage (line 1682) repeatedly, or if a game event awards NT via `earn()` without corresponding `spend()` calls, SYSTEM_POOL drains. With no rebalancing mechanism, this is a permanent leak.
>
> Currently, SYSTEM_POOL starts at 0 (line 34). Every `NT.earn()` call makes it negative. NT created via `topUp()` only increases `_totalIssued`, not `SYSTEM_POOL`. So after the first `NT.earn()`, the accounting equation is broken and stays broken.
>
> **Fix:** `NT.earn()` should either (a) reject if `SYSTEM_POOL < amount`, or (b) increase `_totalIssued` when SYSTEM_POOL is insufficient (making it a true faucet with audit trail), or (c) both `earn` and `spend` should affect `_totalIssued` directly and drop the SYSTEM_POOL concept.

## H13. 弱密码哈希 — 自研算法 + 硬编码盐

**来源**：🛡️安全（H1, High）

> **File:** `js/utils.js:140-155`
>
> ```js
> function encodePassword(password, username) {
>   var SALT = 'N4nT4nG_v2_$4Lt';
>   // 2000 iterations of simple multiply-add hash ...
> }
> ```
>
> This is a custom algorithm, not a standard cryptographic hash (SHA-256, bcrypt, argon2). The salt is hardcoded (not per-user). Since this is a client-only app, this is acceptable for preventing casual snooping but would be trivially brute-forced if the data file leaks.
>
> **Fix:** At minimum, replace with `SHA-256(password + per_user_salt)` using the Web Crypto API (`crypto.subtle.digest`). For a local app, this is a defense-in-depth improvement, not a critical fix.

> ⚠️ **长期主义**：`crypto.subtle.digest()` 返回 Promise（异步），而当前 `encodePassword()` 是同步函数，被 `loginUser()`、`registerUser()`、`changePwd()`、种子数据初始化 4 处同步调用。改为异步需连锁重构所有调用栈为 async/await。评估成本后再决定——对本地单机 App，当前自研哈希的风险等级实际上低于引入异步复杂度的回归风险。

## M19. 轮播滚动中实时 `filter:blur()` 开销巨大

**来源**：⚡性能（High #5 和 Medium #11）

### 性能分析师原始报告（Medium #11）

> **Lines 1662-1671**
>
> Already has `passive: true` (good), but the handler writes to 4 CSS properties on all 3 `.vp-card` elements per scroll frame: opacity, filter (blur), background, color. Each write triggers a style recalculation. The blur filter write is particularly expensive because it forces texture re-reads.

> filter:blur(3px)brightness(.55)
> ```

>
> **Fix**: pre-blur the background image in an image editor, use the pre-blurred version.
